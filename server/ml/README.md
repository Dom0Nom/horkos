# Horkos server model training

This directory contains NumPy training and ONNX export tooling for the optional
server-side aim and broad behavioral scorers. Models are deployment inputs; the
live server loads them only when the corresponding environment variable is set.

## Aim-kinematics classifier

`train_aim_model.py` fits the synthetic honest-vs-bot logistic classifier used
by `ban_engine::scoring::AimScorer`. It consumes the raw four-feature vector
`[overshoot_ratio, settle_ms, jerk_abs, pad]`; feature standardization is folded
into the exported weights before writing the `Gemm + Sigmoid` graph.

```bash
python3 train_aim_model.py /tmp/aim_score.onnx
HORKOS_AIM_MODEL=/tmp/aim_score.onnx cargo run -p api --bin horkos-api
```

The repository does not commit a deployment aim model. As with the behavioral
model below, its hand-specified synthetic distributions validate the training
and export path rather than real-world efficacy.

## Broad behavioral reconstruction model

`behavior_features.json` is the reviewable feature-order contract shared by the
Python tooling and Rust contract tests. `synth_behavior.py` produces window-level
aggregates in that order. `train_anomaly_model.py` fits a rank-4 linear/PCA
autoencoder on 12,000 synthetic honest windows, calibrates reconstruction error
on a separate 4,000-window honest split, and folds feature normalization,
reconstruction, and calibration into one opset-13 ONNX graph.

```bash
python3 train_anomaly_model.py /tmp/horkos-anomaly.onnx
python3 eval_anomaly_model.py /tmp/horkos-anomaly.onnx
HORKOS_ANOMALY_MODEL=/tmp/horkos-anomaly.onnx cargo run -p api --bin horkos-api
```

Dependencies are NumPy and ONNX only. Evaluation executes the saved graph with
`onnx.reference.ReferenceEvaluator`; it does not reuse the trainer's in-memory
forward pass or require onnxruntime, torch, sklearn, or scipy.

The graph emits a non-negative fusion-scale anomaly strength. The independent
calibration split's empirical 99th-percentile reconstruction error maps to the
current weak fusion tier (`2.5`). This is deliberately not described as a
statistical z-score. Rust applies the active `FusionParams.z_weak` floor and
does not normalize or remap the output again.

The live extractor requires a contiguous schema-v4+ suffix with at least 64
ticks, 32 ticks carrying usable HID intervals, and eight valid flick,
direction-change reaction, and target-switch segments apiece. Missing or
invalid evidence emits no signal. Signal 217 still cannot ban alone because
fusion requires at least two distinct Moderate-or-strong signals.

Deployment artifacts should be generated outside the repository. The committed
`ban-engine/tests/fixtures/anomaly_score.onnx` is a deterministic, test-only
train/serve contract fixture generated by the same trainer.

## Honesty and limitations

All training, calibration, and evaluation rows are **synthetic**. The anomalous
generator is hand-defined alongside the honest generator, so ROC-AUC, false-
positive rate, and detection rate are partly circular. They are pipeline sanity
checks for deterministic generation, ONNX export/execution, calibration, and
Rust integration—not evidence of real-cheater efficacy, bypass resistance, or
production false-positive performance.

Production use requires an honestly collected, platform/cohort-aware baseline,
held-out real telemetry, drift monitoring, and signed calibration policy. The
same limitation applies to the aim model and existing placeholder population
baselines in `server/telemetry/src/analyzers/`.
