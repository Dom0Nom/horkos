# Horkos Detection Catalog

Niche observability/detection signals for the Horkos game-integrity client, beyond the basic implemented sensors (Ps* notify routines, ObRegisterCallbacks handle filter, eBPF LSM/tracepoints, EndpointSecurity exec, PCIe/IOMMU DMA check).

Each signal is a **read-only** sample the client reports to the server; all trust decisions are server-side. Generated by a 24-domain specialist design pass; entries are design proposals, not yet implemented.

- **Total signals:** 217
- **Domains:** 24
- **fp_risk legend:** low / medium / high false-positive risk before gating.

> One domain — *Windows Kernel Syscall/ETW/PatchGuard Surface Integrity* — was hand-authored separately (see the dedicated section) because the automated design pass could not generate it.

## Windows Kernel — Object / Notify Callback Integrity  `win-kernel-object-callbacks`

### 1. Own-ObCallback liveness self-poll (Ob registration cookie validation)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** On a periodic timer the driver re-confirms its own ObRegisterCallbacks handle is still the live registration by issuing a known cross-process handle open against a sentinel and confirming HkObPreCallback actually fired and stamped a per-poll nonce; a clean system always sees the callback execute, a deregistered/unhooked driver sees the nonce never get set.
- **Mechanism:** KTIMER/KDPC (KeInitializeTimerEx + KeSetTimer) drives a passive-level work item that NtOpenProcess/ZwOpenProcess-opens a sentinel handle with a magic DesiredAccess; HkObPreCallback recognizes the nonce and records it. ObRegistrationHandle validity is the implicit invariant.
- **Catches:** Silent ObUnRegisterCallbacks of Horkos's own registration by a malicious driver, or NOP-ing of the OB_CALLBACK_ENTRY callback function pointer so the pre-op never dispatches.
- **FP risk:** low — Only fires on genuine non-execution. A heavily loaded box could delay the work item; gate by requiring N consecutive missed polls (not one) and by confirming the timer DPC itself ran (separate heartbeat counter) so scheduler starvation is distinguishable from callback removal.
- **Horkos slot:** `kernel/win/src/CallbackSelfCheck.c (new) declared in kernel/win/include/horkos_kernel.h as HkObSelfPoll`
- **Refs:** ObRegisterCallbacks, ObUnRegisterCallbacks, KeInitializeTimerEx, KeSetTimerEx, IoQueueWorkItem, OB_PRE_OPERATION_CALLBACK

### 2. CallbackList entry walk for our OB_CALLBACK_ENTRY position and altitude neighbors

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Walks the object type's CallbackList (OBJECT_TYPE.CallbackList LIST_ENTRY of OB_CALLBACK_ENTRY) recovered via the registration handle, recording the ordered set of {Altitude, Enabled, PreOperation, PostOperation} so the server sees if a higher-altitude entry was inserted above Horkos, our Enabled flag was cleared, or our PreOperation pointer was swapped, versus the baseline captured at arm time.
- **Mechanism:** ObGetObjectType / *PsProcessType plus the registration handle returned by ObRegisterCallbacks (the handle is the CALLBACK_ENTRY_ITEM); the entry's containing OB_CALLBACK_ENTRY exposes the RegistrationContext/AltitudeIndex list. Pointer-validated walk under no lock at PASSIVE, compared to a baseline snapshot taken in HkObArm.
- **Catches:** Callback-table edits: an attacker clearing the Enabled byte on Horkos's entry, repointing PreOperation to a passthrough stub, or splicing a higher-altitude entry that pre-reduces DesiredAccess so our log sees only already-stripped masks.
- **FP risk:** medium — Undocumented internal layout — version-fragile across Windows builds; legitimate AV/EDR (Defender, CrowdStrike) also occupy entries and legitimately re-register on update, shifting list order. Gate by validating only our own entry's fields against our baseline (not absolute order) and ship per-build offset tables; treat order changes as low-weight context, only field-mutation of OUR entry as high-weight.
- **Horkos slot:** `kernel/win/src/CallbackSelfCheck.c (new), baseline stored in HK_DEVICE_CONTEXT.ObBaseline`

### 3. Ps* notify-routine slot occupancy verification via PsSetCreateProcessNotifyRoutineEx re-arm probe

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Periodically attempts to re-register the SAME already-registered HkProcessNotifyEx/HkThreadNotify/HkLoadImageNotify routine; a still-registered routine returns STATUS_INVALID_PARAMETER (duplicate), whereas a successful re-registration proves the routine had been silently removed from the PspCreateProcessNotifyRoutine array out from under us.
- **Mechanism:** PsSetCreateProcessNotifyRoutineEx / PsSetCreateThreadNotifyRoutine / PsSetLoadImageNotifyRoutine return codes; documented behavior is duplicate-registration rejection. Result is emitted to the ring as a self-integrity event.
- **Catches:** Deregistration of Horkos notify routines (PsSetCreateProcessNotifyRoutineEx(routine, TRUE) called by an attacker driver, or direct nulling of the global notify array slot) that the basic capture path cannot notice because it only fires when called.
- **FP risk:** low — No legitimate software removes another driver's notify routine; the only FP is our own disarm path racing the probe. Gate by holding the same arm/disarm lock and skipping the probe while NotifyRoutinesArmed is mid-transition. If re-arm unexpectedly succeeds, immediately re-disarm the duplicate to keep slot count correct.
- **Horkos slot:** `kernel/win/src/CallbackSelfCheck.c (new), serialized with Notify.c arm state`

### 4. Registered notify-routine count census via PsEnumProcessNotifyRoutines-equivalent array scan

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Counts how many process/thread/image notify callbacks are currently registered system-wide and reports the census; an abnormally LOW count (collapse from the typical AV/EDR/Horkos baseline toward zero) indicates a mass-deregistration cheat that strips all notify routines to blind the whole class, while our own slot disappearing is caught directly.
- **Mechanism:** PsSetLoadImageNotifyRoutineEx registration accounting plus the documented hard cap (PSP_MAX_CREATE_PROCESS_NOTIFY = 64 slots); the driver tracks deltas in the populated-slot count over time. Census taken on the same self-check timer.
- **Catches:** Wholesale callback-table clearing (anti-EDR drivers that zero PspCreateProcessNotifyRoutine to defeat all process-create telemetry), which manifests as the registered count dropping below the boot-time floor.
- **FP risk:** medium — Legitimate AV/EDR load/unload (update, user uninstall) and the system's own count fluctuate at boot; a single low reading is not proof. Gate by establishing a per-host floor after boot settles, alerting only on a sharp monotone drop that includes our own slot, and excluding known AV-uninstall windows the userspace SDK can correlate with service-stop events.
- **Horkos slot:** `kernel/win/src/CallbackSelfCheck.c (new); census struct in hk_event schema as hk_event_callback_census`

### 5. CmRegisterCallbackEx registry-callback observation of protected-key tamper

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Registers a CmRegisterCallbackEx registry filter scoped to Horkos's own service/config/driver-image registry keys and emits RegNtPreSetValueKey/RegNtPreDeleteKey/RegNtPreDeleteValueKey on any non-SYSTEM writer touching ImagePath, Start, altitude, or our policy values — distinguishing the legitimate installer/service-control-manager writer from an attacker repointing or disabling the boot-start driver.
- **Mechanism:** CmRegisterCallbackEx with REG_NOTIFY_CLASS RegNtPreSetValueKey/RegNtPreDeleteKey/RegNtPreDeleteValueKey; the callback's REG_SET_VALUE_KEY_INFORMATION gives target key path + new data, ProcessId of the writer via PsGetCurrentProcessId.
- **Catches:** Registry-level tampering with the AC's own persistence: flipping Start to disabled, repointing ImagePath to a hollowed driver, or editing the registered altitude — all of which precede a callback-surface bypass.
- **FP risk:** medium — Group Policy, Windows Update servicing, and legitimate driver re-install all write these keys via SYSTEM/TrustedInstaller. Gate by allow-listing writes whose requester token is SYSTEM/TrustedInstaller AND occurring during a known servicing window; treat writes from a non-elevated or game-adjacent process as high-weight. Scope the callback to ONLY our keys to bound FP volume.
- **Horkos slot:** `kernel/win/src/RegistryCallback.c (new) implementing HkCmArm/HkCmDisarm in horkos_kernel.h`

### 6. MiniFilter altitude-squatting census via FltEnumerateInstances/FltEnumerateFilters

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Enumerates loaded minifilters and their instance altitudes and flags any filter parked at an altitude numerically adjacent-above Horkos's load-order group (or an unallocated/colliding altitude string) that could intercept the same I/O before our handle filter sees it; a clean system shows only the expected, Microsoft-allocated altitudes.
- **Mechanism:** FltEnumerateFilters + FltGetFilterInformation (FilterFullInformation) and FltEnumerateInstances + FltGetInstanceInformation (InstanceFullInformation -> Altitude) from a companion minifilter, or FilterFindFirst/FilterFindNext from the userspace SDK via fltlib.
- **Catches:** Altitude-squatting: a rogue minifilter registering at a higher altitude than Horkos to shim file/handle operations before Horkos observes them, or an unsigned filter using a stolen/duplicated altitude string.
- **FP risk:** medium — Many legitimate AV/backup/encryption/cloud-sync filters (Defender 328010, Carbon Black, Druva, OneDrive cloud-files) legitimately occupy high altitudes. Gate against Microsoft's published Allocated Altitudes registry plus a signed-publisher allowlist; only flag filters at an UNallocated altitude or with an Authenticode chain that fails, never altitude-occupancy alone.
- **Horkos slot:** `sdk/src/backends/win/MinifilterCensusWin.cpp (new) feeding the SDK; kernel companion stub in kernel/win/src/Whitelist.c census path`

### 7. ObCallbackList Enabled-flag and Flags drift on PsProcessType vs PsThreadType

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Independently snapshots the Enabled state of Horkos's CREATE and DUPLICATE operation registrations on BOTH PsProcessType and PsThreadType and reports any asymmetry — e.g. our thread-type pre-op silently disabled while the process-type one stays live, a surgical partial-unhook that lets THREAD_SET_CONTEXT opens through while keeping the process path looking healthy.
- **Mechanism:** The OB_OPERATION_REGISTRATION[] array Horkos itself passed to ObRegisterCallbacks plus the live OB_CALLBACK_ENTRY.Enabled/Operations fields reachable from the registration handle; per-object-type, per-operation comparison against the arm-time baseline.
- **Catches:** Surgical callback-table edits that disable only one object-type or only the DUPLICATE operation to reopen a specific handle-acquisition path while keeping aggregate liveness checks (signal #1) green.
- **FP risk:** low — Horkos controls both registrations identically at arm time, so any asymmetry is anomalous by construction; the only FP is our own disarm of one type before the other. Gate by treating the brief disarm window as masked and only alerting on asymmetry that persists across two self-check cycles.
- **Horkos slot:** `kernel/win/src/CallbackSelfCheck.c (new), per-type fields in HK_DEVICE_CONTEXT.ObBaseline`

### 8. Driver image-section integrity of the registered callback thunk pages

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Hashes the read-only .text pages backing HkObPreCallback / HkProcessNotifyEx / HkLoadImageNotify at arm time and re-hashes on the self-check timer; a clean driver's callback code is immutable, so a mismatch means an inline patch (jmp-detour or int3) was written over our callback prologue even though the registration pointer still points at our routine.
- **Mechanism:** MmGetSystemRoutineAddress is not used; instead the driver captures the function addresses directly, resolves the owning image base via the loaded-module list (PsLoadedModuleList walk / AuxKlibQueryModuleInformation), and CRC/SHA-256s the code range. Re-hash compared under PASSIVE_LEVEL.
- **Catches:** Inline hooking / code-patching of the callback routines themselves (detour at the prologue, or patching the strip branch to a NOP) that leaves the OB/Ps registration tables pristine and thus invisible to table-walk signals.
- **FP risk:** low — Legitimate code pages are never rewritten post-load; the realistic FP is a hot-patch from Microsoft (KB) or our own runtime relocation, neither of which touches an already-resident signed driver's .text. Gate by hashing only the .text range (exclude IAT/relocs), and by confirming via attestation that the on-disk image matches before trusting the in-memory baseline.
- **Horkos slot:** `kernel/win/src/CallbackSelfCheck.c (new); reuses Attestation interface for the on-disk anchor`

### 9. CmRegisterCallbackEx altitude collision and registry-callback census

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Enumerates the registered registry-callback (Cm) altitudes and confirms Horkos's own CmRegisterCallbackEx cookie is present and that no other callback sits at a colliding/higher altitude that could swallow or rewrite the RegNt* notifications before Horkos sees the tamper on its own keys; baseline is our cookie present + expected neighbor set.
- **Mechanism:** CmRegisterCallbackEx returns a LARGE_INTEGER Cookie and takes a PCUNICODE_STRING Altitude; the count and presence of our cookie are tracked, and the registry-callback list (CmpCallBackVector accounting, with the documented 100-callback cap and altitude-uniqueness contract) is censused on the self-check timer.
- **Catches:** Deregistration (CmUnRegisterCallback of our cookie) or altitude-squatting on the registry-callback path that would blind signal #5 — an attacker silencing registry-tamper detection before editing the AC's persistence keys.
- **FP risk:** medium — AV/EDR and some installers legitimately register Cm callbacks and can collide for benign reasons; the absolute count fluctuates. Gate on OUR cookie's presence (high-weight if missing) rather than total count, and require the missing-cookie signal to coincide with a failed re-registration before escalating, mirroring the notify re-arm probe in signal #3.
- **Horkos slot:** `kernel/win/src/RegistryCallback.c (new), cookie + baseline in HK_DEVICE_CONTEXT.CmBaseline`

## Windows Kernel — Memory Injection & Image Anomalies  `win-kernel-memory-injection`

### 10. Unbacked executable VAD (private commit, no ControlArea)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Every executable region in a clean process is a VadImageMap with a ControlArea pointing at a FILE_OBJECT; an executable VAD of type VadNone/VadDevicePhysicalMemory with no ControlArea is a manually-mapped image.
- **Mechanism:** From the driver, attach with KeStackAttachProcess to the target's EPROCESS, walk the VAD AVL tree from EPROCESS.VadRoot (via the documented MmVad/MMVAD layout offsets), and for each node read VadType and the Subsection->ControlArea->FilePointer chain; classify any leaf whose protection includes EXECUTE (MM_EXECUTE*, MM_EXECUTE_READ, MM_EXECUTE_READWRITE) but whose ControlArea/FilePointer is NULL.
- **Catches:** Manual-mapped DLLs, reflectively-loaded payloads, shellcode in private commit that PsSetLoadImageNotifyRoutine never sees because no section was created.
- **FP risk:** medium — JITs (CLR, V8/Chakra, JVM, ANGLE shader cache) legitimately create private +X. Gate by allow-listing CLR/JIT regions via the loaded-module list owning the allocation and by correlating with a known JIT host module; report only unbacked +X with no JIT owner.
- **Horkos slot:** `kernel/win/src/MemScanVad.c (new TU; HK_EVENT_MEM_UNBACKED_EXEC added to sdk/include/horkos/event_schema.h)`
- **Refs:** VadRoot / MMVAD / _MMVAD_SHORT, Subsection->ControlArea->FilePointer, KeStackAttachProcess/KeUnstackDetachProcess, MM_EXECUTE protection constants

### 11. PTE/VAD protection divergence (W^X stealth flip)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** In a clean process the hardware PTE execute-disable (NX) bit agrees with the VAD's declared protection; a region whose VAD says RW but whose live PTEs are executable (or vice-versa) indicates a post-commit VirtualProtect flip used to hide +X from a VAD-only scan.
- **Mechanism:** After the VAD walk, for committed pages resolve the leaf PTE via MmGetVirtualForPhysical/MiGetPteAddress-equivalent documented translation (or sample with MmGetPhysicalAddress per page) while attached, and compare the PTE NoExecute bit against the VAD-declared protection mask; a mismatch is the signal.
- **Catches:** Module stomping and shellcode that commits RW, writes payload, then flips to RX so the VAD-cached protection lags; transient RWX windows.
- **FP risk:** medium — JIT and trampoline thunks legitimately toggle protection; gate by requiring the mismatched region to also be unbacked or to overlap a signed module's .text. Sample, do not poll hot, to avoid racing legitimate VirtualProtect.
- **Horkos slot:** `kernel/win/src/MemScanPte.c (new TU; emits HK_EVENT_MEM_WX_DIVERGENCE)`
- **Refs:** PTE NoExecute bit, MmGetPhysicalAddress, MM_EXECUTE_* vs hardware NX, VAD protection field

### 12. On-disk .text vs in-memory .text hash mismatch (module stomping)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean signed module's in-memory executable sections (minus applied relocations and IAT) hash-match the on-disk image; a mismatch in a code section that is not explained by relocation/IAT/hotpatch is module stomping or inline patching.
- **Mechanism:** Driver enumerates the target's loaded modules via PsGetProcessSectionBaseAddress + the documented PEB Ldr InLoadOrderModuleList; for each signed module map the on-disk file (ZwCreateSection/ZwMapViewOfSection of the file backing), normalize relocations using the IMAGE_BASE_RELOCATION table and zero the IAT thunks, then compare section bytes against the live mapping read while attached.
- **Catches:** Module stomping (overwriting a legit DLL's .text with payload while keeping its FILE_OBJECT backing), inline hooks, IAT/EAT patches in code regions.
- **FP risk:** high — Legit hotpatch (Microsoft KB hotpatching), Detours-based accessibility/overlay software, and EAF/ASLR relocation create benign diffs; must exactly model relocation + IAT + /hotpatch pad and allow-list known overlay injectors (Discord, Steam, RTSS), else false positives are severe. Emit raw diff offsets to server; server decides.
- **Horkos slot:** `kernel/win/src/ModuleStomp.c (new TU; HK_EVENT_MEM_MODULE_STOMP carries module hash + first-diff RVA)`
- **Refs:** PsGetProcessSectionBaseAddress, PEB.Ldr InLoadOrderModuleList, IMAGE_BASE_RELOCATION, ZwMapViewOfSection, IMAGE_DIRECTORY_ENTRY_BASERELOC/IAT

### 13. Executable VAD absent from PEB loader list (ghost image)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** In a clean process every image-backed executable VAD has a corresponding entry in the PEB Ldr module lists; an image-backed +X VAD (ControlArea->FilePointer present) with NO matching Ldr entry is a manually-mapped-via-section image hidden from the loader.
- **Mechanism:** Cross-reference the kernel VAD walk (image VADs with a non-NULL FilePointer and section name) against the three PEB Ldr lists (InLoadOrder/InMemoryOrder/InInitializationOrder) read while attached; any image VAD whose base/path is in no list is the anomaly.
- **Catches:** LdrpModuleList unlinking, section-mapped manual loaders (NtCreateSection + NtMapViewOfSection without LdrLoadDll), DLL hollowing that maps but never registers.
- **FP risk:** low — Forwarded/SxS and delay-load corner cases briefly exist pre-link; sample post-init and require the VAD to be fully committed +X. Loader-list teardown during process exit can race — skip processes in exit.
- **Horkos slot:** `kernel/win/src/LdrCrosscheck.c (new TU; HK_EVENT_MEM_GHOST_IMAGE)`
- **Refs:** PEB.Ldr PEB_LDR_DATA, LDR_DATA_TABLE_ENTRY three list heads, VAD ControlArea->FilePointer->FileName, KeStackAttachProcess

### 14. Oversized private executable commit (RWX heap spray ceiling)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean game process keeps private (non-image) executable commit small and JIT-attributable; a single private +X (or +RWX) reservation above a baseline threshold, or aggregate private-exec working set exceeding a per-title ceiling, signals a payload staging region.
- **Mechanism:** During the VAD walk sum committed bytes of private (VadNone) regions carrying EXECUTE protection; read region size from the MMVAD StartingVpn/EndingVpn span and corroborate process-wide private-exec totals against EPROCESS working-set / VM counters reachable via the documented QuerySystemInformation memory counters; emit size + base when a region or the aggregate crosses the signed-rule threshold.
- **Catches:** Heap-spray / large RWX shellcode buffers, staged unpacked payloads, allocation patterns of automation frameworks that reserve big executable scratch.
- **FP risk:** medium — Heavy JIT workloads (browsers embedded via CEF, Lua/Mono game scripting) legitimately hold large +X; threshold must be per-title baselined and JIT-owner-attributed. Ship the number, let the server model decide rather than a hard kernel threshold.
- **Horkos slot:** `kernel/win/src/MemScanVad.c (extends the VAD walk; HK_EVENT_MEM_PRIV_EXEC_COMMIT with size_bytes)`
- **Refs:** _MMVAD StartingVpn/EndingVpn, VAD protection EXECUTE bits, EPROCESS working-set counters, MM_COMMIT

### 15. Large-page / VAD-rotate executable region (anti-scan staging)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Clean game code does not allocate private executable large pages (2MB) nor use VAD-rotate/AWE physical-backed executable mappings; an executable large-page private region or a VadRotatePhysical/VadAwe region with EXECUTE is an evasive staging primitive.
- **Mechanism:** In the VAD walk inspect VadType for VadRotatePhysical / VadAwe / VadLargePages and the large-page flag in the MMVAD flags word; flag any such node whose protection includes EXECUTE and that is not owned by a signed graphics/AWE-using module.
- **Catches:** Manual mappers that use MEM_LARGE_PAGES or AWE/rotate VADs specifically to dodge naive per-4K-page scanners and to get RWX without a ControlArea.
- **FP risk:** low — GPU drivers and DBs (SQL/large-page-aware) use AWE/large pages but rarely +X; allow-list signed graphics/AWE modules owning the VAD. Requires SeLockMemoryPrivilege context which games lack — strong signal.
- **Horkos slot:** `kernel/win/src/MemScanVad.c (VadType branch; HK_EVENT_MEM_EXOTIC_VAD)`
- **Refs:** MMVAD VadType (VadRotatePhysical, VadAwe, VadLargePages), MMVAD_FLAGS large-page bit, MEM_LARGE_PAGES

### 16. Image VAD with mismatched section name vs file path (hollow/doppelgang)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** In a clean process the FILE_OBJECT name behind an image VAD matches the path the loader recorded and the FILE_OBJECT is still openable on disk; a VAD whose backing ControlArea references a transacted/deleted/pending-delete file, or whose section name disagrees with the Ldr-recorded path, indicates process hollowing or transactional (Doppelganging) image swap.
- **Mechanism:** For each image VAD read ControlArea->FilePointer->FileName and the FILE_OBJECT flags; cross-check against the Ldr entry path, and probe the backing via ZwQueryInformationFile / FILE_STANDARD_INFORMATION DeletePending and transaction state (the documented FILE_OBJECT/SCB linkage) to detect a backing file that no longer matches what is mapped.
- **Catches:** Process Doppelganging (NTFS transaction-backed section), process hollowing (NtUnmapViewOfSection + remap), ghostly-deleted backing files.
- **FP risk:** medium — Self-deleting installers and packed launchers legitimately run from a since-replaced file; gate on EXECUTE + entry-point region + delete-pending/transacted backing together, not delete-pending alone.
- **Horkos slot:** `kernel/win/src/HollowDetect.c (new TU; HK_EVENT_MEM_HOLLOW_BACKING)`
- **Refs:** ControlArea->FilePointer, FILE_OBJECT.DeletePending, ZwQueryInformationFile FILE_STANDARD_INFORMATION, NtCreateSection SEC_IMAGE semantics

### 17. Entry-point / TLS-callback region outside any image VAD

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean thread's start address and a clean module's TLS callbacks all resolve inside an image-backed executable VAD; a thread StartAddress or an AddressOfEntryPoint/TLS-callback target that lands in private unbacked memory is injected-code execution.
- **Mechanism:** Read each thread's start address via the documented ThreadQuerySetWin32StartAddress path (or PsGetThreadId + ETHREAD start-address field while attached) and each module's IMAGE_TLS_DIRECTORY callback addresses from the in-memory header; map each target through the VAD tree and flag any that resolves to a non-image / unbacked VAD.
- **Catches:** CreateRemoteThread-style and thread-hijack execution starting in shellcode, TLS-callback hijacks that run before main, APC-injected stubs in private memory.
- **FP risk:** medium — JIT trampolines and some loaders (CLR stubs) legitimately start threads in +X private memory; correlate with the unbacked-VAD signal and JIT-owner attribution before alerting.
- **Horkos slot:** `kernel/win/src/ExecOrigin.c (new TU; HK_EVENT_MEM_EXEC_ORIGIN_ANON), correlates with existing PsSetCreateThreadNotifyRoutine path`
- **Refs:** ThreadQuerySetWin32StartAddress, ETHREAD StartAddress, IMAGE_TLS_DIRECTORY AddressOfCallBacks, VAD lookup

### 18. Section object whose backing file lacks code-signing / EKU provenance

- **Platform / layer:** windows-kernel / kernel
- **Signal:** In a clean process every executable image VAD is backed by a section over a catalog- or embedded-signed PE; an executable image-backed region whose underlying file is unsigned or whose signing chain does not match the expected publisher is a planted module loaded by a legitimate-looking path.
- **Mechanism:** For each image VAD's ControlArea->FilePointer, capture the backing file path and request a signing verdict through the documented kernel image-integrity path (ci.dll CiCheckSignedFile / the SeCodeIntegrityQueryInformation interface, or hand the path/hash to the userspace SDK for WinVerifyTrust) and compare the signer against the per-title allow-list; emit publisher + verdict.
- **Catches:** Side-loaded unsigned DLLs mapped by a trusted loader, signature-stripped stomped modules, BYOVD-adjacent unsigned user modules that slipped a section into the address space.
- **FP risk:** medium — Plenty of legit game and middleware DLLs are unsigned or signed by long-tail publishers; this is a provenance enrichment, not a verdict — server correlates unsigned + unbacked/stomped, never bans on unsigned alone.
- **Horkos slot:** `attestation-adjacent: kernel emits path+hash via kernel/win/src/ModuleStomp.c, signing verdict computed in sdk/src/backends/win (WinVerifyTrust) and shipped as HK_EVENT_MEM_UNSIGNED_IMAGE`
- **Refs:** ControlArea->FilePointer->FileName, CiCheckSignedFile / SeCodeIntegrityQueryInformation, WinVerifyTrust, IMAGE_DIRECTORY_ENTRY_SECURITY

## Windows Kernel — Thread Origin Validation  `win-kernel-thread-injection`

### 19. PreviousMode user-origin on thread create vs caller image lineage

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean game thread is created either by ntdll in-process (RtlUserThreadStart lineage) or by a known launcher/overlay; an injected thread is created while the *creating* thread's call originates from a non-image VAD or a thread whose own start address is unbacked, observable as a creator-side provenance mismatch the in-process notify alone cannot see.
- **Mechanism:** PsSetCreateThreadNotifyRoutineEx (the Ex variant gives PS_CREATE_THREAD_NOTIFY_INFO routine semantics) correlated with PsGetCurrentThreadId/PsGetThreadProcessId at callback time, plus ZwQueryInformationThread(ThreadQuerySetWin32StartAddress) on the *creator* thread, resolved against MmGetSystemRoutineAddress-style VAD lookup via ZwQueryVirtualMemory(MemoryBasicInformation) Type=MEM_IMAGE check.
- **Catches:** Manual-mapped / reflectively-loaded injector creating a thread from unbacked executable memory (the creator itself is not a legitimate module).
- **FP risk:** medium — JIT runtimes (CLR, V8, JVM, .NET ReadyToRun tiering) legitimately create threads from non-image RWX/RX regions; gate by allowlisting signed JIT host modules and by requiring the creator region to be both unbacked AND unsigned AND recently VirtualAlloc'd within the protected process, not the game's own scripting VM region.
- **Horkos slot:** `kernel/win/src/Notify.c (new HkThreadNotifyEx creator-provenance path) + new payload in sdk/include/horkos/ioctl.h`
- **Refs:** PsSetCreateThreadNotifyRoutineEx, PS_CREATE_THREAD_NOTIFY_INFO, ZwQueryInformationThread(ThreadQuerySetWin32StartAddress), ZwQueryVirtualMemory(MemoryBasicInformation), MEM_IMAGE

### 20. ETW-TI ALLOCVM_REMOTE -> SETTHREADCONTEXT -> RESUMETHREAD causality window

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A legitimately spawned thread never has its context rewritten by another process between creation and first run; the classic CreateRemoteThread-suspended / set-context / resume sequence and Early Bird APC appear as a tight cross-process ALLOCVM_REMOTE then SETTHREADCONTEXT then RESUMETHREAD chain against the same target TID within milliseconds.
- **Mechanism:** Microsoft-Windows-Threat-Intelligence (ETW-TI) provider events KERNEL_THREATINT_TASK_ALLOCVM_REMOTE, KERNEL_THREATINT_TASK_SETTHREADCONTEXT, KERNEL_THREATINT_TASK_RESUMETHREAD consumed by a protected (PPL/anti-malware-signed) ETW consumer; correlate by TargetThreadId/TargetProcessId across the three event payloads.
- **Catches:** Thread-hijacking and Early-Bird/suspended-process injection where execution is redirected via context manipulation rather than a fresh remote thread.
- **FP risk:** medium — Debuggers (WinDbg/Visual Studio), Steam overlay bootstrap, and DRM packers legitimately set thread context; gate by requiring the SETTHREADCONTEXT source process to be neither a registered debugger (PROCESS_BASIC_INFORMATION DebugPort/IsBeingDebugged consistent) nor in the signed-overlay allowlist, and require the RESUMETHREAD source != target process.
- **Horkos slot:** `sdk/src/backends/win/EtwTiConsumer.cpp (new protected-consumer module) feeding hk_event_thread_inject records`

### 21. NtQueueApcThread user-APC target outside ntdll/kernel32 dispatcher set

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Legitimate user-mode APCs land on documented dispatcher routines (e.g. ntdll!KiUserApcDispatcher reaching a CRT/alertable-wait callback inside a loaded image); a malicious queued APC points its ApcRoutine at an address inside an unbacked VAD or at a ROP/shellcode stub, and the NtQueueApcThreadEx QUEUE_USER_APC_FLAGS_SPECIAL_USER_APC flag is set for forced delivery.
- **Mechanism:** ETW-TI KERNEL_THREATINT_TASK_QUEUEUSERAPC_REMOTE / QUEUEUSERAPC_REMOTE_API event (ApcRoutine, TargetThreadId, SpecialApc flag) correlated with a MEM_IMAGE check on the ApcRoutine address via the userspace SDK's module map; no documented kernel callback exposes APC enqueue directly, so ETW-TI is the only sanctioned source.
- **Catches:** APC-injection (including NtQueueApcThreadEx special user APC 'no alertable wait needed' variant) used to bootstrap shellcode into the game without a remote thread.

### 22. Wow64 thread Win32StartAddress in 64-bit space on a 32-bit game

- **Platform / layer:** windows-kernel / kernel
- **Signal:** In a WOW64 (32-bit) protected game, every legitimate thread's Win32 start address and TEB-resolved start lie below the 4 GB boundary inside the wow64 image set; an injector that runs native-64 code or redirects the wow64 transition (heaven's gate) produces a thread whose real start address or context CS selector indicates 64-bit execution.

- **Catches:** Heaven's-gate / native-64 shellcode injected into a 32-bit game to evade 32-bit-only scanners, and wow64 transition redirection.
- **FP risk:** low — Pure WOW64 games legitimately have only sub-4GB thread starts; the only benign 64-bit threads in a wow64 process are the wow64cpu/ntdll transition stubs, which sit in known system modules and can be allowlisted by image path.
- **Horkos slot:** `sdk/src/backends/win/ThreadProvenanceWin.cpp (new) + ioctl payload extension`

### 23. TEB.Win32StartAddress vs ETHREAD StartAddress divergence

- **Platform / layer:** windows-kernel / kernel
- **Signal:** On a clean thread the kernel ETHREAD StartAddress (true entry, captured at create) and the user-mode TEB-stored Win32StartAddress agree (modulo the RtlUserThreadStart shim); injectors that spoof Win32StartAddress to a benign-looking value to defeat the basic 'start outside module' check leave the kernel-side StartAddress pointing at the real shellcode, so the two disagree.

### 24. First-instruction page provenance: thread entry in private RX, not image

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean thread's entry page is part of a MEM_IMAGE mapping backed by a signed, on-disk module whose page hashes match; an injected thread enters a MEM_PRIVATE / MEM_MAPPED page (manual-map, module-stomping into a hollowed legit module, or RWX scratch) — distinguished by the page's Type, protection history, and whether its bytes match the on-disk image at that RVA.

- **Catches:** Module stomping / DLL hollowing (thread starts inside a legit module's address range but the bytes were overwritten) and manual-mapped private-memory threads.
- **FP risk:** medium — Self-modifying packers and DRM (VMProtect/Themida/Denuvo) legitimately execute from rewritten image pages, and hot-patch/Detours trampolines alter prologues; gate by excluding the game's own known-packed sections (declared in a signed module manifest) and by only flagging stomped pages whose backing module is a system DLL the game never legitimately rewrites.
- **Horkos slot:** `sdk/src/backends/win/ThreadProvenanceWin.cpp + dma_detect-style page-hash helper reuse`

### 25. Hidden-from-debugger / ThreadHideFromDebugger thread create

- **Signal:** Legitimate game threads are not marked HideFromDebugger; an injected worker frequently sets NtSetInformationThread(ThreadHideFromDebugger) immediately after creation to suppress debug events and breakpoint forensics, observable as a HideFromDebugger flag on a freshly created thread whose start address is already non-image.

### 26. Cross-session / cross-desktop thread creator (Session 0 or alt-desktop injector)

- **Platform / layer:** windows-kernel / kernel

### 27. Thread-creation rate / call-stack-less worker burst on game module

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean game's thread population is stable and threads are created from in-module call sites with a coherent return stack; cheat frameworks that spin up periodic 'tick' worker threads (aim/ESP loops) produce bursts of short-lived or periodic threads whose creation call stack has no frame inside a legitimate game/system module, observable as a creator-stack-unbacked rate anomaly.

- **FP risk:** high — Thread pools, GC, async runtimes, and game engines create many short-lived threads, and overlays/streaming tools (OBS, Discord) add bursts; gate by requiring the *creator stack* to be fully unbacked (not merely a high rate), reporting raw events to the server and letting the trust model weigh rate against the unbacked-stack provenance rather than thresholding on the client.

### 28. FltMgr callback-node altitude gap audit

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean minifilter stack has every registered FLT_INSTANCE callback node owned by a signed FltRegisterFilter caller at a Microsoft-allocated altitude; a rogue node sits at an unregistered/duplicated altitude or has a callback pre-op pointer outside any registered filter image, which the baseline IRP-table check never inspects because FltMgr dispatch is layered above DRIVER_OBJECT.MajorFunction.

- **Catches:** Filter-manager-layer hooking and altitude collision: a cheat that registers a fake minifilter or splices a legitimate filter's pre/post-op callbacks to intercept file/registry I/O above the classic dispatch table.
- **FP risk:** medium — Legitimate AV/EDR/backup/encryption minifilters (Defender, CrowdStrike, Carbon Black, Veeam, controlled-folder-access) register many instances and dynamically-allocated callback thunks. ### 29. Driver image .text section hash vs. on-disk authenticode

- **Platform / layer:** windows-kernel / kernel
- **Signal:** For each loaded driver, the in-memory code sections (minus legitimate IAT/relocation fixups) hash to the same value as the on-disk PE's signed sections; a divergence means the driver image was patched in memory after load (inline kernel hooks, detoured exports) even though the loaded-driver list and its disk hash both look clean.
- **Mechanism:** Walk PsLoadedModuleList via AuxKlibQueryModuleInformation / ZwQuerySystemInformation(SystemModuleInformation) to get DllBase+ImageSize, parse the in-memory PE headers to locate executable sections, normalize relocations/IAT using the on-disk copy opened via ZwCreateFile + ZwReadFile, then compare a per-section hash against the on-disk file's section hash and its embedded WIN_CERTIFICATE.
- **Catches:** In-memory code patching / inline hooking of an already-loaded legitimate signed driver (the disk file and module-list entry are untouched, so blocklist + Authenticode-on-disk checks pass).
- **FP risk:** medium — Hotpatch/KB live-patching, kernel shim engine, and some HVCI-incompatible drivers legitimately self-modify; Microsoft hotpatch images and retpoline/CFG thunk rewrites at boot also differ from disk. Gate by excluding the hotpatch region descriptors, normalizing against known fixup types, and treating only NON-relocation, NON-import-thunk code-byte deltas as anomalies; report the delta offset+image to the server, never auto-ban.
- **Horkos slot:** `kernel/win/src/ImageHashAudit.c (new); on-disk read helper guarded behind PASSIVE_LEVEL work item`
- **Refs:** ZwQuerySystemInformation(SystemModuleInformation), AuxKlibQueryModuleInformation, RtlImageNtHeader, IMAGE_SECTION_HEADER, WIN_CERTIFICATE, ZwCreateFile/ZwReadFile

### 30. CI!g_CiOptions / code-integrity state probe

- **Platform / layer:** windows-kernel / kernel
- **Signal:** On a clean machine the kernel reports DSE enabled, test-signing off, and HVCI/KMCI consistent with boot config; a manipulated CI state (g_CiOptions cleared, ci.dll callbacks neutered, or SeValidateImageData bypassed) lets unsigned drivers load without ever appearing as 'test-signed', which the baseline unsigned/test-signed presence check cannot see once the validator itself is the thing disabled.

### 31. Registered driver-callback table residency check

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Every OS-maintained callback registration (process/thread/image notify, Ob callbacks, registry CmRegisterCallback, bug-check, power) points into a loaded module's image range; an entry whose handler address falls in unbacked/private pool or an unknown image is a rogue registrant or a hijacked slot — distinct from the baseline which only registers its OWN callbacks, never audits the global tables.
- **Mechanism:** Enumerate PspCreateProcessNotifyRoutine / PspLoadImageNotifyRoutine / CmCallbackListHead-equivalents indirectly through the documented enumeration surfaces where available, plus self-test: register a sentinel callback and verify our own slot is intact; for each discoverable handler, resolve the address against the SystemModuleInformation image map and flag any pointer with no backing image.

- **FP risk:** medium — EDR/AV legitimately register many notify and registry callbacks; some load handlers into pool-allocated trampolines. ### 32. Driver gap / non-image executable allocation scan

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Legitimate kernel code executes only from image-backed sections owned by a loaded driver; a region of system-address-space memory that is executable but maps to NO entry in PsLoadedModuleList (a 'manually-mapped' driver loaded by a vulnerable signed driver) is the residue of BYOVD code injection that leaves no module-list entry and thus passes every list-based check including the implemented blocklist.
- **Mechanism:** Compare the loaded-module address ranges (SystemModuleInformation) against system PTE / big-pool ownership: enumerate large nonpaged allocations via ZwQuerySystemInformation(SystemBigPoolInformation) and the system region map, and flag executable system-space ranges whose [base,base+size) intersects no module image and no known pool-tag owner.

- **FP risk:** high — JIT'd kernel code is rare but legitimate executable nonpaged allocations exist (some AV engines, .NET kernel components, GPU drivers' command buffers). Gate hard: require executable+private (non-image) classification, correlate with SystemBigPoolInformation tag ownership, sample rather than enumerate every PTE, and treat as a server-side scored anomaly with image-map corroboration — never a standalone ban signal.

### 34. Driver-object DeviceObject / FastIo dispatch divergence

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Beyond the MajorFunction table, a clean DRIVER_OBJECT has its FastIoDispatch table entries, DriverStartIo, DriverUnload, and the DriverInit/DriverSection pointers all resolving inside the owning image; a FastIoDispatch->FastIoDeviceControl or AddDevice pointer detoured outside the image is a hook the implemented MajorFunction-only check misses.

### 35. Shadow SSDT / KiServiceTable range integrity

### 36. WdFilter / ELAM-ordering and boot-start driver-load sequence audit

## Windows — Hypervisor / Virtualization State  `win-hypervisor-detection`

### 37. Hyper-V TLFS partition-privilege divergence

- **Platform / layer:** windows-user / hypervisor
- **Signal:** On a genuine Microsoft Hyper-V/VBS host the synthetic hypervisor CPUID leaves 0x40000000-0x4000000A are internally self-consistent (vendor='Microsoft Hv', interface signature 0x31237648 'Hv#1', and the partition-privilege/feature bitmaps in 0x40000003/04 match what the running Windows build exposes); a thin out-of-OS inspection hypervisor that only fakes the vendor leaf leaves these downstream leaves zeroed, truncated, or feature-inconsistent with the OS-reported VBS state.

- **Catches:** Out-of-OS / nested inspection hypervisor that spoofs only the CPUID vendor leaf to masquerade as Hyper-V while the deeper TLFS leaves and OS VBS state disagree.
- **FP risk:** medium — Legitimate nested virtualization (Hyper-V in a VM, WSL2, Windows Sandbox, Credential Guard, Cloud PC/Azure) all present real Hyper-V leaves; gate by treating leaf-consistency as the signal (report the full leaf vector + OS VBS flags to the server, which models the known-good nested-Hyper-V vectors) rather than presence/absence of a hypervisor.

### 38. Serialized-instruction vmexit latency fingerprint

- **Platform / layer:** windows-user / hypervisor
- **Signal:** CPUID is an unconditional vmexit; on bare metal or genuine Hyper-V the CPUID round-trip distribution measured between two serializing RDTSCP reads is tight and clusters at one mode, whereas an additional inspecting hypervisor underneath adds a second, higher latency mode (double vmexit) producing a bimodal / right-shifted histogram that survives even when the attacker clamps the visible TSC.
- **Mechanism:** Usermode RDTSCP/RDTSC + __cpuid timing loop with SetThreadAffinityMask pinning and SetThreadPriority(TIME_CRITICAL); collect a latency histogram (not a single delta) and additionally read QueryPerformanceCounter and the KUSER_SHARED_DATA InterruptTime as an independent clock to detect TSC-offset/scaling tampering.
- **Catches:** A hidden/nested hypervisor performing exit interception; also TSC-offsetting used to hide instrumentation timing.
- **FP risk:** high — Power management (SpeedStep/turbo), SMM interrupts, noisy-neighbor cloud hosts, and legitimate Hyper-V all perturb latency; gate by shipping the full histogram + CPU model + power state to the server for population-relative modeling, never a hardcoded threshold, and require divergence between TSC and an independent clock before flagging.
- **Horkos slot:** `ac/src/hv/hv_vmexit_latency.cpp declared in ac/include/horkos/hv_signals.h`
- **Refs:** RDTSCP, __cpuid, QueryPerformanceCounter, KUSER_SHARED_DATA.InterruptTime, SetThreadAffinityMask

### 39. EPT exec/read permission-split via self-modifying probe

- **Platform / layer:** windows-kernel / hypervisor
- **Signal:** A clean page returns identical bytes whether the CPU fetches it for execution or reads it as data; an EPT/SLAT permission-split (used to hide a patched code page behind a clean read view) makes the executed instruction stream diverge from the data-read view of the same physical page, observable as a value that differs between a function's runtime result and a data read of its bytes, or as an unexpected #VE.
- **Mechanism:** KMDF driver computes a checksum of a monitored code region two ways - by reading the bytes (MmGetPhysicalAddress + mapped read) and by an independent integrity hash of the loaded image on disk/section - and additionally arms #VE (EXCEPTION_VIRTUALIZATION_FAULT, STATUS 0xC0000420) handling expectations; divergence between executed-vs-read view of the same VA/PA is the signal. Reported over the existing SPSC ring -> IOCTL bridge.
- **Catches:** EPT-based code-patch hiding (split exec/read TLB views) that defeats naive in-place CRC of code pages.
- **FP risk:** medium — Legitimate shadow-stack/CET, Hyper-V HVCI/Code Integrity, and some DRM (e.g. anti-tamper that legitimately remaps) can introduce exec/read asymmetry; gate by whitelisting HVCI-managed regions (cross-check SystemCodeIntegrityInformation) and only flagging splits over the game's own signed sections.
- **Horkos slot:** `kernel/win/src/HvEptProbe.c (kernel sensor) with schema field in event schema; userspace correlation in ac/src/hv/hv_ept_split.cpp`
- **Refs:** MmGetPhysicalAddress, EXCEPTION_VIRTUALIZATION_FAULT (#VE, 0xC0000420), NtQuerySystemInformation(SystemCodeIntegrityInformation), MmMapIoSpaceEx

### 45. Cross-vCPU TSC coherence and invariant-TSC consistency

- **Platform / layer:** windows-user / hypervisor
- **Signal:** On genuine hardware/Hyper-V the TSC is invariant and synchronized across logical processors so reads pinned to different vCPUs advance monotonically and stay within a tight skew; an inspection hypervisor that applies per-vCPU TSC offsetting/scaling to hide its own execution time produces inconsistent cross-vCPU skew or a TSC that contradicts the invariant-TSC capability advertised by CPUID 0x80000007 EDX[8].
- **Mechanism:** Usermode pins a thread to each logical processor (SetThreadAffinityMask over GetLogicalProcessorInformationEx) and samples RDTSCP (whose IA32_TSC_AUX returns the CPU id, confirming the pin) in a tight cross-CPU schedule, comparing skew and monotonicity against the invariant-TSC bit from __cpuid(0x80000007); skew anomalies or capability/behavior contradiction is the signal.
- **Catches:** Per-vCPU TSC offset/scaling manipulation used by a hidden hypervisor to defeat the existing RDTSC vmexit-latency timing checks.
- **FP risk:** medium — Older multi-socket hardware, deep C-states, and some cloud hosts have legitimate cross-core TSC skew; gate by reading CPUID invariant-TSC first, using RDTSCP's CPU-id to confirm pinning actually took effect, and modeling skew distributions server-side per CPU SKU rather than a fixed tolerance.

### 58. Inter-report timing entropy collapse

- **Platform / layer:** windows-user / userspace
- **Signal:** The distribution of inter-arrival deltas between consecutive WM_INPUT mouse-move/button reports from a given hDevice shows quantization to a fixed period or near-zero variance/jitter inconsistent with the human neuromuscular tremor and polling-aliasing of real HID input.

- **Catches:** Scripted/automated input (recoil scripts, triggerbots, macro engines) that emits reports on a fixed timer or with synthetic Perlin/uniform jitter lacking the spectral signature of a physical sensor at its true polling rate.
- **FP risk:** high — High-polling-rate mice (1000-8000Hz) genuinely produce very regular deltas, and frame-locked/vsync-driven input can alias to fixed periods; gate by combining timing with the polling-rate the device reports and by requiring the regularity to persist across context switches and aim phases — ship features to the server model, never a client-side ban, because timing alone is too noisy.

### 64. Sub-PAGE_GUARD WriteVm cadence on code sections

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean game never receives external NtWriteVirtualMemory into its own +X code sections; a trickle of small foreign writes whose target VA resolves inside an IMAGE_SCN_MEM_EXECUTE region of a known loaded module is the patch-write fingerprint of a code cave / inline hook.

### 65. Cross-process working-set residency probe (foreign page touch)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A scanner that walks our address space leaves a fingerprint: pages of our private/image VADs that were paged out get faulted back into our working set with no corresponding access by our own threads, observable as a working-set growth burst uncorrelated with game thread activity.
- **Mechanism:** QueryWorkingSetEx (PSAPI_WORKING_SET_EX_INFORMATION.Shared/ShareCount/Valid) sampled from the userspace SDK against the game's own working set, cross-referenced with the driver's KTHREAD CPU-time deltas; a residency delta with zero owning-thread CPU implies a foreign reader (MmCopyVirtualMemory/ReadProcessMemory) faulted the pages in.
- **Catches:** full-address-space external memory scanning (signature/pattern scanners enumerating our heap and image)
- **FP risk:** high — The OS prefetcher, memory compression, and Superfetch repopulate working sets on their own; also our own GC/JIT or texture streaming faults pages. Gate by requiring the residency burst to coincide with an ETW-TI ReadVm event or an Ob handle-open with PROCESS_VM_READ from a foreign PID within the same sampling window, otherwise discard.
- **Horkos slot:** `sdk/src/backends/win/WorkingSetWatchWin.cpp (new; userspace sampler reporting via SDK)`
- **Refs:** QueryWorkingSetEx, PSAPI_WORKING_SET_EX_BLOCK, K32QueryWorkingSetEx, GetProcessTimes for thread-CPU correlation

### 66. Handle-duplication provenance chain to game object

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean opener acquires a game handle directly; an evasive tool launders access by DuplicateHandle from a third broker process (or from csrss/lsass leaked handles), so the OB_OPERATION_HANDLE_DUPLICATE source-process != the requester and the duplicated handle's granted access includes VM_READ/VM_WRITE/VM_OPERATION targeting our PID.
- **Mechanism:** ObRegisterCallbacks pre-op on OB_OPERATION_HANDLE_DUPLICATE reading DuplicateHandleInformation.SourceProcessId/TargetProcessId/OriginalDesiredAccess; correlate the source PID against a driver-maintained map of who previously opened our process, flagging dup chains whose root opener never appeared in our create-handle log.
- **Catches:** handle laundering / brokered access (access obtained via a privileged or unrelated process then duplicated to the cheat)
- **FP risk:** medium — Legit launchers/anti-cheat-adjacent tools, debuggers, and the Windows Error Reporting/crash-dump path duplicate process handles; gate on the duplicated access mask containing VM_WRITE or VM_OPERATION AND the source process being unsigned / not on the platform allow-list, and exclude WerFault.exe and our own launcher.

### 67. Granted-vs-requested access delta on our process object

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Compare OriginalDesiredAccess (true intent) against the access another higher-altitude callback or a stealth driver actually grants; a foreign tool that suppresses our rights-stripping leaves a mismatch where the granted handle carries MORE access than any callback in the chain should have allowed, indicating a competing Ob callback at a different altitude tampering with the access mask.
- **Mechanism:** ObRegisterCallbacks post-operation callback (PostOperation, currently NULL) reading the final granted ACCESS_MASK in the OB_POST_OPERATION_INFORMATION, diffed against what HkObPreCallback recorded for the same handle-create; a divergence implies another callback above/below us altered DesiredAccess.
- **Catches:** competing/hostile Ob callback or driver that strips our protection (rights-reduction suppression, BYOVD that unhooks Ob)
- **FP risk:** low — Other legitimate security products (AV/EDR) also register Ob callbacks and legitimately reduce access; a grant LARGER than our pre-op desired is the anomaly, not a reduction. Gate strictly on granted > pre-op recorded mask, never on reductions.

### 68. PsLookupProcessByProcessId polling signature via PID-reuse canary

### 69. MmCopyVirtualMemory via foreign kernel caller (non-ETW path)

- **Platform / layer:** windows-kernel / kernel

### 70. Self-handle table audit for unexpected VM-rights holders

### 72. ETW-TI ProtectVm/AllocVm burst preceding foreign writes

- **Platform / layer:** windows-kernel / kernel
- **Signal:** The temporal order is diagnostic: a foreign process issuing NtAllocateVirtualMemory then NtProtectVirtualMemory then NtWriteVirtualMemory against our PID within a tight window is the classic remote-shellcode/manual-map staging sequence; the clean baseline shows none of these cross-process Vm ops targeting us.

### 73. ptrace_access_check authorization-context capture

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A clean game process is never the target of a successful ptrace mode-attach (PTRACE_MODE_ATTACH) authorization from a non-parent, non-debugger task; the security_ptrace_access_check pass on the protected task is the tamper observable, captured with full cred context rather than just the raw syscall args the sys_enter_ptrace baseline sees.
- **Mechanism:** BPF LSM hook SEC("lsm/ptrace_access_check") (BPF_PROG with struct task_struct *child, unsigned int mode). - **Catches:** Foreign debugger / scanner attaching to the protected process; distinguishes an authorized PTRACE_MODE_READ peek (e.g. a profiler) from a PTRACE_MODE_ATTACH write-capable seize, which the sys_enter_ptrace tracepoint cannot since it only sees the request code before the kernel's permission decision.
- **FP risk:** medium — gdb/lldb/strace, Proton/Wine's own ptrace-based thread management, EasyAntiCheat/BattlEye-in-Proton, and crash handlers (breakpad/sentry) legitimately ptrace. Gate by allow-listing the caller's exe path/cgroup (parent is the Steam/Proton supervisor) and by mode: only ATTACH-with-write from a caller outside the game's own process tree escalates.

- **Refs:** security_ptrace_access_check, ptrace_access_check LSM hook, PTRACE_MODE_ATTACH/PTRACE_MODE_READ (include/linux/ptrace.h), BPF_PROG_TYPE_LSM

### 74. process_vm_writev foreign-writer fexit byte accounting

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** No legitimate party writes into the game's address space via process_vm_writev(2); any successful cross-mm write where the target mm is the protected process and the caller is foreign is a near-certain external memory write. fexit gives the actual bytes-transferred return value, so partial/probing writes are quantified rather than just intent-logged.
- **Mechanism:** fexit/fentry on the kernel implementation: SEC("fexit/process_vm_writev") (or kprobe on process_vm_rw with vm_write=1). Resolve the target task from the pid arg, compare target->mm to the protected mm; read the long return (bytes written). bpf_get_current_task_btf() identifies the writer. Pair with fexit/process_vm_readv for the read direction beyond the basic ptrace path.
- **Catches:** External cheat trainers and overlay tools that use process_vm_writev for ptrace-less memory writes (the modern alternative to PTRACE_POKE), including byte-count fingerprinting of repeated small writes to the same region.
- **FP risk:** low — Very few legitimate writers: CRIU (checkpoint/restore) and some debuggers' fast-memory paths. CRIU runs at restore time on a frozen task, not during live gameplay; gate on "target is actively scheduled / has a live ES session" and allow-list the CRIU/debugger cgroup. Reads (process_vm_readv) are noisier (htop-like tools) and should be a separate lower-severity signal.
- **Horkos slot:** `kernel/linux/bpf/src/fexit_process_vm.bpf.c (new); HK_EVENT_VM_WRITE / HK_EVENT_VM_READ tags`
- **Refs:** process_vm_writev(2), process_vm_readv(2), mm/process_vm_access.c process_vm_rw, BPF fexit/BPF_PROG_TYPE_TRACING

### 75. mem_open / proc_mem foreign opener via fentry on the proc mem fop

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** /proc/<game-pid>/mem is opened by the kernel's mem_open file operation; a clean system only opens the game's own /proc/self/mem (same tgid) — any open of the protected pid's mem inode from a different tgid is the precise tamper observable, captured at the proc fop rather than inferred from a path string in lsm/file_open.
- **Mechanism:** fentry/fexit on fs/proc/base.c mem_open (or kprobe). The struct file's inode backs a proc_mem_inode; recover the target pid from the proc pid_entry (task = get_proc_task / read the proc_inode->pid). Compare target tgid to protected set; caller is bpf_get_current_pid_tgid()>>32. This is path-independent so it survives bind-mounts, chroots, and renamed /proc views that defeat the string-based lsm/file_open baseline.
- **Catches:** Memory editors (scanmem/GameConqueror, Cheat Engine-for-Linux, edit-via-/proc/pid/mem) that open the mem pseudo-file to seek+pread/pwrite arbitrary offsets; the open is the choke point regardless of subsequent read vs write.
- **FP risk:** low — Debuggers and ASAN/leak tooling read /proc/self/mem (same tgid — excluded). Cross-tgid openers are rare: scanmem, gdb attached remotely, some profilers. Gate by allow-listing debugger cgroups and treating same-tgid self-reads as benign; only foreign-tgid opens of the protected pid escalate.
- **Horkos slot:** `kernel/linux/bpf/src/fentry_proc_mem.bpf.c (new); HK_EVENT_PROC_MEM_OPEN tag`
- **Refs:** fs/proc/base.c mem_open / proc_mem_operations, proc_pid_get_task, BPF fentry, struct proc_inode->pid

### 76. RWX/W^X-violation mapping inside the protected mm (lsm/mmap_file + lsm/file_mprotect)

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A correctly built game binary, after dynamic-linker relocation, holds no anonymous PROT_WRITE|PROT_EXEC mapping and never mprotect()s an existing executable text page back to writable; an in-process RWX region or a W^X-violating mprotect on the game's own VMAs is the signature of inline-hook/trampoline patching.
- **Mechanism:** BPF LSM SEC("lsm/mmap_file") and SEC("lsm/file_mprotect") filtered to current->mm == protected mm. Inspect prot for (PROT_WRITE|PROT_EXEC) on mmap, and for file_mprotect read the target vma's vm_flags vs the requested prot to catch text->writable transitions. Audit-only (return ret). Timestamp relative to process start so post-link patches are separable from loader activity.
- **Catches:** In-process code patching: inline hooks, detour trampolines, JIT-stomping of game text, and self-injected shellcode stubs that must briefly become RWX or flip text writable — the classic same-process cheat that no cross-process ptrace/vm_writev signal sees.
- **FP risk:** high — JITs (the game's own V8/LuaJIT/Mono/IL2CPP, Wine/Proton's code generator, GPU shader JIT, BPF JIT) legitimately make RWX or W->X->W transitions. Gate hard: only fire on mprotect of VMAs backed by the game's main executable inode (file-backed text), exclude anonymous JIT arenas and known-runtime mappings, and learn the per-title baseline of legitimate W^X flips during a clean attested boot before treating any as anomalous.
- **Horkos slot:** `kernel/linux/bpf/src/lsm_mmap_mprotect.bpf.c (new); HK_EVENT_RWX_MAP / HK_EVENT_WX_FLIP tags; baseline list in server signed-rule plumbing`
- **Refs:** security_mmap_file, security_file_mprotect LSM hooks, PROT_EXEC/PROT_WRITE (mman.h), vma->vm_flags VM_WRITE|VM_EXEC, kernel/security/security.c

### 77. memfd_create -> fexecve/execveat(AT_EMPTY_PATH) fileless-exec correlation

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A memfd opened by a process in or adjacent to the game's tree and then executed via execveat with AT_EMPTY_PATH (fd-based, no on-disk path) is fileless execution; the joined memfd_create->write->execveat sequence is the observable that no single-syscall hook surfaces.
- **Mechanism:** tracepoint syscalls/sys_enter_memfd_create (record fd + MFD_* flags, current tgid) keyed into a per-task BPF LRU hash, then tracepoint/LSM lsm/bprm_creds_for_exec (or fentry on do_execveat_common) to detect exec of an fd whose backing file's path begins with "/memfd:" / is an anonymous shmem inode with no dentry path. Correlate by (tgid, inode) across the two hooks in the ringbuf consumer.
- **Catches:** Fileless loader stages and unpacked cheat payloads that never touch disk — built in an anonymous memfd and executed in place to evade lsm/file_open and disk-scan integrity checks; also MFD_CLOEXEC + sealing patterns used to hide the staged blob.
- **FP risk:** medium — Legitimate users of memfd+exec exist: glibc/systemd sealed-memfd patterns, some sandboxing runtimes, and a few language toolchains. Gate on exec-of-memfd specifically (memfd_create alone is common and benign) and allow-list known runtime inodes; the rare legitimate fileless-exec on a gaming desktop is allow-listable per cgroup.
- **Horkos slot:** `kernel/linux/bpf/src/memfd_exec.bpf.c (new) + correlation in Loader.cpp; HK_EVENT_MEMFD_CREATE / HK_EVENT_FILELESS_EXEC tags`
- **Refs:** memfd_create(2), execveat(2) AT_EMPTY_PATH, fexecve(3), security_bprm_creds_for_exec LSM hook, do_execveat_common, MFD_CLOEXEC/MFD_ALLOW_SEALING

### 78. /dev/mem, /dev/kmem and /proc/kcore physical-memory window opens

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** On a hardened desktop these physical-RAM windows are normally compiled out or locked (CONFIG_STRICT_DEVMEM, lockdown); any successful open of /dev/mem, /dev/kmem, or /proc/kcore while the protected game runs is a physical-memory side-channel attempt, identified by device major/minor and proc inode rather than a spoofable path.
- **Mechanism:** BPF LSM lsm/file_open filtered by the opened inode's i_rdev (MEM major 1: minor 1=/dev/mem, minor 2=/dev/kmem) read via BPF_CORE_READ(file, f_inode, i_rdev), plus a check for the kcore proc inode. Cross-check the kernel-lockdown state via the security_locked_down hook to know whether the open should even be possible.
- **Catches:** Kernel-/physical-memory reading paths used to dump or patch the game's pages out from under userspace protections, and DKOM-adjacent tooling reading kernel structures to locate the game's mm — complementary to the existing DMA detector but for the software /dev/mem route.
- **FP risk:** low — Legitimate /dev/mem users are rare on a gaming box: X11 on legacy non-KMS setups, some firmware/BIOS tools, hwinfo utilities. /proc/kcore is read by crash, makedumpfile, perf-on-kernel, and live forensics. Gate by device id + reader cgroup allow-list; most modern Wayland/KMS gaming systems never touch these, making a foreign open high-signal.
- **Horkos slot:** `kernel/linux/bpf/src/lsm_devmem.bpf.c (or extend lsm_file_open with an i_rdev branch); HK_EVENT_PHYSMEM_OPEN tag`
- **Refs:** /dev/mem & /dev/kmem (drivers/char/mem.c, MEM_MAJOR=1), /proc/kcore, CONFIG_STRICT_DEVMEM, security_locked_down LOCKDOWN_DEV_MEM, inode->i_rdev

### 79. Foreign mmap of the game's own file-backed text/data inode

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** Only the game process maps the game executable's and its private data files' inodes; a different tgid calling mmap on the game binary's inode (to read or shared-map its text/data for offset discovery or signature scraping) is anomalous, observed at the inode level so it is independent of the path used to reach the file.
- **Mechanism:** BPF LSM lsm/mmap_file: read file->f_inode i_ino and i_sb->s_dev, compare against the protected binary's (dev,inode) recorded at attest time in a BPF map; flag when the mapping task tgid differs from the protected tgid. Distinguish MAP_SHARED vs MAP_PRIVATE and PROT_READ-only scraping from PROT_WRITE.
- **Catches:** Static-analysis / pattern-scan stages that mmap the game binary from a separate helper process to compute byte-pattern offsets for later in-memory patching, and shared-mapping tricks that alias the game's data pages into a foreign address space.
- **FP risk:** medium — Steam, the Proton/Wine loader, ldconfig, file indexers (tracker/baloo), antivirus, and backup tools all mmap or read game files legitimately. Gate by excluding the platform launcher cgroup and read-only opens during the known load window; escalate only on foreign MAP_PRIVATE+later-mprotect or mid-session mappings of the main executable inode.
- **Horkos slot:** `kernel/linux/bpf/src/lsm_mmap_mprotect.bpf.c (shared with signal 4); HK_EVENT_FOREIGN_MAP tag; protected (dev,inode) set from attestation/backends/linux`
- **Refs:** security_mmap_file LSM hook, inode->i_ino, super_block->s_dev, MAP_SHARED/MAP_PRIVATE (mman.h), struct file->f_inode

### 80. vma-iter executable-region inventory drift (BPF task/vma iterator)

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** The set of executable VMAs in the game's mm (their start/end, vm_flags, and backing inode) is stable after load; periodically iterating the protected task's maps in-kernel and diffing against the attested baseline catches injected executable regions, hidden/unlinked code mappings, and flag changes that a per-event hook would miss if the cheat raced or batched its mprotects.
- **Mechanism:** BPF iterator program SEC("iter/task_vma") (bpf_iter, kernel >=5.13) driven on a timer/userspace trigger: walk vma's of the protected task, emit (vm_start, vm_end, vm_flags, file dev/inode, anon flag) for every VM_EXEC region. Userspace diffs against the baseline snapshot taken right after the dynamic linker settled. Complements the event hooks by providing a current-state ground truth resistant to missed/ordered events.
- **Catches:** Injected .so / manual-map regions, anonymous executable blobs, and text-region permission drift that accumulated via batched or hook-evading operations — a state-based detector orthogonal to the edge-triggered mmap/mprotect signals.
- **FP risk:** medium — Legitimately growing executable surface: JIT runtimes (Mono/IL2CPP/LuaJIT/V8), dlopen of game plugins/mods, and Proton's on-demand code emission expand VM_EXEC VMAs at runtime. Gate by per-title baselining of expected JIT arenas and an allow-list of dlopen'd plugin inodes; only unaccounted anonymous VM_EXEC regions or text-inode flag changes escalate.
- **Horkos slot:** `kernel/linux/bpf/src/iter_task_vma.bpf.c (new) + snapshot/diff in Loader.cpp; HK_EVENT_VMA_DRIFT tag; baseline from attestation backend`
- **Refs:** bpf_iter task_vma (BPF_PROG_TYPE_TRACING / SEC("iter/task_vma")), bpf_iter_task_vma, vm_area_struct vm_flags VM_EXEC, struct mm_struct, kernel >=5.13

### 81. PTRACE_TRACEME self-debug pre-arm and ptrace_traceme LSM

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A clean game does not call PTRACE_TRACEME on itself except via its known crash-handler; an unexpected traceme (the anti-anti-debug trick of self-attaching to block other debuggers, or a launcher pre-arming a tracer relationship) sets a tracer on the protected task and is observable at the dedicated ptrace_traceme LSM hook with the future-tracer's context.
- **Mechanism:** BPF LSM SEC("lsm/ptrace_traceme") (BPF_PROG with struct task_struct *parent) — fires when current requests PTRACE_TRACEME naming parent as its tracer. Record current==protected tgid, parent tgid/exe (the would-be tracer), and the int ret decision. Pair with reading task->ptrace / task->parent in a periodic check to confirm whether a tracer is currently attached.
- **Catches:** Self-ptrace anti-tamper-evasion (cheat occupies the single ptrace slot so AC/debuggers can't attach) and launcher-injected tracer relationships established before the game's own protection initializes — distinct from the inbound-attach signal because here the protected process itself initiates the relationship.
- **FP risk:** medium — Crash handlers (breakpad/crashpad, Proton's debug stubs) and some sandboxes legitimately PTRACE_TRACEME. Gate by allow-listing the known crash-handler call site / parent exe and the documented Proton launcher; a traceme naming an unexpected parent, or occurring after init when no crash handler should arm, escalates.
- **Horkos slot:** `kernel/linux/bpf/src/lsm_ptrace.bpf.c (shared with signal 1); HK_EVENT_PTRACE_TRACEME tag`
- **Refs:** security_ptrace_traceme, ptrace_traceme LSM hook, PTRACE_TRACEME (ptrace.h), task_struct->ptrace/->parent, BPF_PROG_TYPE_LSM

## Linux eBPF — Userspace Loader / Injection  `linux-ebpf-injection`

### 82. DT_NEEDED vs runtime DSO set divergence

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** The set of shared objects mapped into the game (from /proc/self/maps) contains a library that is not transitively reachable from the executable's and its dependencies' DT_NEEDED entries, revealing a force-loaded/preloaded DSO that the static link graph never references.
- **Mechanism:** uprobe on the game's own ld.so via a uretprobe on _dl_map_object (glibc ld-linux) to capture each DSO as the loader maps it, cross-referenced against the static DT_NEEDED closure parsed once at attach from the ELF .dynamic section read through the userspace loader's open of /proc/<pid>/exe; ringbuf carries the offending soname + build-id.
- **Catches:** LD_PRELOAD / dlopen-based library injection where the injected DSO has no link-graph provenance (cheat hook libs, function-trampoline frameworks).
- **FP risk:** medium — Legitimate audio/input/graphics shims (PipeWire/PulseAudio modules, OBS game-capture vklayer, Steam's gameoverlayrenderer.so, MangoHud, vkBasalt, Feral GameMode) also load via LD_PRELOAD/implicit Vulkan layer discovery with no DT_NEEDED edge. Gate with a server-side signed allowlist of soname+build-id for the known overlay/layer ecosystem, scoped per distro/Steam runtime, and only score DSOs whose build-id is unknown AND whose path is outside the Steam runtime / system lib dirs.
- **Horkos slot:** `kernel/linux/bpf/src/dl_map_object.bpf.c (uprobe set) + needed-closure parser in kernel/linux/userspace/DsoProvenance.cpp`
- **Refs:** glibc _dl_map_object / dl-load.c; uprobe BPF_PROG_TYPE_KPROBE w/ SEC("uprobe/"); ELF DT_NEEDED (elf(5)); /proc/pid/maps

### 83. GOT entry pointing outside the resolving DSO's own PLT-target space

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** A sampled .got/.got.plt slot in the protected process resolves to an address that falls outside the [base,end) VMA of any legitimately-mapped library that exports that symbol — i.e. the relocation target lands in an anonymous/RWX mapping or a foreign DSO, the fingerprint of GOT/PLT hooking.
- **Mechanism:** Periodic uprobe-triggered sampling: a uprobe on a benign, frequently-called game function fires a BPF program that bpf_probe_read_user's the game's .got.plt slots (offsets computed userspace-side from the ELF section headers + load bias) and compares each pointer against the VMA map snapshot maintained from _dl_map_object events; slots resolving into anon/RWX or unexpected-module ranges are reported.
- **Catches:** GOT/PLT redirection hooks (the classic ELF import-table patch used to intercept libc/engine calls without touching .text).
- **FP risk:** medium — Legitimate interposers re-point GOT slots too: AddressSanitizer/TSan interceptors, glibc IFUNC resolvers (which legitimately land in the resolving DSO but at runtime-chosen targets), profilers (LD_PROFILE), and Vulkan/GL layer dispatch. Gate by: ignore IFUNC slots (flagged in reloc type R_*_IRELATIVE), allowlist targets that resolve into a signed-known overlay DSO, and only score targets landing in writable+executable or unbacked anonymous memory.
- **Horkos slot:** `kernel/linux/bpf/src/got_sample.bpf.c + section-offset computation in kernel/linux/userspace/GotPltMap.cpp`
- **Refs:** ELF .got.plt / R_X86_64_JUMP_SLOT, R_*_IRELATIVE; bpf_probe_read_user; uprobe sampling; /proc/pid/maps VMA permissions

### 84. PT_INTERP mismatch between on-disk ELF and the actually-mapped loader

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** The dynamic loader mapped into the running game (the r-xp VMA whose backing file is the ELF interpreter) differs in path or build-id from the PT_INTERP string recorded in the game binary's program headers, indicating interpreter substitution (a patched/shim ld.so that performs injection or relocation tampering).
- **Mechanism:** At sched_process_exec for the protected image, the userspace loader reads PT_INTERP from /proc/<pid>/exe program headers; separately it identifies the mapped interpreter by scanning /proc/<pid>/maps for the first executable file-backed mapping above the main exe and reading its build-id via the NT_GNU_BUILD_ID note; a BPF uprobe on the entry of the mapped loader confirms which file actually serviced relocation. Divergence (path, inode, or build-id) is reported.
- **Catches:** ELF interpreter substitution / shim-loader injection (replacing or LD_TRACE-style wrapping ld.so to inject before main()).
- **FP risk:** low — Legitimate sources: explicit invocation via /lib/ld-linux.so as a command, patchelf-modified binaries (common in Nix/Flatpak where PT_INTERP is rewritten to a store path), and Steam's pressure-vessel/bubblewrap runtime which legitimately swaps the interpreter to the runtime's ld.so. Gate by resolving the expected interpreter through the active container manifest (Flatpak metadata / Steam runtime soldier-or-sniper build-id list) before scoring.
- **Horkos slot:** `kernel/linux/bpf/src/interp_entry.bpf.c (uprobe on loader entry) + PT_INTERP/build-id check in kernel/linux/userspace/InterpCheck.cpp`
- **Refs:** PT_INTERP (elf(5)); NT_GNU_BUILD_ID; /proc/pid/maps; patchelf; Steam pressure-vessel

### 85. ld.so.preload / preload-env observed at exec but absent from the file on disk

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** At the moment of execve, the bprm credentials carry an LD_PRELOAD/LD_LIBRARY_PATH/LD_AUDIT environment string OR /etc/ld.so.preload is non-empty, yet a later read of /etc/ld.so.preload by the daemon returns empty — exposing a transient or memory-only preload (env injected only into the game's launch, or a preload file written-then-deleted post-exec).
- **Mechanism:** BPF LSM lsm/bprm_committing_creds (or lsm/bprm_creds_for_exec) reads the linux_binprm->envp to detect LD_PRELOAD/LD_AUDIT/LD_LIBRARY_PATH presence at exec time and stamps the event; the userspace daemon independently inotify-watches /etc/ld.so.preload (IN_MODIFY|IN_CLOSE_WRITE|IN_DELETE). The signal is the disagreement between the at-exec env/file state and the steady-state file content.
- **Catches:** Transient LD_PRELOAD/LD_AUDIT injection scoped to the game launch and per-process env injection that leaves no persistent on-disk artifact.
- **FP risk:** medium — Launchers legitimately inject LD_PRELOAD per-process: Steam sets gameoverlayrenderer/LD_PRELOAD, MangoHud/vkBasalt wrappers, Lutris/Heroic, GameMode, and accessibility tools. Gate by attributing the env-setting ancestor PID: only score when the LD_PRELOAD value's target DSO build-id is unknown to the signed overlay allowlist, and record the launcher chain so the server can distinguish Steam-set from foreign-set.
- **Horkos slot:** `kernel/linux/bpf/src/bprm_env.bpf.c (lsm/bprm_creds_for_exec) + inotify watcher in kernel/linux/userspace/PreloadWatch.cpp`
- **Refs:** lsm/bprm_creds_for_exec, linux_binprm->envp; LD_AUDIT (rtld-audit(7)); inotify(7)

### 86. dlopen of a DSO whose backing inode is deleted, memfd, or anonymous

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** A dlopen/dlmopen call in the protected process succeeds with a path that resolves to a deleted file ('(deleted)' in maps), a /memfd:/ backing, or /proc/self/fd/N — meaning the loaded library has no stable on-disk identity for later verification, the hallmark of fileless library injection.
- **Mechanism:** uprobe on dlopen/dlmopen (and the internal _dl_open) in the game's libc; the BPF program reads the path argument via bpf_probe_read_user_str. The userspace side, on each such event, stats the resolved fd/path and inspects the corresponding /proc/<pid>/maps line for '(deleted)', 'memfd:', or anonymous-with-exec backing. memfd/deleted-backed executable maps are reported.
- **Catches:** Fileless / memfd_create-based library injection where the cheat DSO never lands on a durable filesystem path (defeats on-disk hash checks).
- **FP risk:** low — Legitimate memfd-backed code exists: some JIT/AOT caches, glibc's own dlopen of compiled locale/gconv modules (file-backed, not memfd), and a few DRM/license schemes self-load from memfd. Gate by requiring the memfd/deleted mapping to be executable AND to export symbols the game later calls; whitelist known JIT runtimes (e.g. the game's own scripting VM) by ancestor and build-id.
- **Horkos slot:** `kernel/linux/bpf/src/dlopen_uprobe.bpf.c + fd/maps backing resolver in kernel/linux/userspace/DlopenBacking.cpp`
- **Refs:** dlopen(3)/dlmopen(3); glibc _dl_open; memfd_create(2); /proc/pid/maps '(deleted)' suffix; bpf_probe_read_user_str

### 87. DSO load-order inversion: libc-interposing object mapped before libc

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** The runtime DSO map ordering places a non-allowlisted object earlier in the loader's search/scope list than libc/the engine library it exports symbols clashing with — symbol-interposition via load-order, where an injected DSO is positioned to win ELF symbol resolution over the real implementation.
- **Mechanism:** uretprobe on _dl_map_object records the loader's append order (link_map insertion sequence) per DSO; the userspace loader builds the per-process link_map chain order and, for each exported symbol that also exists in libc/engine, checks whether a non-allowlisted DSO precedes the canonical provider. Inversions affecting hookable symbols (malloc, recv, glXSwapBuffers, vkQueuePresentKHR, time, rand) are scored.
- **Catches:** Symbol-interposition injection that relies on ELF first-wins resolution (LD_PRELOAD ordering abuse, or dlopen with RTLD_DEEPBIND/global-scope manipulation to shadow engine symbols).
- **FP risk:** high — This is the highest-FP signal: legitimate interposers (jemalloc/tcmalloc preloaded before libc, MangoHud/vkBasalt interposing present functions, ASan, fakeroot, libfaketime, accessibility libs) all intentionally win symbol resolution by load-order. Gate hard: score only when the interposing DSO is BOTH outside the signed overlay/allocator allowlist AND lacks DT_NEEDED provenance AND interposes a security-relevant symbol; treat as a corroborating signal, never standalone.
- **Horkos slot:** `kernel/linux/bpf/src/dl_map_object.bpf.c (shared with provenance) + ordering/scope analysis in kernel/linux/userspace/LinkMapOrder.cpp`
- **Refs:** glibc link_map / _dl_map_object insertion order; RTLD_DEEPBIND, symbol interposition (ld.so(8)); DT_SYMBOLIC

### 88. .dynamic / DT_DEBUG r_brk hook divergence

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** The r_brk function pointer in the game's _r_debug structure (the loader's debugger-notification breakpoint reached via DT_DEBUG) points outside ld.so's own VMA, or the DT_RELRO-protected portion of .dynamic is writable, revealing an injector that hooked the loader's link-map update path to hide or rewrite loaded-module bookkeeping.
- **Mechanism:** The userspace loader locates _r_debug via the executable's PT_DYNAMIC DT_DEBUG entry (read through /proc/<pid>/mem at the load-biased address), then a BPF uprobe-driven sample reads r_brk and r_map; r_brk is range-checked against ld.so's mapped extent and the page protection of the relro span is read from /proc/<pid>/maps (expected r--p after relro). A writable relro page or foreign r_brk target is reported.
- **Catches:** Loader-bookkeeping tampering: injectors that hook r_brk to suppress link_map entries (hiding their own DSO from /proc maps walkers) or that re-mark .dynamic writable to rewrite relocation data.
- **FP risk:** low — Debuggers (gdb/lldb) legitimately set r_brk and toggle relro pages; some hardened loaders relocate r_debug. Gate by suppressing when a ptrace tracer is attached (already observed via the sys_enter_ptrace signal) and by allowlisting the distro ld.so build-id's expected r_brk offset; only score foreign-VMA r_brk with no attached debugger.
- **Horkos slot:** `kernel/linux/bpf/src/rdebug_sample.bpf.c + _r_debug locator in kernel/linux/userspace/RDebugCheck.cpp`
- **Refs:** _r_debug / r_brk (link.h); DT_DEBUG, DT_RELRO; /proc/pid/mem, /proc/pid/maps page perms; ld.so(8)

### 89. LD_AUDIT rtld-audit namespace activation

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** The game process has an active rtld-audit module — observable as a la_* callback DSO in its link scope and a populated _dl_audit list — even though no LD_AUDIT value persists in its current environment, indicating an audit-interface injector (a documented but rarely-legitimate loader instrumentation channel that can rewrite every symbol binding).
- **Mechanism:** At exec, lsm/bprm_creds_for_exec captures LD_AUDIT from envp (companion to the preload-env signal); at steady state a uprobe on the loader's _dl_audit_symbind / la_symbind entry points detects that audit callbacks are firing for the protected process. The presence of firing la_symbind without a recorded, allowlisted LD_AUDIT at exec is the anomaly.
- **Catches:** LD_AUDIT-based injection — the rtld-audit interface lets a module intercept and rewrite every PLT binding, a stealthier and more powerful channel than LD_PRELOAD.
- **FP risk:** low — LD_AUDIT is genuinely rare in production game launches; legitimate users are profilers (latrace), some sanitizer setups, and a few enterprise tracing agents. Gate by allowlisting the audit module's build-id and by attributing the LD_AUDIT-setting ancestor; absence of any legitimate launcher reason makes this high-confidence. Near-zero FP for the consumer Steam Deck path.
- **Horkos slot:** `kernel/linux/bpf/src/dl_audit.bpf.c (uprobe on _dl_audit_symbind) + env capture shared with bprm_env.bpf.c`
- **Refs:** rtld-audit(7); LD_AUDIT; glibc _dl_audit_symbind / la_symbind; lsm/bprm_creds_for_exec envp

### 90. Text-segment page no longer file-backed / MAP_PRIVATE COW-broken in a loaded DSO

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** An r-xp executable page belonging to a legitimately-loaded library or the game itself reports a non-zero pss_dirty / private-dirty count or its pagemap entry shows the page is no longer file-backed (anonymous after COW), proving the read-only code of an already-mapped DSO was patched in memory after load — inline hooking of an otherwise-legitimate library.
- **Mechanism:** The userspace daemon reads /proc/<pid>/smaps_rollup and per-VMA /proc/<pid>/smaps for executable file-backed mappings, then uses /proc/<pid>/pagemap to test the file-backed bit per page; any executable page that has gone private-dirty/anonymous is sampled, and the divergence offset is reported. A BPF uprobe on a hot game function provides the trigger cadence so sampling is event-driven, not pure polling.
- **Catches:** In-memory inline patching / trampoline hooking of a properly-loaded library (the case GOT-integrity misses because the import table is untouched and the .text itself is overwritten).
- **FP risk:** medium — Legitimate dirty executable pages occur with: glibc IFUNC/relro resolution writing into the first text page, ld.so applying TEXTREL relocations (rare, R_*_RELATIVE in .text), JIT-patched code, and dynamic instrumentation by allowed overlays. Gate by ignoring the known-relocated prologue range, suppressing when a tracer is attached, and only scoring dirty exec pages outside documented relocation/IFUNC spans of the DSO's build-id.
- **Horkos slot:** `kernel/linux/userspace/TextPageBacking.cpp (smaps/pagemap reader) + trigger uprobe in kernel/linux/bpf/src/text_sample.bpf.c`
- **Refs:** /proc/pid/smaps (Private_Dirty), /proc/pid/pagemap (page-file-backed/soft-dirty bits), /proc/pid/smaps_rollup; TEXTREL/IFUNC; proc(5)

## Linux — Kernel / Module Trust  `linux-module-integrity`

### 91. kallsyms text-address drift vs .text section bounds

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A symbol that /proc/kallsyms resolves to an address outside the running kernel's _stext.._etext (or the owning module's core_layout.base..base+text_size) range, or two distinct symbols colliding on one address, indicates a relocated/trampolined syscall handler rather than the in-tree symbol.
- **Mechanism:** Userspace reads /proc/kallsyms (requires CAP_SYSLOG; addresses non-zero) and cross-checks against section bounds exposed via /sys/kernel/notes and /proc/iomem 'Kernel code' resource; per-module text bounds via /sys/module/<m>/sections/.text. Client snapshots and ships deltas; server compares to a per-kernel-build baseline (from the signed BTF/System.map fingerprint).
- **Catches:** Syscall-table or function-pointer relocation, runtime kernel patching that moves a handler's effective entry point, symbol-shadowing rootkits that register a same-named symbol.
- **FP risk:** medium — Live kernel patching (kpatch, kGraft, Ubuntu Livepatch, Oracle Ksplice) legitimately relocates patched functions into a new module's text; gate by allowlisting the livepatch module name prefix (klp_*, kpatch_*) and confirming the relocation target lands inside a /sys/module entry whose .ko is in the loaded-module set with a valid signature.
- **Horkos slot:** `kernel/linux/userspace/KallsymsAudit.cpp (new), reported via Loader.cpp ringbuf bridge`
- **Refs:** /proc/kallsyms, kallsyms_lookup_name semantics, /sys/module/*/sections/.text, /proc/iomem 'Kernel code', /sys/kernel/notes (.note.gnu.build-id)

### 92. module present in kobject tree but absent from /proc/modules (or vice versa)

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A module that appears in one canonical enumeration source but not another — e.g. a /sys/module/<name> kobject directory with no matching line in /proc/modules, or a struct module reachable by walking module_kset that THIS_MODULE list-unlinking hid from both — reveals a module that delinked itself from the global modules list to hide.
- **Mechanism:** Userspace diffs three independent views: /proc/modules (modules list), /sys/module/* (module_kset kobjects), and an eBPF iterator (BPF_TASK_ITER-style bpf_iter over the modules kset / kallsyms) walking the kobject tree directly; discrepancy is the signal. eBPF reads via bpf_for_each_kernel_symbol / module iterator and CO-RE struct module field reads.
- **Catches:** list_del-based module hiding (classic LKM rootkit self-unlink from THIS_MODULE->list), modules that scrub their /proc/modules entry but leave sysfs kobjects.
- **FP risk:** low — Brief races during insmod/rmmod can show transient single-source presence; gate with a re-sample debounce (two consecutive snapshots >500ms apart must agree) and ignore modules in MODULE_STATE_COMING/GOING read from struct module->state.
- **Horkos slot:** `kernel/linux/bpf/src/module_iter.bpf.c (new) + diff logic in kernel/linux/userspace/Loader.cpp`
- **Refs:** bpf_iter, /proc/modules, /sys/module/, struct module->list/->state, module_kset

### 93. ftrace handler ownership audit on syscall entry functions

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** An entry in /sys/kernel/tracing/enabled_functions showing a callback attached to a security-relevant function (sys_*, __x64_sys_*, do_exit, commit_creds, the BPF/kprobe machinery) whose owning ftrace_ops does not belong to a known tracer (the client's own probes, perf, livepatch) signals an ftrace-based function hook.
- **Mechanism:** Userspace reads /sys/kernel/tracing/enabled_functions and /sys/kernel/tracing/touched_functions (5.18+), which list every function with an active ftrace callback plus the owning op's address; cross-reference op addresses against the loaded-module text map from signal #1. CAP_SYS_ADMIN / tracefs mount required.
- **Catches:** ftrace_ops-based syscall hooking (the dominant modern Linux rootkit technique, e.g. khook/ftrace hooks), unauthorized function-graph interception of credential or exec paths.
- **FP risk:** medium — Legitimate tracers attach here constantly: bpftrace/perf/SystemTap, eBPF fentry programs, kprobes from observability agents (Falco, Tetragon, Datadog), and livepatch. Gate by attributing each ftrace_ops owner to a signed module or a registered tracer PID, and only flag callbacks on the credential/exec/ptrace-shadow function set rather than all instrumented functions.
- **Horkos slot:** `kernel/linux/userspace/FtraceAudit.cpp (new)`

### 94. kprobe/kretprobe blacklist-bypass on credential and exec symbols

- **Platform / layer:** linux-ebpf / kernel

### 97. foreign eBPF program enumeration on the game's own hook points

### 98. /dev/mem, /dev/kmem and /dev/port open/mmap on the live kernel

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** Any process opening or mmap-ing /dev/mem, /dev/kmem, or /dev/port (or /proc/kcore for write-capable mappings) on a kernel where CONFIG_STRICT_DEVMEM would normally forbid it signals direct physical/kernel memory access used to patch kernel structures without loading a module.

- **Catches:** Module-less kernel memory patching via /dev/mem, DKOM performed by writing physical memory directly, kcore-based kernel introspection by cheats that avoid the module loader entirely.
- **FP risk:** medium — Legitimate users of /dev/mem include X11 (older), dmidecode, flashrom, hwinfo, mcelog, and some GPU/firmware tools; gate by flagging WRITE-intent opens (O_RDWR/O_WRONLY) and mmaps with PROT_WRITE rather than read-only DMI scans, and allowlist known tool comm names with short-lived read-only access.

### 99. MSR write access to control/debug/syscall-entry registers

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A write to /dev/cpu/*/msr targeting MSR_LSTAR (0xC0000082, the syscall entry RIP), MSR_IA32_SYSENTER_EIP, or the debug/feature-control MSRs is a strong indicator of syscall-entry redirection or hardware-breakpoint abuse performed from userspace without a module.
- **Mechanism:** An eBPF tracepoint on sys_enter_pwrite64/sys_enter_write filtered to fds whose path resolves under /dev/cpu/N/msr, decoding the file offset (= MSR index) to flag writes to the sensitive MSR set; complemented by an lsm/file_open record on the msr device node. The msr driver must be loaded for the device to exist, which is itself recorded.
- **Catches:** Syscall-entry hijacking via LSTAR rewrite, hardware-breakpoint-based hooking through debug-register MSRs, SMEP/SMAP-disabling CR-shadow tricks surfaced as feature-control MSR writes.
- **FP risk:** low — Legitimate MSR writers are narrow: undervolting/overclocking tools (intel-undervolt, ryzenadj, zenstates, throttled/lenovo), turbostat, and msr-tools writing power/turbo MSRs (e.g. 0x150, 0x199, 0x1A0); gate by MSR index — power/perf MSRs are benign, LSTAR/SYSENTER_EIP/debug MSRs are not — and exclude read-only rdmsr access entirely.

### 100. WINEDLLOVERRIDES native-DLL injection delta

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** A clean Proton launch resolves a known set of builtin/native DLL override tokens; an injected cheat loader prepends a 'native' override for a DLL that ships builtin-only (e.g. ntdll, kernel32, d3d11, dxgi) or adds an override naming a DLL absent from the Proton dist manifest.

- **Catches:** DLL-override-based injection: forcing a 'native' attacker-supplied PE/SO to shadow a Wine builtin so cheat code loads inside the game's address space at startup.
- **FP risk:** medium — DXVK/VKD3D-Proton legitimately set native overrides for d3d11/d3d9/dxgi/d3d12; Proton itself, dgVoodoo, reshade, and per-game protonfixes set overrides. ### 101. Foreign SO mapped into prefix from outside the dist/compat tree

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** Every file-backed executable mapping in the game process should resolve to a path under the Proton install (steamapps/common/Proton*/), the Steam Linux Runtime sniper sysroot, the game's own install dir, or the prefix's drive_c; an executable mapping backed by an inode under /tmp, /dev/shm, $HOME (non-Steam), or a deleted/anon file is foreign.

- **Signal:** Cross-process memory access into the game (process_vm_readv/writev, or /proc/<pid>/mem open+pread) legitimately originates only from the game's own wineserver (same prefix, parent of the wine process tree) and from the Horkos client itself; any other peer reading/writing the game's address space is an external cheat scanner.

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** On Steam Deck Game Mode (and locked-down SteamOS) the kernel module set is fixed at boot and the rootfs is read-only; any finit_module/init_module after the boot window, or load of a module not in the SteamOS signed-module baseline, indicates BYOVD-style cheat-driver loading where an LKM was assumed impossible.

### 107. Wine PE-section integrity: builtin DLL .text backed by non-dist inode

- **Platform / layer:** linux-ebpf / userspace
- **Signal:** Wine builtin DLLs loaded into the game are file-backed by .so/.dll files under the Proton dist tree; if the executable VMA for a core builtin (ntdll, kernelbase, win32u) is backed by an inode whose path/inode does not match the dist file, or its .text page protections were flipped W^X (RW then RX) post-load, the builtin has been swapped or hot-patched.
- **Mechanism:** Userspace verifier walks /proc/<pid>/maps for the game, identifies VMAs whose pathname matches Wine builtin SO names, and stat()s the backing inode comparing dev:ino and SHA256 against the Proton dist manifest; complemented by a BPF LSM file_mprotect hook flagging any PROT_EXEC re-protection of a VMA inside a builtin's address range (W^X violation) during gameplay.
- **Catches:** In-memory builtin patching / DLL substitution: a cheat that replaces or hot-patches Wine's ntdll/win32u to hook syscall thunks (e.g. NtUserGetAsyncKeyState for synthetic input, NtReadVirtualMemory for scanning) after load.
- **FP risk:** medium — JIT-ish paths and some legitimate hotpatch/relocation passes flip protections; Proton's own builtin relocation and DXVK shader stores touch protections. Gate the mprotect arm to VMAs that resolve to a builtin SO path AND were previously RX (re-arming exec on a code page), and gate the inode arm to the per-Proton-version manifest. ESYNC/FSYNC and Wine's PE loader relocations are accounted for by hashing the on-disk dist file rather than the in-memory image where relocations apply.
- **Horkos slot:** `kernel/linux/bpf/src/mprotect_wx_audit.bpf.c + kernel/linux/userspace/WineBuiltinIntegrity.cpp`
- **Refs:** lsm/file_mprotect, /proc/pid/maps, vm_area_struct->vm_file dev:ino, Proton dist manifest, Wine builtin loader (libwine PE)

### 108. Synthetic input via injected uinput/evdev device into the Deck input stack

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** On the Deck, input events reaching the game originate from the embedded controller (Steam Input / hid-steam) and the known evdev nodes; a uinput virtual device created during gameplay, or evdev EV_KEY/EV_ABS events injected from a process that is not Steam Input / gamescope's input path, indicates a synthetic-input (recoil/aim) macro feeding the game.
- **Mechanism:** BPF tracepoints/kprobes on uinput device creation (kprobe uinput_create_device / open of /dev/uinput) and on input_event / input_inject_event in the input core (kprobe input_inject_event), attributing the injecting tgid; baseline is the Steam Input daemon tgid and gamescope's libinput consumer resolved at session start.
- **Catches:** Synthetic-input cheats (no-recoil, triggerbot, aim macros) that fabricate controller/keyboard events via a uinput virtual device rather than touching game memory — the input-side analogue the memory probes miss.
- **FP risk:** high — Steam Input itself synthesizes a virtual gamepad via uinput; AntiMicroX, sc-controller, qjoypad, on-screen keyboards, and accessibility remappers all legitimately use uinput. Gate by allowlisting the Steam Input / hid-steam path and any uinput device created before the game window gains focus; only a uinput device created mid-session by a non-allowlisted tgid that then drives EV events correlated with game-relevant keys is reportable, and even then as a weighted signal needing server-side correlation.

## macOS — Process Inspection / Injection  `macos-injection`

### 109. Foreign task port acquisition on the game task

- **Platform / layer:** macos-es / kernel
- **Signal:** A clean game process is the target of zero ES_EVENT_TYPE_GET_TASK events; any non-Apple, non-debugger process resolving a task control port for the game indicates someone is about to read/write its address space.
- **Mechanism:** ES_EVENT_TYPE_NOTIFY_GET_TASK — es_message_t.event.get_task.target is the game; record source audit_token (es_message_t.process), its signing-id, team-id, and platform-binary flag; correlate against game pid.
- **Catches:** Foreign task_for_pid()/processor_set_tasks acquisition of the game's task control port, the prerequisite for mach_vm_read/write injection.
- **FP risk:** medium — Crash reporters (ReportCrash, spindump, sample/leaks), Xcode/lldb debug-stub, and Rosetta runtime legitimately call GET_TASK. Gate on: source is non-platform-binary AND not in the signed allowlist (Apple team-ids for diagnostics) AND game was not launched under a debugger session. Server makes the trust call.
- **Horkos slot:** `kernel/macos/es/EsClient.mm (subscribe ES_EVENT_TYPE_NOTIFY_GET_TASK; new hk_event payload hk_es_get_task in horkos/event_schema.h)`
- **Refs:** ES_EVENT_TYPE_NOTIFY_GET_TASK, es_event_get_task_t, es_process_t.is_platform_binary, audit_token_to_au32

### 110. GET_TASK_NAME vs GET_TASK control-port discrimination

- **Platform / layer:** macos-es / kernel
- **Signal:** Benign inspectors (Activity Monitor, name-port introspection) request the task NAME port which cannot write memory; an injector needs the task CONTROL port — observing which flavor was requested separates harmless introspection from write-capable access.
- **Mechanism:** Distinguish ES_EVENT_TYPE_NOTIFY_GET_TASK (control port, write-capable) from ES_EVENT_TYPE_NOTIFY_GET_TASK_NAME and ES_EVENT_TYPE_NOTIFY_GET_TASK_READ; only the control-port flavor against the game is high-signal. Record event subtype + es_get_task_type_t.
- **Catches:** Acquisition of a write-capable task control port (as opposed to read-only/name introspection), narrowing false positives from generic task-port telemetry.
- **FP risk:** low — GET_TASK_NAME is emitted constantly by Activity Monitor/top and is benign — it must NOT be flagged. Only the control-port (and to a lesser degree READ) flavor against the game warrants reporting. The three distinct ES event types make discrimination exact.
- **Horkos slot:** `kernel/macos/es/EsClient.mm (handle the three GET_TASK* event types in the same handler; tag subtype in hk_es_get_task)`
- **Refs:** ES_EVENT_TYPE_NOTIFY_GET_TASK_NAME, ES_EVENT_TYPE_NOTIFY_GET_TASK_READ, es_get_task_type_t (ESTypes.h, macOS 11.3+)

### 111. Non-self executable MMAP into the game address space

- **Platform / layer:** macos-es / kernel
- **Signal:** A clean game maps only its own bundle's dylibs and the dyld shared cache; an MMAP carrying PROT_EXEC backed by a file the game never links, or an anonymous PROT_EXEC|PROT_WRITE region, marks freshly injected code.
- **Mechanism:** ES_EVENT_TYPE_NOTIFY_MMAP — inspect es_event_mmap_t.protection for PROT_EXEC, es_event_mmap_t.flags for MAP_ANON, and es_event_mmap_t.source (es_file_t) path; flag exec mappings of files outside the game bundle / not in the signed-dylib manifest.
- **Catches:** Code injection via mmap of a foreign mach-o or RWX anonymous page into the game (post task-port write, the loader maps the payload).
- **FP risk:** high — JITs (JavaScriptCore in embedded WebViews, Mono/Unity IL2CPP fallback, Wine/CrossOver, game scripting VMs) legitimately create RWX/exec anon maps. Gate by building a per-title baseline of expected exec-map sources and the game's own JIT entitlement (com.apple.security.cs.allow-jit); report only exec maps from unexpected file sources or unexpected processes mapping into the game.
- **Horkos slot:** `kernel/macos/es/EsClient.mm (subscribe ES_EVENT_TYPE_NOTIFY_MMAP; new hk_es_mmap payload) + baseline manifest in daemon/macos/horkosd.cpp`
- **Refs:** ES_EVENT_TYPE_NOTIFY_MMAP, es_event_mmap_t (protection/flags/source), PROT_EXEC, MAP_ANON

### 112. DYLD_INSERT_LIBRARIES survival past the hardened-runtime strip

- **Platform / layer:** macos-es / kernel
- **Signal:** The game ships hardened-runtime, so dyld strips DYLD_* env vars at exec; observing the variable present in the exec environment AND a corresponding non-system dylib actually loaded means the protection was defeated (e.g. via a re-signed/entitlement-stripped binary).
- **Mechanism:** At ES_EVENT_TYPE_NOTIFY_EXEC, walk es_exec_env_count()/es_exec_env() for DYLD_INSERT_LIBRARIES and DYLD_FRAMEWORK_PATH; cross-check against the executable's CS flags (es_process_t.codesigning_flags CS_RESTRICT/CS_RUNTIME) — presence of the var on a runtime-hardened binary is anomalous, and an actual injected load confirms it.
- **Catches:** Library injection via DYLD_INSERT_LIBRARIES on a binary whose hardened runtime should have neutralized it — implies the game binary was re-signed or the dyld protection bypassed.
- **FP risk:** medium — Legit profilers, asan/tsan runtimes, and some launchers set DYLD_INSERT_LIBRARIES; on a non-hardened dev build this is normal. Gate on production builds where codesigning_flags shows CS_RUNTIME — for those, the var should have been stripped, so its presence is the anomaly. Allowlist known instrumentation team-ids.
- **Horkos slot:** `kernel/macos/es/EsClient.mm (extend the EXEC handler env-var scan; emit hk_es_dyld_inject) `
- **Refs:** es_exec_env(), es_exec_env_count(), es_event_exec_t, es_process_t.codesigning_flags, CS_RUNTIME/CS_RESTRICT

### 113. Foreign mach exception port hijack on the game task

- **Platform / layer:** macos-user / kernel
- **Signal:** A clean game owns its own exception ports (or none); a foreign process installing itself as the game's EXC_MASK_ALL / EXC_BAD_ACCESS handler is a hardware-breakpoint debugger or a fault-driven memory hook, not normal operation.
- **Mechanism:** From the daemon, periodically call task_get_exception_ports(game_task, EXC_MASK_ALL, ...) (game task obtained via the daemon's own privileged audit-token-validated handle) and compare the returned exception port set against a baseline captured at game launch; a new non-self port owner indicates a hijack. Pair with ES_EVENT_TYPE_NOTIFY_GET_TASK source attribution.
- **Catches:** Mach exception-port registration by another task on the game (breakpoint/watchpoint-based cheats, EXC_BAD_ACCESS page-fault memory hooks).
- **FP risk:** medium — Crash-reporting frameworks (Sentry, Crashlytics, PLCrashReporter) and the system ReportCrash legitimately register exception ports — but in-process, by the game itself. The anomaly is a port owned by a FOREIGN task. Gate by resolving the receiving task and excluding the game's own in-process handlers and Apple's diagnostics.
- **Horkos slot:** `daemon/macos/horkosd.cpp (periodic task_get_exception_ports poll; new HKExceptionPortBaseline module) + correlate in kernel/macos/es/EsClient.mm`
- **Refs:** task_get_exception_ports(), exception_mask_t, EXC_MASK_ALL, mach/task.h

### 114. Foreign-created thread with non-bundle entry point

- **Platform / layer:** macos-user / kernel
- **Signal:** Every thread in a clean game has a start address inside the game's own __TEXT or a linked dylib; a thread whose PC/entry lies in an anonymous or foreign-mapped region (classic thread_create_running injection) is execution of injected code.
- **Mechanism:** From the daemon enumerate task_threads(game_task,...) then thread_get_state(THREAD_STATE) reading the arm64 __pc / x86_64 __rip start, and resolve each thread's start address via the dyld image list (dladdr-equivalent over /usr/lib/dyld image headers); flag any thread whose entry is not covered by a known mach-o image region.
- **Catches:** Remote thread injection via thread_create_running()/thread_set_state from another task, where the new thread executes from an unmapped/foreign region.
- **FP risk:** medium — JIT engines and some GPU/driver worker threads have start addresses in anonymous JIT regions. Gate by intersecting with the MMAP-exec baseline (signal #3): a foreign-entry thread is only high-signal when its region is NOT a sanctioned JIT map. Sample at low frequency to bound cost.
- **Horkos slot:** `daemon/macos/horkosd.cpp (task_threads enumeration + image-region resolver; new HKThreadIntegrity module)`
- **Refs:** task_threads(), thread_get_state(), ARM_THREAD_STATE64 / x86_THREAD_STATE64, _dyld_get_image_header

### 115. PROC_CHECK / pidfd-style probing of the game by foreign tools

- **Platform / layer:** macos-es / kernel
- **Signal:** A clean game is not the subject of repeated foreign proc_pidinfo/region-walk probing; a process enumerating the game's memory regions or task info via proc_info is profiling it for an attach, which ES surfaces as PROC_CHECK.
- **Mechanism:** ES_EVENT_TYPE_NOTIFY_PROC_CHECK — es_event_proc_check_t.target is the game, .type is the proc_info call class (e.g. PROC_PIDREGIONPATHINFO, PROC_PIDTASKINFO, PROC_PIDVMRMAPINFO); record source + flavor and rate over a window. High-rate region walks of the game by a foreign, non-platform binary is the signal.
- **Catches:** Reconnaissance preceding injection — a foreign tool walking the game's VM map / task info (proc_pidinfo) to locate writable code regions.
- **FP risk:** medium — Activity Monitor, htop, lldb, and security tools issue proc_info constantly and legitimately. Gate on: foreign non-platform-binary source AND high-cardinality VM-region flavors (region/vmrmap) AND no debugger session — not single benign PIDTASKINFO polls. Report rate + flavor mix, let the server score.
- **Horkos slot:** `kernel/macos/es/EsClient.mm (subscribe ES_EVENT_TYPE_NOTIFY_PROC_CHECK; aggregate per-source rate into hk_es_proc_check)`
- **Refs:** ES_EVENT_TYPE_NOTIFY_PROC_CHECK, es_event_proc_check_t, es_proc_check_type_t, proc_info(2) flavors

### 116. Debugger attach via P_TRACED / get-task-allow drift

- **Platform / layer:** macos-es / kernel
- **Signal:** A production game runs un-traced with no get-task-allow entitlement; the game becoming traced (P_TRACED set) or a ptrace(PT_ATTACHEXC) against it at runtime indicates a live debugger, the most direct memory-tamper vector.
- **Mechanism:** Two correlated sources: (a) the daemon polls the game's kinfo_proc via sysctl(CTL_KERN,KERN_PROC,KERN_PROC_PID) and checks p_flag & P_TRACED; (b) on Linux-parity macOS there is no ptrace ES event, so detect the attach side-channel via the resulting ES_EVENT_TYPE_NOTIFY_GET_TASK from the debugger plus a P_TRACED transition. Report the transition edge with the attaching pid.
- **Catches:** Live debugger attach (lldb/gdb/custom ptrace tool) to the game process for runtime memory inspection/patching.
- **FP risk:** low — Only an issue for dev builds intentionally run under a debugger; production titles are never legitimately traced. Gate by build channel: a P_TRACED edge on a release-signed binary with no get-task-allow entitlement is unambiguous. Self-attach by the game's own crash handler does not set P_TRACED.
- **Horkos slot:** `daemon/macos/horkosd.cpp (sysctl P_TRACED poller; correlate with GET_TASK source in EsClient.mm)`
- **Refs:** sysctl KERN_PROC_PID, struct kinfo_proc.kp_proc.p_flag, P_TRACED, ptrace(2) PT_ATTACHEXC

### 117. Foreign mach_vm_protect W^X violation on the game's __TEXT

- **Platform / layer:** macos-user / kernel
- **Signal:** A clean code-signed game keeps __TEXT pages r-x and never writable; a page in the game's signed __TEXT range becoming writable (or losing its codesign-validated bit) is the artifact left by mach_vm_protect()-then-patch inline hooking.
- **Mechanism:** From the daemon, walk the game task with mach_vm_region_recurse(game_task, ...) over the __TEXT segment range (resolved from the on-disk mach-o load commands) and inspect vm_region_submap_info_64.protection for VM_PROT_WRITE and the .user_tag / share_mode; a writable or COW-broken page inside signed __TEXT signals an inline patch. Cross-check codesign validity via csops(CS_OPS_STATUS).
- **Catches:** Inline/trampoline code patching: mach_vm_protect to RW, byte patch, restore — detectable by the residual writable mapping or the broken code-signature page hash.
- **FP risk:** low — Legit in-process JIT/trampoline libs (Fishhook does symbol rebinding in __DATA, not __TEXT) rarely flip __TEXT writable on a hardened binary. The narrow scope (writable page strictly inside the signed __TEXT range of a CS_RUNTIME binary) keeps FPs low. Gate out the game's own sanctioned hot-patch ranges if any are declared.
- **Horkos slot:** `daemon/macos/horkosd.cpp (mach_vm_region_recurse __TEXT scanner + csops status check; new HKTextIntegrity module)`
- **Refs:** mach_vm_region_recurse(), vm_region_submap_info_64, VM_PROT_WRITE, csops()/CS_OPS_STATUS, mach-o segment load commands

## macOS — Code-Signing & Platform Trust  `macos-codesign-integrity`

### 118. CS_HARD/CS_KILL flag drift on the running image

- **Platform / layer:** macos-user / kernel
- **Signal:** A notarized, hardened-runtime game ships with CS_HARD|CS_KILL|CS_RUNTIME set, so the kernel will SIGKILL it on any page-validation failure; a process whose csflags lack CS_KILL/CS_HARD was launched with code-signing enforcement deliberately weakened.
- **Mechanism:** csops(pid, CS_OPS_STATUS, &flags, sizeof(flags)) reading the csflags bitmask (CS_VALID, CS_HARD, CS_KILL, CS_RUNTIME, CS_RESTRICT, CS_ENFORCEMENT, CS_REQUIRE_LST) declared in <sys/codesign.h>; sampled on self and on sibling game PIDs.
- **Catches:** Launch-time code-signing enforcement strip (e.g. running under a re-signed/ad-hoc binary, AMFI_get_out_of_my_way boot-arg, or a launcher that clears CS_KILL so page hashes can be mutated without the kernel killing the process).
- **FP risk:** medium — Developer/debug builds and Rosetta-translated binaries carry different flag sets; some VM and translation layers clear CS_RUNTIME. Gate by comparing the observed flags against the flags Horkos baked into the shipped binary's own signature (known-good constant), not an absolute expected mask.
- **Horkos slot:** `daemon/macos/csops/CsFlagsProbe.cpp (new), behind a CsIntegrityProbe interface declared in daemon/macos/csops/CsIntegrityProbe.h`
- **Refs:** csops(2) / csops_audittoken(2); CS_OPS_STATUS; <sys/codesign.h> CS_* flags; <System/codesign.h>

### 119. cdhash mismatch between live image and notarization ticket

- **Platform / layer:** macos-user / kernel
- **Signal:** The kernel-reported code-directory hash of the running main executable must equal the cdhash embedded in the binary on disk and the one stapled in the notarization ticket; a divergence means the in-memory image was re-signed or the on-disk binary was swapped after launch.
- **Mechanism:** csops(pid, CS_OPS_PIDCDHASH, buf, sizeof(buf)) (and CS_OPS_TEISDISABLED for guard) to read the kernel's live cdhash, cross-checked against SecCodeCopySigningInformation(kSecCSDefaultFlags|kSecCSSigningInformation) cdhashes from SecCodeCreateWithPath on the on-disk bundle.
- **Catches:** Post-launch binary substitution / re-signing (ad-hoc resign of a patched executable, fat-binary slice swap) where the disk path still looks valid but the kernel loaded a different code directory.
- **FP risk:** low — App-translocation (Gatekeeper quarantine randomized path) and Rosetta AOT caches can shift the on-disk path; resolve the real path via the translocation API and compare against the architecture slice actually executing.
- **Horkos slot:** `daemon/macos/csops/CdHashProbe.cpp (new) feeding the CsIntegrityProbe interface`
- **Refs:** csops CS_OPS_PIDCDHASH; SecCodeCopySigningInformation; kSecCodeInfoUnique / kSecCodeInfoCdHashes; SecTranslocateCreateOriginalPathForURL

### 120. Dynamic SecCode validity (SecCodeCheckValidity) vs static-only checks

- **Platform / layer:** macos-user / userspace
- **Signal:** A clean process passes SecCodeCheckValidity against its own designated requirement at runtime; failure (errSecCSGuestInvalid / errSecCSVmwMapping) indicates pages were modified in memory after load even though the on-disk static signature still verifies.
- **Mechanism:** SecCodeCopyGuestWithAttributes(NULL, {kSecGuestAttributePid: pid}, ...) then SecCodeCheckValidity(code, kSecCSEnforceRevocation, requirement) where the requirement is SecRequirementCreateWithString of Horkos's own DR; run periodically, not just at start.
- **Catches:** In-memory page patching of executable segments (function hooks, inline patches) that leave the disk binary untouched, which a static SecStaticCodeCheckValidity would miss.
- **FP risk:** medium — JIT regions, legitimately-unsigned plugin bundles, and code that the OS lazily pages can transiently fail; restrict the requirement to the main executable's signed __TEXT and treat transient failures as needing N-of-M confirmation before reporting.
- **Horkos slot:** `daemon/macos/seccode/DynamicValidityProbe.cpp (new), Security.framework linked`
- **Refs:** SecCodeCopyGuestWithAttributes; SecCodeCheckValidity; SecRequirementCreateWithString; errSecCSGuestInvalid

### 121. NOTIFY_CS_INVALIDATED correlated to specific mapped image + RtD fault

- **Platform / layer:** macos-es / kernel
- **Signal:** Beyond merely receiving CS_INVALIDATED, a clean process never has its CS_VALID bit cleared; the discriminator is which image faulted and whether it coincides with a NOTIFY_MMAP of an unsigned/foreign page or a NOTIFY_REMOUNT — distinguishing a benign self-update from a runtime tamper of the live text.
- **Mechanism:** ES ES_EVENT_TYPE_NOTIFY_CS_INVALIDATED joined in the daemon with ES_EVENT_TYPE_NOTIFY_MMAP (es_event_mmap_t: source FD signing id, protection PROT_EXEC) and a follow-up csops CS_OPS_STATUS read confirming CS_VALID cleared on that exact audit_token.
- **Catches:** Runtime code-signing invalidation caused by writing to a signed executable page (CoW break of __TEXT, mprotect-to-RWX of signed region) — the kernel clears CS_VALID and emits the event the moment the tamper lands.
- **FP risk:** low — Self-updating apps, Sparkle updaters, and shared-cache repaging can legitimately invalidate; gate on the invalidated image being the game's own signed binary AND a correlated PROT_EXEC mmap from a non-platform FD within a short window.
- **Horkos slot:** `kernel/macos/es/EsClient.mm subscription extension + daemon/macos/csops/CsInvalidationCorrelator.cpp (new)`
- **Refs:** ES_EVENT_TYPE_NOTIFY_CS_INVALIDATED; ES_EVENT_TYPE_NOTIFY_MMAP; es_event_mmap_t; <sys/codesign.h> CS_VALID

### 122. Per-dylib team-id / platform-binary divergence under library validation

- **Platform / layer:** macos-es / kernel
- **Signal:** With the library-validation hardened-runtime flag, every loaded dylib must share the main binary's Team ID or be an Apple platform binary; a loaded image whose team-id differs (or is ad-hoc/unsigned) while LV is nominally enabled means LV was bypassed or the flag was stripped.
- **Mechanism:** ES ES_EVENT_TYPE_NOTIFY_MMAP / dyld image loads, reading es_message_t.process.team_id, es_process_t.is_platform_binary, and signing_id; cross-checked against csops CS_OPS_STATUS CS_REQUIRE_LST bit on the host process and SecCodeCopySigningInformation team-id of each mapped dylib.
- **Catches:** Foreign dylib injection (DYLD_INSERT_LIBRARIES, dlopen of a third-party/ad-hoc library) into a process that is supposed to enforce library validation — i.e. the unsigned-code-loading class.
- **FP risk:** medium — Legitimate plugin hosts, OBS virtual-camera, audio AU plugins, and accessibility tools inject differently-signed dylibs; gate by the host being the game itself (not the launcher) and by LV being declared in the game's own entitlements, with an allowlist of known-good Apple/Steam signing ids.
- **Horkos slot:** `daemon/macos/seccode/DylibTeamIdProbe.cpp (new), correlated in CsIntegrityProbe`
- **Refs:** es_process_t.team_id / is_platform_binary / signing_id; CS_REQUIRE_LST; SecCodeCopySigningInformation kSecCodeInfoTeamIdentifier

### 123. amfid external trust-override via task_for_pid / Mach port on amfid

- **Platform / layer:** macos-es / userspace
- **Signal:** On a clean system nothing holds amfid's task port; a process acquiring amfid's task port (or amfid having an unexpected debug exception port) is the signature of the classic amfid hijack that forces every signature check to return valid.
- **Mechanism:** ES ES_EVENT_TYPE_NOTIFY_GET_TASK / ES_EVENT_TYPE_NOTIFY_GET_TASK_READ where the target audit_token resolves to /usr/libexec/amfid, plus daemon-side task_get_exception_ports parity check on amfid (requires the daemon's own debug entitlement); also watch for unexpected MISVALIDATION log subsystem entries.
- **Catches:** amfid hijacking / trust-cache override (intercepting amfid's MISValidate so a patched/unsigned binary is reported as correctly signed) — the kernel delegates ambiguous validations to amfid in userspace.
- **FP risk:** low — Apple's own debugging tools and lldb can legitimately task_for_pid system daemons only with SIP off; on a SIP-on production machine any get-task on amfid is high-signal. Gate by SIP-state cross-check so SIP-disabled dev boxes are reported separately, not as cheating.

- **Refs:** ES_EVENT_TYPE_NOTIFY_GET_TASK; ES_EVENT_TYPE_NOTIFY_GET_TASK_READ; task_get_exception_ports; com.apple.MobileFileIntegrity / amfid

### 124. AMFI developer-mode / boot-arg / nvram trust posture

- **Platform / layer:** macos-user / hardware
- **Signal:** A retail trusted machine has Developer Mode off, no amfi_get_out_of_my_way/cs_enforcement_disable boot-args, and full-security recovery posture; any of these being set system-wide means AMFI page-hash enforcement was globally weakened before the game ever launched.

- **FP risk:** medium — Legitimate macOS developers enable Developer Mode and set boot-args; Apple Silicon Permissive Security is a supported state. Report posture verbatim to the server as a trust-tier input rather than a ban; let the server policy decide whether reduced-security boots may play.

### 125. Quarantine xattr + provenance: bundle that never passed Gatekeeper assessment

- **Platform / layer:** macos-user / userspace
- **Signal:** A legitimately distributed game bundle carries a resolved com.apple.quarantine xattr (or a provenance record) and passes SecAssessment; a main bundle with the quarantine bit stripped, a spoofed LSQuarantine entry, or a Gatekeeper-disabled host indicates the notarization gate was deliberately circumvented.
- **Mechanism:** getxattr(path, "com.apple.quarantine", ...) on the bundle, SecAssessmentCreate/SecAssessmentCopyResult (Gatekeeper assessment API, kSecAssessmentOperationTypeExecute), and the system-wide GK state via SecAssessmentControlReason / spctl-equivalent SecAssessmentTicketLookup for staple presence.
- **Catches:** Gatekeeper / notarization bypass (xattr-stripped or re-distributed bundle, globally disabled assessment) where a tampered or unnotarized game image is run as if it had passed first-launch validation.
- **FP risk:** high — Self-built, Steam/Epic-managed, or App-Store-delivered binaries legitimately lack quarantine xattrs; enterprises disable GK by policy. This is weak alone — gate hard by combining with cdhash/notarization-ticket presence and only treating it as corroborating signal, never standalone.
- **Horkos slot:** `daemon/macos/trust/GatekeeperProbe.cpp (new), Security.framework SecAssessment`
- **Refs:** getxattr com.apple.quarantine; SecAssessmentCopyResult; SecAssessmentTicketLookup; LSFileQuarantineEnabled

### 126. Entitlement set drift: blob the kernel granted vs blob signed into the binary

- **Platform / layer:** macos-user / kernel
- **Signal:** The entitlements the kernel actually granted the live task must byte-match the DER entitlements signed into the binary; an extra get-task-allow, debugger, or removed library-validation entitlement at runtime means the entitlement blob was substituted (re-signed binary or entitlement injection).
- **Mechanism:** csops(pid, CS_OPS_ENTITLEMENTS_BLOB, buf, size) to pull the kernel-held entitlement blob for the running task, compared against SecCodeCopySigningInformation(kSecCSRequirementInformation) kSecCodeInfoEntitlementsDict from the on-disk binary; diff the canonicalized sets.
- **Catches:** Entitlement injection / re-sign to grant debuggability or drop hardened-runtime restrictions (adding com.apple.security.get-task-allow or com.apple.security.cs.disable-library-validation) so a debugger/instrumentation can attach to the game.
- **FP risk:** low — Rosetta and some Apple-shimmed processes have kernel-added entitlements not present on disk; maintain a small allowlist of OS-injected entitlements and diff only the security-relevant keys (get-task-allow, disable-library-validation, debugger, dyld-environment-variables).
- **Horkos slot:** `daemon/macos/csops/EntitlementDiffProbe.cpp (new) under CsIntegrityProbe`

## DMA / Peripheral Hardware Trust  `dma-hardware`

### 127. Config-space serial-number / DSN capability forgery

- **Platform / layer:** cross / hardware
- **Signal:** A clean NIC/capture card exposes a Device Serial Number (DSN) extended capability whose EUI-64 OUI matches the vendor in the Vendor ID register; a PCILeech/FPGA board cloning a NIC ID either omits the DSN ext-cap entirely or presents an OUI inconsistent with its claimed Vendor/Subsystem-Vendor ID.
- **Mechanism:** Linux: read the full 4KB extended config space via /sys/bus/pci/devices/<BDF>/config (pread at offset 0x100+) and walk the PCIe extended-capability linked list for cap ID 0x0003 (DSN). Windows: SetupDi + IRP_MN_READ_CONFIG / the bus-interface BusInterfaceStandard.GetBusData, or the documented HalGetBusDataByOffset. macOS: IOPCIDevice configRead32 over IOKit.
- **Catches:** Hardware DMA inspection board spoofing a legitimate device's Vendor/Subsystem IDs to hide among real peripherals (ID-cloning).
- **FP risk:** medium — Many genuine low-cost devices legitimately omit the DSN ext-cap, so absence alone is weak; gate on the stronger case (DSN present but OUI mismatches the claimed vendor) and only treat plain absence as suspicious when combined with another flag (unbound driver, BAR mismatch). Whitelist known-DSN-less legitimate VID/DID pairs server-side.
- **Horkos slot:** `dma_detect/backends/{linux,win,macos}/ConfigSpaceForensics.* declared via new hk_dma_configspace_report in dma_detect/include/horkos/dma_detect.h`
- **Refs:** PCIe Base Spec 'Device Serial Number Extended Capability' (cap ID 0x0003); sysfs pci config; SetupAPI BusInterfaceStandard; IOKit IOPCIDevice::configRead32

### 128. Extended config-space readback stability / wraparound

- **Platform / layer:** cross / hardware
- **Signal:** On real silicon, reads of unimplemented config-space registers return architectural constants (RO bits stable, RsvdP read as 0, absent functions return 0xFFFFFFFF) and repeated reads are byte-identical; an FPGA shadow-config implementation often mirrors/aliases the low 256 bytes into the 0x100-0xFFF extended region or returns drifting/non-constant values for reserved offsets across repeated samples.
- **Mechanism:** Linux: multiple pread() samples of /sys/bus/pci/devices/<BDF>/config across the 0x000-0xFFF window, comparing offset 0x000-0x0FF vs 0x100-0x1FF for aliasing and checking RsvdP=0 invariants. Windows: repeated IRP_MN_READ_CONFIG via the PCI bus interface. macOS: IOPCIDevice configRead32 sweep.
- **Catches:** FPGA-emulated PCIe endpoint whose config-space is software-synthesized rather than hardwired (DMA-board firmware emulation).
- **FP risk:** medium — Some legitimate bridges and multifunction devices alias or sparsely populate extended space, and a few real devices have quirky reserved-bit behavior (the kernel maintains a PCI quirk table for exactly this). Gate by cross-referencing the kernel/OS quirk list and only flagging aliasing on simple single-function endpoints claiming a NIC/storage class.
- **Horkos slot:** `dma_detect/backends/*/ConfigSpaceForensics.* (shared with signal 1)`
- **Refs:** PCIe config-space RsvdP/RO semantics; Linux drivers/pci/quirks.c behavior; sysfs config pread; Windows IRP_MN_READ_CONFIG

### 129. MSI-X table BIR / PBA structural plausibility

- **Platform / layer:** cross / hardware
- **Signal:** A genuine device's MSI-X capability points its table and Pending-Bit-Array at a BAR (BIR field) and offset that fall inside that BAR's actual decoded size and are correctly aligned; spoofed/emulated endpoints frequently declare an MSI-X table count, BIR, or offset that is internally inconsistent (offset+table_size exceeds the referenced BAR, BIR points at an unimplemented BAR, or table size is implausible for the claimed device class).
- **Mechanism:** Read the MSI-X capability (cap ID 0x11) Message Control (table size), Table Offset/BIR and PBA Offset/BIR registers from config space, then read the referenced BAR's size via the standard write-all-ones/read-back sizing the OS already recorded (Linux: /sys/.../resource and config BAR registers; Windows: CM_Get_Device_Interface translated resources / IRP config read; macOS: IOPCIDevice deviceMemoryWithIndex length). Compare offset+entries*16 against BAR length.
- **Catches:** Emulated PCIe endpoint whose interrupt structures are cosmetic and don't map to real decoded memory (firmware-emulated inspection device).
- **FP risk:** low — Legitimate devices essentially never violate the spec's BIR/size containment; the main false trigger is mis-reading BAR size on 64-bit BARs (the high dword must be combined) — gate by correctly reconstructing 64-bit BARs before comparison and reporting only hard containment violations, not mere unusual offsets.
- **Horkos slot:** `dma_detect/backends/*/MsixForensics.* under a new hk_dma_msix_report field`
- **Refs:** PCIe MSI-X Capability (cap ID 0x11) Table Offset/BIR, PBA Offset/BIR; PCI BAR sizing; sysfs resource; IOPCIDevice deviceMemoryWithIndex

### 130. Expansion / option-ROM presence on a class that should have none

- **Platform / layer:** cross / hardware
- **Signal:** A device claiming to be a plain capture card, audio device, or generic NIC normally has its Expansion ROM Base Address (offset 0x30) either unimplemented (reads 0) or pointing at a vendor-signed UEFI/PCI option ROM with a valid 0xAA55 signature and matching PCIR vendor ID; an FPGA DMA board often either has no ROM where its cloned device would have one, or an option ROM whose PCIR Vendor/Device ID does not match the spoofed config-space IDs.
- **Mechanism:** Read Expansion ROM BAR at config offset 0x30; on Linux map /sys/bus/pci/devices/<BDF>/rom (after writing '1' to enable, per kernel rom sysfs protocol) and parse the 0xAA55 ROM header + PCIR structure, comparing PCIR VID/DID to the config-space VID/DID. Windows: read ROM via the PCI bus interface / firmware tables. macOS: IOPCIDevice config read of ROM BAR.
- **Catches:** ID-cloned hardware where the option ROM identity contradicts the spoofed config-space identity (inconsistent firmware identity).
- **FP risk:** medium — Many legitimate add-in cards ship with no option ROM at all, so ROM absence is only weakly suspicious; the strong signal is a present ROM whose PCIR IDs mismatch the device IDs. Mapping the ROM has a small side effect (enabling decode) so gate behind read-only confirmation and skip on devices whose driver is actively using the ROM region.
- **Horkos slot:** `dma_detect/backends/*/OptionRomForensics.*`
- **Refs:** PCIe Expansion ROM Base Address (0x30); PCI Firmware Spec PCIR data structure / 0xAA55 signature; Linux sysfs rom attribute protocol

### 131. BAR0 decoded-size vs declared-device-class mismatch (deep sizing)

- **Platform / layer:** cross / hardware
- **Signal:** For a given spoofed identity the expected BAR0 aperture is well-bounded (e.g. an Intel I210 NIC decodes a specific register-window size); an FPGA board reusing a generic PCIe IP core commonly exposes a BAR0 whose size, prefetchability, or 32/64-bit-ness does not match what the real device with that VID/DID exposes, or exposes a single huge prefetchable window used as a DMA scratch aperture.
- **Mechanism:** Reconstruct each BAR's true size via the OS-recorded sizing (Linux: combine the masked BAR registers in /sys/.../config plus /sys/.../resource lengths; Windows: IRP config read of BARs + CM translated resource descriptors; macOS: IOPCIDevice deviceMemoryWithIndex getLength) and compare {size, 64-bit flag, prefetchable flag} against a server-side reference table keyed on VID/DID/SVID.
- **Catches:** Spoofed device whose memory aperture geometry betrays generic FPGA IP rather than the cloned ASIC (BAR geometry forgery) — deeper than the implemented 'BAR0 size vs declared capability' by keying on per-VID/DID reference profiles.
- **FP risk:** medium — Hardware revisions and stepping changes legitimately alter BAR sizes, so the reference table must be revision-tolerant and server-maintained; gate by treating mismatch as a scored signal, never a standalone ban, and exclude VID/DIDs not present in the reference set rather than guessing.
- **Horkos slot:** `dma_detect/backends/*/BarProfile.* feeding a per-VID/DID reference table validated server-side`
- **Refs:** PCI BAR sizing (write 0xFFFFFFFF / read-back mask); sysfs resource & config; CM_Get_... translated resources; IOPCIDevice deviceMemoryWithIndex

### 132. Config-space read latency / TLP completion-timing fingerprint

- **Platform / layer:** cross / hardware
- **Signal:** Real ASIC endpoints answer config-read completions with low, tight-variance latency; an FPGA that synthesizes config-space in soft logic (or a board bridging config reads over a slower internal fabric) shows higher mean and higher jitter on repeated config reads of the same register relative to peer devices on the same root complex.
- **Mechanism:** Time a tight loop of identical config-space reads (same register) using a monotonic high-resolution clock (Linux clock_gettime CLOCK_MONOTONIC_RAW around pread of /sys/.../config; Windows QueryPerformanceCounter around the PCI bus-interface read; macOS mach_absolute_time around IOPCIDevice configRead32), collect a latency histogram, and compare distribution shape against sibling devices under the same root port.
- **Catches:** FPGA-emulated endpoint distinguishable by soft-logic config-path timing (emulation timing side-channel).
- **FP risk:** high — Latency is heavily confounded by OS scheduling, sysfs/IRP overhead, ASPM power states, and CPU contention; this is noisy and must be a low-weight corroborating signal only. Gate by sampling many iterations, using robust statistics (median/IQR), comparing only against same-root-port peers measured in the same pass, and never firing standalone.
- **Horkos slot:** `dma_detect/backends/*/TlpLatencyProbe.* (explicitly low-weight, server combines with structural signals)`
- **Refs:** clock_gettime CLOCK_MONOTONIC_RAW; QueryPerformanceCounter; mach_absolute_time; sysfs config / IRP_MN_READ_CONFIG

### 133. ACS / topology anomaly — peer-to-peer reachability under the root port

- **Platform / layer:** cross / hardware
- **Signal:** On a correctly configured system every downstream port / multifunction device that the platform expects to isolate has Access Control Services (ACS) capability with Source Validation and P2P Request Redirect enabled, and the device's IOMMU group reflects that isolation; a DMA board inserted behind a switch with ACS disabled (or a topology where the suspect device shares an IOMMU group with the GPU/RAM-mapping path) yields an oversized IOMMU group enabling peer-to-peer DMA that bypasses translation.
- **Mechanism:** Walk the ACS extended capability (cap ID 0x000D) control register bits on each switch downstream port and bridge in the path; on Linux corroborate with the membership size of /sys/kernel/iommu_groups/<n>/devices/. Windows: ACS bits via PCI bus-interface config reads of the bridge. macOS: IOPCIDevice config read of bridge ACS cap.
- **Catches:** DMA device positioned to exploit a P2P path that escapes IOMMU translation, or ACS-disabled switch insertion (topology-based DMA isolation bypass).
- **FP risk:** medium — Many consumer motherboards legitimately ship with multifunction devices and switches lacking ACS, producing large 'normal' IOMMU groups — a well-known pain point for VFIO users. Gate by scoring only when an unbound/bus-master-enabled suspect device shares a group with a sensitive endpoint, and maintain a server-side allowlist of common consumer chipset topologies.
- **Horkos slot:** `dma_detect/backends/*/AcsTopology.* extending hk_dma_report with group-membership detail`
- **Refs:** PCIe ACS Extended Capability (cap ID 0x000D) Source Validation / P2P Request Redirect; /sys/kernel/iommu_groups/<n>/devices/

### 134. Mid-session hot-plug / surprise-removal correlated to game start

- **Platform / layer:** cross / hardware
- **Signal:** A baseline gaming machine's PCIe topology is static after boot; a DMA board is frequently inserted (or its FPGA re-enumerates a new BDF) after the game and AC client have started, appearing as a hot-add / surprise-link-up event for a device that then enables bus-master without binding a driver.
- **Mechanism:** Subscribe to live device-arrival events rather than one-shot scanning. Linux: udev/netlink KOBJECT_UEVENT monitor on the pci subsystem (ADD/REMOVE actions) plus a re-scan diff. Windows: RegisterDeviceNotification / CM_Register_Notification for DEVICE_INTERFACE arrivals on GUID_DEVINTERFACE for PCI, timestamped against AC start. macOS: IOServiceAddMatchingNotification (kIOMatchedNotification) on IOPCIDevice.
- **Catches:** Post-start insertion or post-POST BDF-swap of a DMA inspection board (hot-plug-after-start, the FPGA re-enumeration case from the threat model).
- **FP risk:** medium — Thunderbolt/USB4 docks, external GPUs, hot-plug NVMe, and laptop dock attach legitimately generate PCIe hot-add events mid-session. Gate by correlating the arrival with the suspect-device structural flags (unbound + bus-master + ID/BAR anomaly) and by recognizing Thunderbolt root ports / known dock VID-DIDs as benign hot-plug domains.
- **Horkos slot:** `dma_detect/backends/*/HotplugMonitor.* emitting timestamped arrival events into the telemetry stream`
- **Refs:** udev netlink KOBJECT_UEVENT; CM_Register_Notification / RegisterDeviceNotification (DBT_DEVICEARRIVAL); IOServiceAddMatchingNotification kIOMatchedNotification

### 135. IOMMU fault-counter and unmapped-access telemetry

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** On a clean system the IOMMU/DMAR hardware logs essentially zero DMA remapping faults during steady-state gameplay; a DMA board issuing transactions to addresses outside its (absent or stale) IOVA mappings generates a stream of IOMMU page faults / 'unhandled context fault' events that the platform IOMMU driver records.
- **Mechanism:** Sample the kernel's IOMMU fault accounting: Linux read /sys/kernel/debug/iommu/ counters where exposed, and/or attach an eBPF tracepoint/kprobe to the iommu fault report path (e.g. the iommu:io_page_fault tracepoint / report_iommu_fault) to count faults per source-BDF over a window; cross-reference the faulting BDF against the suspect-device set. Windows analogue: DMA-remapping fault events surfaced via ETW (Microsoft-Windows-Kernel-* / DeviceGuard providers) where available.
- **Catches:** Active DMA reads/writes from a device not properly constrained by the IOMMU — direct evidence of out-of-policy DMA rather than mere capability (live unauthorized-DMA activity).
- **FP risk:** low — Steady-state IOMMU faults are rare on healthy systems; the main benign source is buggy drivers during init or specific GPU/driver fault-recovery flows, which are bursty-at-init not steady-state. Gate by ignoring boot/init windows, attributing faults to a specific BDF, and requiring the faulting BDF to also carry a structural suspicion flag. Requires CAP_BPF/CAP_SYS_ADMIN or debugfs read access — fall back to absent gracefully.
- **Horkos slot:** `kernel/linux/ eBPF program (iommu fault tracepoint) feeding the CO-RE ringbuf, mirrored into dma_detect telemetry; Windows path in dma_detect/backends/win/IommuFaultEtw.*`
- **Refs:** Linux iommu:io_page_fault tracepoint / report_iommu_fault; /sys/kernel/debug/iommu; eBPF kprobe/tracepoint; Windows ETW DeviceGuard/DMA-remapping events

### 136. HID report-descriptor structural fingerprint vs OS collection tree

- **Platform / layer:** windows-user / userspace
- **Signal:** A real branded mouse/keyboard emits a complex, vendor-specific HID report descriptor (usage pages, feature reports, vendor collections); an Arduino/Leonardo-class emulator ships a minimal textbook boot-protocol descriptor with a tell-tale byte sequence and no vendor usage pages.

- **Catches:** Synthetic input device: a microcontroller HID board presenting as a mouse/keyboard whose descriptor is the canonical Arduino HID library / V-USB / LUFA template rather than a genuine vendor descriptor.
- **FP risk:** medium — Legitimate cheap OEM mice and DIY keyboard firmware (QMK/VIA, ZMK) also emit minimal or library-derived descriptors. Gate by treating the descriptor hash as a server-side reputation lookup against a corpus of known-good vendor descriptors, never a local verdict; QMK/ZMK descriptors are distinguishable from Arduino-HID-library descriptors and can be allowlisted by their own fingerprint cluster.

### 137. USB device-descriptor BCD/string self-consistency audit

- **Platform / layer:** windows-user / userspace
- **Signal:** Genuine devices have internally consistent bcdUSB / bcdDevice / bMaxPacketSize0 / serial-string presence matching their VID/PID's known production runs; serial-bridge boards (CH340/CP2102/FTDI repurposed as HID) carry the bridge chip's stock bcdDevice and a null or library-default iSerialNumber that does not match the claimed HID vendor.

- **Catches:** Serial-bridge or repurposed-MCU device masquerading as an HID input device: claimed VID/PID inconsistent with the descriptor numerics or string-descriptor provenance of that VID/PID's real hardware.
- **FP risk:** medium — Legitimate composite dongles and KVM/USB-switch hardware also rewrite descriptors. Gate server-side by correlating the (VID,PID,bcdDevice,bMaxPacketSize0) tuple against a known-hardware corpus; flag only tuples that claim a major-vendor VID but carry a bridge-chip descriptor signature.

### 138. Raw Input source-handle to enumerated-device reconciliation

- **Platform / layer:** windows-user / userspace
- **Signal:** Every legitimate WM_INPUT pointer/keyboard packet carries an hDevice that resolves to a live entry in GetRawInputDeviceList; injected input (SendInput, driver-level synthetic injection, or hidden virtual HID) arrives with no hDevice (NULL source) or an hDevice whose device name/path is absent from the enumerated HID set.
- **Mechanism:** RegisterRawInputDevices for HID mouse/keyboard usage pages, then for each WM_INPUT compare RAWINPUTHEADER.hDevice against GetRawInputDeviceList + GetRawInputDeviceInfo(RIDI_DEVICENAME); flag deltas whose source handle has no matching enumerated USB endpoint.
- **Catches:** Synthetic/injected input: pointer deltas appearing with no corresponding enumerated USB endpoint, including SendInput-class injection and virtual-HID drivers not backed by physical enumeration.
- **FP risk:** medium — Legitimate accessibility tools, remote-desktop (RDP/Parsec/Moonlight) and on-screen keyboards inject via SendInput and produce NULL-source RAWINPUT. Gate by reporting the session type (WTSGetActiveConsoleSessionId / remote-session flag) and a process-context tag so the server can suppress when a known remote-control session or accessibility flag is active.
- **Horkos slot:** `ac/src/backends/win/input/RawInputSourceReconcileWin.cpp`
- **Refs:** GetRawInputDeviceList, GetRawInputDeviceInfo(RIDI_DEVICENAME), RAWINPUTHEADER.hDevice, RegisterRawInputDevices

### 139. USB polling-interval (bInterval) vs observed report cadence mismatch

- **Platform / layer:** windows-user / userspace
- **Signal:** A real HID device's endpoint advertises a bInterval that bounds its physical report rate (e.g. 1ms for 1000Hz, 8ms for 125Hz boot mice); an MCU emulator that bit-bangs reports can deliver them at a cadence finer than, or mathematically inconsistent with, its own declared endpoint bInterval, or at a rate the declared full/low-speed endpoint physically cannot sustain.
- **Mechanism:** Read the interrupt-IN endpoint descriptor (bInterval, wMaxPacketSize, speed) via IOCTL_USB_GET_DESCRIPTOR_FROM_NODE_CONNECTION, then compare against per-report QueryPerformanceCounter arrival timestamps captured in the WM_INPUT path; report the ratio of observed rate to descriptor-permitted rate.
- **Catches:** Synthetic input device whose firmware-generated report timing violates the USB endpoint contract it advertises — finer-grained than the already-implemented zero-jitter check because it indicts the device against its own descriptor, not against a fixed jitter threshold.
- **FP risk:** medium — Legitimate high-end mice with adjustable polling and USB hubs that batch/coalesce transfers shift observed cadence. Gate by requiring a sustained statistical violation (observed rate exceeding the descriptor ceiling, which is physically impossible for a compliant device) rather than mild jitter, and by excluding wireless dongles where the receiver re-clocks reports.
- **Horkos slot:** `ac/src/backends/win/input/PollIntervalCoherenceWin.cpp + server/telemetry/src/input_cadence.rs`
- **Refs:** IOCTL_USB_GET_DESCRIPTOR_FROM_NODE_CONNECTION (USB_ENDPOINT_DESCRIPTOR.bInterval), QueryPerformanceCounter, WM_INPUT

### 140. evdev/uinput virtual-device provenance on Linux/Steam Deck

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A physical input device's struct input_dev has a backing USB/Bluetooth phys path and bus type (BUS_USB); a synthetic device created via /dev/uinput has bus type BUS_VIRTUAL (or no parent in the USB device tree) yet emits EV_REL/EV_KEY events — pointer deltas with no enumerated USB endpoint.

Gate by allowlisting the creating process (Steam's input stack) via the existing exec/process-provenance pipeline and reporting the uinput creator PID so the server attributes the virtual device to a trusted origin before any verdict; on Game Mode, Steam's own virtual devices are the dominant benign source and must be baselined.

## Client Self-Integrity  `memory-integrity-selfcheck`

### 145. Cross-view .text hash divergence (mapped image vs section object vs disk)

- **Platform / layer:** cross / userspace
- **Signal:** A clean process yields three identical SHA-256 hashes for our .text when read via our own in-process VA, via the kernel-cached section object, and via re-reading the on-disk file; an inline patch or COW-redirected page makes the in-process view diverge from the other two.

- **Signal:** When a guarded function executes once, a clean run trips the AC's re-armed PAGE_GUARD canaries a fixed, small number of times with microsecond-scale spacing; a stepper that walks the function instruction-by-instruction produces a long burst of guard-fault re-arms with characteristic uniform millisecond-scale inter-arrival timing (one fault per executed instruction window) that no normal control flow generates.
- **Mechanism:** Re-arming PAGE_GUARD via VirtualProtect inside the guard-fault VEH, timestamping each STATUS_GUARD_PAGE_VIOLATION with QueryPerformanceCounter, and analyzing the inter-arrival distribution and count per logical invocation; a tight high-count uniform-cadence burst is the single-step fingerprint.
- **Catches:** Trap-flag (EFLAGS.TF) and DR-based single-stepping through guarded code (timing-trace observability that complements raw RDTSC-delta stepping detection).
- **FP risk:** medium — Heavy CPU contention, JIT recompilation, and frequent legitimate re-entry inflate fault counts; gate by correlating with EFLAGS.TF/DR6 state read in the same VEH and requiring the uniform-cadence statistical signature, reporting the histogram rather than locally banning.
- **Horkos slot:** `ac/src/timing/guard_fault_cadence_win.cpp (iface ac/include/horkos/timing/guard_cadence.h)`
- **Refs:** VirtualProtect(PAGE_GUARD), STATUS_GUARD_PAGE_VIOLATION, AddVectoredExceptionHandler, CONTEXT.EFlags(TF)/Dr6, QueryPerformanceCounter

### 162. CPUID VMEXIT-latency leaf-fan baseline drift

- **Platform / layer:** windows-user / hardware
- **Signal:** On bare metal the serializing latency of CPUID is roughly uniform across leaves; under a hypervisor each CPUID is a VMEXIT and stealth VMMs emulate only a subset of leaves (e.g. hypervisor-reserved 0x4000xxxx and 0x1/0xB topology) with handler-specific cost, so the per-leaf latency fan-out develops a distinctive non-uniform profile distinct from the already-implemented single-leaf CPUID-latency check.
- **Mechanism:** Time __cpuidex across a fixed sweep of leaves (0x0,0x1,0x7,0xB,0x15,0x40000000-0x40000010) with RDTSCP fences, building a per-leaf latency vector; bare-metal baselines are near-flat while a VMM's selectively-emulated leaves spike, and the leaf-fan shape fingerprints the hypervisor family.
- **Catches:** Hypervisor-assisted cheats and TSC/CPUID-hiding VMMs that pass a single-leaf timing check but reveal themselves in the cross-leaf cost profile.
- **FP risk:** medium — Legitimate hypervisors (Hyper-V/VBS on by default in Win11, cloud gaming, enterprise VDI) produce the same fan-out; gate strictly as a VM-context tag combined with the APERF/MPERF skew and hypervisor-present bit for server-side scoring, never a standalone ban given how many clean machines run VBS.
- **Horkos slot:** `ac/src/timing/cpuid_leaf_fan_win.cpp (cross-platform core in ac/include/horkos/timing/cpuid_fan.h, x86-only guard)`
- **Refs:** __cpuidex, __rdtscp, _mm_lfence, CPUID leaf 0x40000000 (hypervisor vendor), leaf 0x15, leaf 0xB

### 163. Raw HID report-to-render provenance delta

- **Platform / layer:** cross / userspace
- **Signal:** On a clean session every applied aim_delta is backed by a Raw Input WM_INPUT (or evdev EV_REL) packet whose hardware timestamp precedes the render tick; injected aim shows view-angle motion with no matching upstream HID report or with a report timestamp that post-dates the angle change.

Client emits per-tick the count of HID reports consumed, their summed raw counts, and the newest HID hardware timestamp alongside the applied angle delta.
- **Catches:** Synthetic input / aimbot that writes view angles or mouse deltas directly into the game without generating a real HID report (memory-write aim, SetCursorPos/mouse_event injection, virtual HID with mismatched timing).
- **FP risk:** medium — Legitimate remapping/virtual-HID software (reWASD, Steam Input, Razer Synapse, AutoHotKey-free interception drivers, accessibility head-trackers) also generates HID reports lacking a physical device. Gate by attesting the HID source device path/VID-PID against an attested enumeration and treating virtual-HID provenance as a separate cohort, not an instant flag; let the server model decide.

### 164. Sub-count residual / quantization-floor violation

- **Platform / layer:** cross / userspace
- **Signal:** Human mouse motion arrives in integer HID counts scaled by a fixed DPI->sensitivity transform, so applied angle deltas cluster on a discrete lattice with a measurable quantization residual; software-generated micro-corrections produce sub-count angle adjustments finer than any integer HID count can express.
- **Mechanism:** Client carries the raw integer HID count (RAWMOUSE.lLastX/lLastY or EV_REL value), the active in-game sensitivity/DPI scalar, and the actually-applied angle delta per tick. Server reconstructs expected_angle = raw_count * scalar and reports the residual; persistent non-zero residual = motion not sourced from integer HID counts.
- **Catches:** Sub-pixel aim-correction / micro-smoothing aimbots and silent-aim that nudge the view by fractional amounts no physical mouse count can produce; also raw-input-spoofers that forget to quantize.
- **FP risk:** medium — Mouse smoothing/acceleration (Windows Enhance Pointer Precision), in-engine interpolation, and high-polling-rate fractional accumulation legitimately create sub-count residuals. Gate by sampling the OS pointer-acceleration state (SystemParametersInfo SPI_GETMOUSE) and the engine's own smoothing config, and only flag residuals that violate the lattice AFTER accounting for declared smoothing.
- **Horkos slot:** `server/telemetry/src/schema.rs (TickPayload: hid_raw_dx, hid_raw_dy, sens_scalar_q16, applied_angle_dx, applied_angle_dy) + sdk/src/input/quantization_probe`
- **Refs:** RAWMOUSE.lLastX, evdev EV_REL value, SystemParametersInfo SPI_GETMOUSE, SPI_GETMOUSESPEED

### 165. Polling-interval jitter spectrum

- **Platform / layer:** cross / userspace
- **Signal:** A real USB/Bluetooth mouse delivers HID reports at a hardware-fixed polling cadence (1000/500/125 Hz) with bounded jitter and occasional dropped frames; injected input arrives synchronized to the game frame loop or to a software timer, collapsing the inter-report-interval distribution to a frame-locked spike.
- **Mechanism:** Client timestamps each WM_INPUT/evdev report with QueryPerformanceCounter / clock_gettime(CLOCK_MONOTONIC_RAW) and emits per-tick the inter-arrival histogram (or compact moments: mean, variance, count of intervals == frame period) of HID reports since the last tick. Server builds the polling-rate spectrum and looks for frame-locked or timer-quantized cadence.
- **Catches:** Frame-hook / present-hook aimbots and tick-synchronized input emulators whose injected reports inherit the render or v-sync cadence instead of the USB polling cadence.
- **FP risk:** medium — Wireless mice with power-save report batching, USB hubs, and some KVM switches alter the cadence; v-synced low-end systems can frame-lock real input via coalescing. Gate by recording the attested device polling rate (HID descriptor / USB bInterval) and comparing the observed spectrum to THAT device's nominal rate, not a global prior.
- **Horkos slot:** `server/telemetry/src/schema.rs (TickPayload: hid_interval_mean_ns, hid_interval_var_ns, hid_interval_framelock_count) + sdk/src/input/polling_jitter_probe`
- **Refs:** QueryPerformanceCounter, clock_gettime CLOCK_MONOTONIC_RAW, HID report descriptor bInterval, RIDEV_INPUTSINK

### 166. Flick onset-to-target curvature signature

- **Platform / layer:** cross / server
- **Signal:** Human flicks are ballistic-then-corrective (Fitts/2-phase: fast open-loop launch, overshoot, visual-feedback settle) producing a characteristic minimum-jerk velocity profile; bot flicks are point-to-point with monotone deceleration onto the exact target angle and near-zero overshoot, giving an anomalous curvature/jerk signature.
- **Mechanism:** Client emits per-tick angular velocity and acceleration (third-difference jerk) of the view vector plus the angular distance to the nearest enemy hitbox center (from the game's own target list — already client-authoritative). Server segments flick events and fits the velocity profile, scoring overshoot ratio, settle-time, and jerk minima against a human-floor model in the ort/ONNX path.
- **Catches:** Snap/flick aimbots and humanized-curve aimbots whose interpolation curve (bezier/easing) lacks the stochastic overshoot+microcorrection of biological motor control.
- **FP risk:** high — High-skill players, low-sensitivity flick-heavy playstyles, and controller players with aim-assist all produce smooth low-jerk arcs that mimic eased interpolation. This is a population-model signal, not a per-event flag; gate on long-horizon distributional drift per player_id and require corroboration from a provenance signal before any enforcement weight.
- **Horkos slot:** `server/telemetry/src/schema.rs (TickPayload: ang_vel, ang_accel, ang_jerk, dist_to_nearest_target_rad) + server/ban-engine feature extractor`
- **Refs:** minimum-jerk model, Fitts's law; client view-matrix + engine target list (no OS API — pure feature transport)

### 167. Reaction-latency floor vs. visibility-onset

- **Platform / layer:** cross / server
- **Signal:** Time from a target first becoming render-visible (un-occluded, on-screen) to the first corrective aim impulse toward it has a human physiological floor (~150-250ms with a long right tail); a triggerbot/aim-snap reacts in sub-frame time with no tail, collapsing the reaction-time distribution below the floor.
- **Mechanism:** Client carries per-tick a visibility-onset timestamp for each target that crossed from occluded->visible (from the engine's own occlusion/PVS query) and the timestamp of the first aim impulse and fire event toward it, all stamped with QueryPerformanceCounter / CLOCK_MONOTONIC_RAW. Server computes the per-player reaction-time CDF and tests the left tail against the human floor.
- **Catches:** Triggerbots and aim-on-visible bots that act on engine state in the same frame the target becomes visible, faster than human optic-motor latency.
- **FP risk:** medium — Pre-aiming, pre-firing on audio cues, and prediction (player already tracking the spot) produce legitimately sub-floor apparent reactions. Gate by requiring the impulse to be a genuine direction CHANGE toward a newly-visible target (not continuation of prior tracking) and by modeling the full CDF rather than single events; corroborate with provenance.
- **Horkos slot:** `server/telemetry/src/schema.rs (TickPayload: target_vis_onset_ts_ns, first_impulse_ts_ns, fire_ts_ns, impulse_is_direction_change) + server/ban-engine reaction model`

### 168. Recoil-compensation phase-lock residual

- **Platform / layer:** cross / server
- **Signal:** A human compensating recoil pulls down with noisy, lagging, amplitude-imperfect counter-motion; a recoil-control script applies the exact inverse of the weapon's deterministic recoil pattern, so the residual (view-motion minus expected recoil-inverse) collapses to near-zero variance and zero phase lag during sustained fire.
- **Mechanism:** Client emits during full-auto fire (input_state fire-bit set) the per-tick applied aim delta and the weapon id / shot index (engine state). Server knows the weapon's recoil curve from signed rule data, computes residual = applied_delta - (-recoil_step), and tracks residual variance and cross-correlation lag between applied motion and the recoil pattern.
- **Catches:** No-recoil / recoil-compensation scripts and macros (mouse-software macro, Logitech/Razer onboard macros, Cronus/XIM translation) that subtract the known pattern with super-human regularity.
- **FP risk:** medium — Drilled players develop low-variance recoil control, and per-weapon muscle memory genuinely tightens the residual. Gate by requiring phase-lag at or below human reaction latency (a script can be zero-lag; a human cannot) and by comparing residual variance against the player's own learning curve over time rather than a fixed threshold.
- **Horkos slot:** `server/telemetry/src/schema.rs (TickPayload: applied_angle_dx, applied_angle_dy, weapon_id, shot_index, fire_active) + server/ban-engine recoil-residual model (signed recoil rules)`

### 169. Target-switch latency vs. saccade floor

- **Platform / layer:** cross / server
- **Signal:** When the optimal target changes (new higher-threat enemy appears), human re-acquisition incurs a saccadic decision+motor latency and a curved transit; an aimbot re-targets in one tick with a straight-line angular jump, producing an anomalously short and uniform target-switch latency distribution.
- **Mechanism:** Client carries per-tick the id of the currently-aimed-at target and the set of candidate targets with their angular offsets (engine target list). Server detects target-switch events (aim vector discretely re-locks onto a different enemy), measures switch latency and transit straightness, and tests against a human saccade-reaction floor.
- **Catches:** Multi-target / best-target-selection aimbots whose retarget logic switches lock with no decision latency and no transit curvature.
- **FP risk:** high — Spray-transfer between close-grouped enemies is a legitimate trained skill with short switch times. Gate on angular separation (only score switches across large angular gaps where human saccade latency is unavoidable) and treat as distributional, never per-event; corroborate with provenance and reaction-floor signals.
- **Horkos slot:** `server/telemetry/src/schema.rs (TickPayload: aimed_target_id, candidate_target_offsets[], switch_event_flag) + server/ban-engine target-switch model`

### 174. Peeker-advantage exceeding measured RTT+jitter budget

- **Platform / layer:** server / server
- **Signal:** The time between a player's hitbox first becoming mutually visible (server-authoritative) and that player landing a registered shot is shorter than their own measured network RTT plus interpolation delay plus the human reaction floor — i.e. they fired before the information could physically have reached them.

### 180. Recoil/spread-aware compensation under zero visual feedback

- **Catches:** No-recoil / recoil-compensation cheats that subtract the exact server-computed recoil, including the randomized component the player cannot see or have memorized.
- **FP risk:** low — Skilled players learn deterministic recoil patterns, so correlation to the MEAN pattern is legitimate. Gate strictly on correlation to the per-shot RANDOM component (which is unlearnable) under no-LoS no-tracer conditions, and require statistical significance over many magazines.
- **Horkos slot:** `server/telemetry/src/analyzers/recoil_rng_corr.rs`
- **Refs:** server-side weapon recoil RNG (authoritative); aim_delta_* integration (TickPayload); CLOCK_MONOTONIC

## Network-Layer Integrity  `network-anomaly`

### 181. QPC-vs-NIC-hardware-timestamp skew (send-cadence ground truth)

- **Platform / layer:** windows-user / network
- **Signal:** On a clean client the wall-clock interval between successive uplink datagrams measured by QueryPerformanceCounter agrees within a few hundred microseconds with the NIC's hardware/software TX timestamp for the same packets; a lag-switch or packet-pacing shim that buffers and re-bursts datagrams shows the application-level QPC cadence diverging from the actual on-wire TX timestamp sequence.
- **Mechanism:** QueryPerformanceCounter for the app-level send timestamp; SO_TIMESTAMPING via WSARecvMsg/WSASendMsg control messages (SIO_TIMESTAMPING WSAIoctl, SO_TIMESTAMP) to read the kernel/NIC TX/RX timestamp of the same datagram; the client reports the per-packet (qpc_ts, hw_tx_ts) pair delta as a new tick field.
- **Catches:** User-mode WFP/LSP send-shim, lag-switch tools that queue outbound packets in a userland proxy or TAP, and pacing layers inserted between the game socket and the wire (the on-wire timestamp lags or batches relative to the app's QPC send call).
- **FP risk:** medium — NIC offload/coalescing (LSO, interrupt moderation), Wi-Fi power-save buffering, and VPN/tunnel adapters legitimately batch TX timestamps. Gate by requiring sustained, directional divergence (TX consistently lagging app cadence by a growing queue depth) over many ticks rather than per-packet jitter, and exclude sessions where the bound adapter reports a tunnel/virtual media type.
- **Horkos slot:** `sdk/src/backends/win/NetTimingWin.cpp implementing a new hk::net::TxCadenceProbe declared in sdk/include/horkos/net_timing.h; reported via a new TickPayload field tx_cadence_skew_ns in server/telemetry/src/schema.rs`
- **Refs:** WSAIoctl SIO_TIMESTAMPING, SO_TIMESTAMP, WSARecvMsg/LPFN_WSARECVMSG control messages, QueryPerformanceCounter

### 182. Monotonic-vs-realtime clock-domain drift on the send loop

- **Platform / layer:** cross / network
- **Signal:** A clean client's network send loop advances CLOCK_MONOTONIC_RAW and CLOCK_REALTIME at the same rate (modulo NTP slew); a speed-hack that hooks the game's timing source or runs the process under a manipulated clock (frozen/scaled QueryPerformanceCounter, vDSO clock_gettime hook, or a hypervisor that scales TSC) produces a measurable ratio drift between the two independent clock domains sampled around each packet send.
- **Mechanism:** clock_gettime(CLOCK_MONOTONIC_RAW) vs clock_gettime(CLOCK_REALTIME) on Linux/macOS; QueryPerformanceCounter vs GetSystemTimePreciseAsTime / QueryUnbiasedInterruptTimePrecise vs rdtsc on Windows; client computes the per-window rate ratio of the two domains and reports it.
- **Catches:** Time-dilation speed manipulation that hooks a single timing API (the hooked domain drifts from the un-hooked reference), and TSC-scaling hypervisors used to slow the game clock relative to the network/server tick.
- **FP risk:** medium — NTP/chrony step+slew, suspend/resume, and VM live-migration legitimately desync REALTIME from MONOTONIC. Gate by ignoring discrete steps (NTP corrections are step or bounded-slope) and alerting only on a sustained, smoothly-scaled ratio (e.g. monotonic running 0.8x realtime) that correlates with abnormal movement-per-tick.
- **Horkos slot:** `sdk/src/ClockDomainProbe.cpp (cross-platform, dispatching through platform/platform.h time primitives) feeding a clock_ratio_ppm field in server/telemetry/src/schema.rs`
- **Refs:** clock_gettime CLOCK_MONOTONIC_RAW/CLOCK_REALTIME, QueryUnbiasedInterruptTimePrecise, GetSystemTimePreciseAsFileTime, __rdtsc

### 183. TCP retransmission/RTT struct vs reported lag divergence (fake-lag unmasking)

- **Platform / layer:** windows-user / network
- **Signal:** When a player exhibits gameplay symptoms of lag (rubber-banding, delayed hits) the kernel's own per-connection TCP_INFO (SmoothedRtt, RetransmitCount, congestion state) should corroborate real network distress; a lag-switch that drops only the application's outbound game packets while the socket stack stays healthy shows healthy kernel RTT/zero retransmits despite the client self-reporting or exhibiting stall — kernel truth contradicts the symptom.
- **Mechanism:** WSAIoctl SIO_TCP_INFO (TCP_INFO_v1) on the game's TCP control channel, or GetPerTcpConnectionEStats / GetTcpStatisticsEx for the connection; Linux equivalent getsockopt(TCP_INFO) tcpi_rtt/tcpi_retrans. Client samples the kernel connection struct alongside its own perceived stall and reports the pair.
- **Catches:** Selective-drop lag-switches and userland packet droppers that stall the UDP game channel while leaving the TCP control/matchmaking socket and the OS network stack healthy, exposing a contradiction the server can adjudicate.
- **FP risk:** medium — Genuine asymmetric congestion (one queue saturated, another idle) and dual-homed routing can legitimately make one channel healthy while another stalls. Gate by requiring the contradiction to coincide with movement-extrapolation outliers and to recur across multiple stall events, not a single transient.
- **Horkos slot:** `sdk/src/backends/win/TcpInfoProbeWin.cpp + posix counterpart, behind hk::net::ConnHealthProbe in net_timing.h; new conn_rtt_us / conn_retrans fields in schema.rs`
- **Refs:** WSAIoctl SIO_TCP_INFO TCP_INFO_v1, GetPerTcpConnectionEStats, getsockopt SOL_TCP TCP_INFO tcp_info

### 184. Socket-buffer backlog growth without OS congestion (held-uplink detector)

- **Platform / layer:** cross / network
- **Signal:** On a clean stall, outbound bytes queued in the kernel send buffer drain at the OS-negotiated rate; a lag-switch implemented by suspending the game's send thread or holding packets in a userland queue leaves the kernel send-buffer occupancy LOW (nothing handed to the kernel) while the application's own outbound queue grows — the discriminator is 'app queue rising while kernel TX queue empty and link reports no congestion'.
- **Mechanism:** ioctl(SIOCOUTQ)/TIOCOUTQ on Linux and getsockopt(SO_NWRITE)/FIONWRITE on macOS to read unsent kernel-buffer bytes; Windows SIO_IDEAL_SEND_BACKLOG_QUERY / idealsendbacklog notifications plus the app's own send-ring depth. Client reports (app_queue_depth, kernel_unsent_bytes) per window.
- **Catches:** Lag-switches that pause the send thread or buffer outbound game state in process memory (the classic 'freeze then dump' burst), distinguishable from real congestion where the kernel send buffer fills because the link cannot drain it.
- **FP risk:** low — A genuinely saturated CPU starving the send thread can mimic app-side backlog; gate by cross-checking that overall process scheduling isn't starved (the held-uplink case has a healthy process otherwise) and that the subsequent burst exceeds physically-plausible aggregate throughput for the link.
- **Horkos slot:** `sdk/src/SendBacklogProbe.cpp dispatching via platform/platform.h; app_queue_depth + kernel_unsent_bytes fields in server/telemetry/src/schema.rs`
- **Refs:** ioctl SIOCOUTQ, getsockopt SO_NWRITE, WSAIoctl SIO_IDEAL_SEND_BACKLOG_QUERY/SIO_IDEAL_SEND_BACKLOG_CHANGE

### 185. Input-frame monotonic sequence and capture-timestamp coherence

- **Platform / layer:** windows-user / network
- **Signal:** Each game-input frame carries the OS input timestamp from the raw-input/HID pipeline; on a clean client these timestamps are monotonic, dense, and rate-consistent with the packet send cadence, whereas duplicated or replayed input frames (out-of-order/duplicated input injection) show repeated, non-monotonic, or backdated OS input timestamps that the app-level send sequence does not — the OS capture clock contradicts the app frame ordering.
- **Mechanism:** GetMessageTime / RAWINPUT + GetCurrentInputMessageSource (Windows) and the timestamp field of input events; on the send path the client correlates each outbound input frame's embedded OS-capture timestamp against its send sequence number and reports any non-monotonic or duplicated-timestamp frames.
- **Catches:** Out-of-order / duplicated input-frame injection and input replay (a synthetic frame reuses or fabricates a capture timestamp that breaks the OS-monotonic invariant the genuine HID pipeline guarantees).
- **FP risk:** medium — Legitimate macro/remap software (Razer Synapse, Logitech G HUB, AutoHotkey for accessibility) and remote-desktop input also synthesize events with timestamps. Gate via GetCurrentInputMessageSource (which flags injected vs hardware origin) and treat synthetic-origin frames as a soft signal scored server-side rather than an immediate verdict.
- **Horkos slot:** `sdk/src/backends/win/InputFrameProbeWin.cpp behind hk::input::FrameCoherenceProbe; reported as input_frame_anomaly_flags in server/telemetry/src/schema.rs (declared alongside event_schema.h additions, documented in data-categories.md)`
- **Refs:** GetMessageTime, GetRawInputData RAWINPUT, GetCurrentInputMessageSource, RIM_INPUT/RIM_INPUTSINK

### 186. Client-send-interval vs server-receive-interval auto-correlation (relativistic burst detector)

- **Platform / layer:** server / server
- **Signal:** For a clean client the distribution of inter-arrival gaps at the server (server_received_ts deltas) is a jittered-but-stationary version of the client's nominal tick interval; a lag-switch produces a bimodal arrival pattern — a silent gap (stall) immediately followed by a compressed burst whose summed displacement equals the gap — a signature distinguishable from ordinary jitter by the gap/burst anti-correlation, which ordinary packet loss does not produce.
- **Mechanism:** Server-side only: the existing server_received_ts in TickPayload plus the client tick counter; the ban-engine computes the running auto-correlation / gap-then-burst detector over arrival deltas vs tick deltas. No new client API — pure consumer of fields already in schema.rs.
- **Catches:** Lag-switch stall+burst and packet-cadence time-dilation (the conserved-displacement burst after an engineered silence is a positive signature, unlike random loss which drops state permanently rather than dumping it late).
- **FP risk:** low — Mobile/Wi-Fi roaming and NAT rebind can cause a real gap-then-burst once at handover. Gate by requiring repeated occurrences within a session and a tight conservation ratio (burst displacement ≈ gap duration × tick rate), which random network events do not satisfy repeatedly.
- **Horkos slot:** `server/ban-engine/src/ (new arrival_cadence.rs module consuming TickPayload), no client change; threshold rule carried as a signed rule via the existing rule plumbing`
- **Refs:** TickPayload.server_received_ts and TickPayload.tick (server/telemetry/src/schema.rs)

### 187. Dual-channel RTT divergence via independent probe socket

- **Platform / layer:** cross / network
- **Signal:** The integrity client opens its own low-rate probe/heartbeat socket to the same edge as the game traffic; on a clean host the probe-channel RTT tracks the game-channel RTT within a tight band, but a lag-switch or selective shim that targets only the game socket/port leaves the probe RTT healthy while the game RTT spikes — divergence between two channels to the same endpoint isolates application-targeted manipulation from real path congestion.
- **Mechanism:** A dedicated UDP/ICMP echo socket (separate ephemeral port to the same server IP) timestamped with QueryPerformanceCounter/clock_gettime; on Linux SO_TIMESTAMPING gives kernel RX times. Client reports (game_rtt, probe_rtt) pairs; server compares divergence.
- **Catches:** Port/process-targeted lag-switches and packet filters that throttle only the game flow (matched by 5-tuple) while leaving collateral traffic untouched — real congestion hits both channels, targeted manipulation hits one.
- **FP risk:** low — QoS/SD-WAN policies and ISP traffic shaping that deprioritize a specific port can legitimately diverge the two channels. Gate by correlating divergence with gameplay anomalies and by sampling the probe on the same port-range class where feasible so benign per-port QoS is less likely to split them.
- **Horkos slot:** `sdk/src/ProbeChannel.cpp implementing hk::net::ProbeChannelProbe in net_timing.h; game_rtt_us/probe_rtt_us fields in schema.rs; data-categories.md entry for the new probe-RTT category`
- **Refs:** QueryPerformanceCounter, clock_gettime, SO_TIMESTAMPING, sendto/recvmsg on a dedicated socket

### 188. Path/interface change without OS route or link-state event

- **Platform / layer:** windows-user / network
- **Signal:** When game traffic genuinely reroutes (new gateway, NIC switch, VPN up/down), the OS emits a route-change / interface-change notification and the bound socket's effective source address or interface index changes accordingly; a transparent userland interceptor that re-injects traffic through a TAP/proxy can shift the on-wire path or TTL/source without any corresponding OS NotifyRouteChange2 event — packets behave as if rerouted while the routing table is silent.
- **Mechanism:** NotifyRouteChange2 / NotifyIpInterfaceChange / NotifyUnicastIpAddressChange (Windows iphlpapi) and GetBestInterfaceEx for the game's destination; Linux rtnetlink RTMGRP_IPV4_ROUTE/LINK via NETLINK_ROUTE. Client snapshots bound-interface/route identity per window and reports a hash plus a 'changed-without-event' flag.
- **Catches:** Transparent TAP/proxy-based interceptors and lag/jitter tools that splice into the path below the socket but above the wire (TTL or source-interface shifts that the OS routing subsystem never authorized).
- **FP risk:** medium — Legitimate VPN clients, split-tunnel software, and fast Wi-Fi/cellular failover do reroute — but they go through the OS stack and therefore DO fire the notification. The signal is specifically 'observed path identity changed while no OS route/link event fired'; gate by requiring the absence of the notification, which benign rerouting cannot satisfy.
- **Horkos slot:** `sdk/src/backends/win/RouteWatchWin.cpp + posix rtnetlink counterpart behind hk::net::RouteIntegrityProbe; reported as route_change_unattested flag in schema.rs`
- **Refs:** NotifyRouteChange2, NotifyIpInterfaceChange, GetBestInterfaceEx, rtnetlink NETLINK_ROUTE RTM_NEWROUTE/RTM_NEWLINK

### 189. Loopback/local-proxy interposition on the game 5-tuple

- **Platform / layer:** windows-user / network
- **Signal:** On a clean client the game socket's remote peer in the OS connection table is the real server endpoint and no local process owns a socket sitting between the game and the wire on that flow; a man-in-the-middle lag/jitter proxy shows up as a loopback or 127.0.0.1/local-address owner in the TCP/UDP table, or the game's connection terminating at a local PID rather than the server IP — a self-inspection of the host's own socket ownership table reveals the interposer.
- **Mechanism:** GetExtendedTcpTable / GetExtendedUdpTable with TCP/UDP_TABLE_OWNER_PID_ALL (Windows iphlpapi) to map the game flow to its owning PID and remote endpoint; Linux ss/netlink sock_diag (INET_DIAG) and /proc/net/{udp,tcp}; macOS via proc_pidfdinfo/PROC_PIDFDSOCKETINFO. Client reports whether the game flow terminates at the expected server IP or at a local proxy.
- **Catches:** Local MITM/proxy lag-switches and jitter injectors that re-home the game connection through 127.0.0.1 or a local listener, plus debug proxies (Clumsy, NetLimiter-style local hooks) sitting on the flow.
- **FP risk:** high — Legitimate local proxies are common: corporate TLS-inspection, Steam/Epic overlays, parental-control filters, and bandwidth managers (NetLimiter) all interpose on flows. Gate hard server-side: treat as a low-weight contextual signal, allowlist signed known-good interposers by image hash (cross-referenced with the existing image_sha256 catalog), and only escalate when the loopback owner coincides with cadence/RTT anomalies.
- **Horkos slot:** `sdk/src/backends/win/SocketTableProbeWin.cpp + posix sock_diag counterpart behind hk::net::FlowOwnerProbe; flow_owner_local flag + owner_image_hash in schema.rs and a new data-categories.md row`
- **Refs:** GetExtendedTcpTable/GetExtendedUdpTable TCP_TABLE_OWNER_PID_ALL, sock_diag INET_DIAG, proc_pidfdinfo PROC_PIDFDSOCKETINFO, /proc/net/udp

## Cross-Platform — Analysis-Tooling Presence  `anti-analysis-environment`

### 190. Hardware Debug Register Occupancy (DR0-DR7) via Self-Inspect Context

- **Platform / layer:** windows-user / userspace
- **Signal:** On a clean process all four hardware breakpoint slots Dr0-Dr3 and the control register Dr7 read as zero across every thread; a non-zero Dr7 enable bit or a populated Dr0-Dr3 with no Horkos-owned breakpoint indicates an external debugger (or hardware-breakpoint-based hook) is set on the game.
- **Mechanism:** GetThreadContext with CONTEXT_DEBUG_REGISTERS on each enumerated thread (Thread32First/Next over TH32CS_SNAPTHREAD), reading Dr0..Dr3 and Dr7; cross-checked against the kernel driver's own thread-create records so newly spawned threads cannot be missed. Kernel-side corroboration via PsGetContextThread.
- **Catches:** Stealth debugger attach that scrubs PEB.BeingDebugged/NtGlobalFlag but still relies on CPU debug registers for breakpoints; hardware-breakpoint hooking (e.g. VEH + DR-based interception) used by trainers to intercept functions without patching bytes.
- **FP risk:** low — Some legitimate crash handlers and a few DRM packers set DRs transiently. Gate by sampling repeatedly: a persistent non-zero Dr7 across multiple samples and threads, excluding any DR the client itself programs, plus correlating against a known-good thread baseline. Report raw register state, let server decide.
- **Horkos slot:** `ac/src/anti_analysis/DebugRegisterScan.cpp (declared in ac/include/horkos/ac.h flag bit)`
- **Refs:** GetThreadContext, CONTEXT_DEBUG_REGISTERS, Thread32First/Thread32Next, CreateToolhelp32Snapshot; kernel PsGetContextThread

### 191. KdDebuggerEnabled / Kernel-Debugger Shared User Data Flag

- **Platform / layer:** windows-kernel / kernel
- **Signal:** On a clean retail machine the KUSER_SHARED_DATA KdDebuggerEnabled byte is 0 and SharedUserData->KdDebuggerEnabled bit-flags are clear; a set flag, or KdRefreshDebuggerNotPresent returning that a kernel debugger is attached, signals a live kernel debugger (WinDbg/KD over the wire or local) supervising the system the game runs on.
- **Mechanism:** Kernel driver reads SharedUserData->KdDebuggerEnabled and calls KdRefreshDebuggerNotPresent() / inspects KdDebuggerNotPresent and KdDebuggerEnabled exported globals; result forwarded over the existing SPSC ring -> IOCTL bridge as a host-environment flag.
- **Catches:** Kernel-level analysis of the game or driver itself: live kernel debugging used to single-step the anti-cheat driver, set kernel breakpoints, or patch kernel callbacks — a class invisible to any usermode-only check.
- **FP risk:** low — Driver developers and a small set of advanced users run with the boot debugger enabled (bcdedit /debug on) even without an active session. Distinguish 'debugger enabled in boot config' from 'debugger actively present' via KdDebuggerNotPresent; report both so server can soft-flag enabled-but-absent vs hard-flag actively-attached.
- **Horkos slot:** `kernel/win/src/HostEnvProbe.c (new), surfaced through kernel/win/include/horkos_kernel.h`
- **Refs:** KUSER_SHARED_DATA.KdDebuggerEnabled, KdRefreshDebuggerNotPresent, KdDebuggerNotPresent global, SharedUserData (0xFFDF0000/0x7FFE0000)

### 192. Inline-Hook Prologue Divergence vs On-Disk Image (.text CRC drift)

- **Platform / layer:** cross / userspace
- **Signal:** For a clean process the first bytes of each exported/critical function in a loaded module match the bytes in the module's on-disk image (after applying relocations); a divergence where the in-memory prologue starts with an absolute/relative branch (E9 jmp, FF25 jmp [rip], push+ret, 48 B8 mov rax+jmp rax) that points outside the owning module's address range is a trampoline left by a hooking library.
- **Mechanism:** Map the module file read-only with CreateFileMapping/MapViewOfFile (Windows) or mmap (POSIX), parse PE/ELF headers to locate .text and the export/PLT addresses, apply base relocations, then memcmp the first 16 bytes of each target against the live in-memory copy read via the process's own VA; classify any mismatch by decoding the leading opcode and resolving the branch target against the module's VA range (VirtualQuery / /proc/self/maps / mach vm_region).
- **Catches:** Inline (trampoline) hooking by Detours/MinHook/Frida-gum/EasyHook/SafetyHook-style libraries that redirect API or game functions; specifically catches hooks placed on functions the anti-cheat itself relies on (NtProtectVirtualMemory, recv/send, OpenGL/D3D present).
- **FP risk:** medium — Legitimate in-process software hooks the same prologues: Steam overlay (GameOverlayRenderer), Discord overlay, RTSS/MSI Afterburner, graphics-vendor overlays, accessibility tools, and some EDR usermode agents. Gate by maintaining a server-side allowlist of (module, hooked-function, trampoline-target-module) tuples for known overlays, and report the resolved target module name + signature so benign overlays are excludable rather than hard-flagged.
- **Horkos slot:** `ac/src/anti_analysis/InlineHookScan.cpp`
- **Refs:** CreateFileMapping/MapViewOfFile, VirtualQueryEx, /proc/self/maps, mach vm_region_recurse, PE IMAGE_EXPORT_DIRECTORY, ELF .dynsym/.plt relocations

### 193. IAT / GOT-PLT Redirection to Foreign Module

- **Platform / layer:** cross / userspace
- **Signal:** On a clean process every Import Address Table (Windows) / GOT entry (Linux/ELF) for an imported symbol resolves to an address inside the module that legitimately exports that symbol; an IAT/GOT slot pointing into an unbacked region, an unsigned module, or a module that does not export the symbol is an import-redirection hook.
- **Mechanism:** Parse the loaded module's IMAGE_IMPORT_DESCRIPTOR / IMAGE_DELAYLOAD_DESCRIPTOR (Windows) or ELF .rela.plt + .got (Linux) to enumerate each import thunk, read the current pointer, and resolve which module backs that address via VirtualQueryEx+GetMappedFileName / dladdr+/proc/self/maps; flag entries whose backing module is not the documented exporter or is not file-backed.
- **Catches:** Import-table hooking and GOT/PLT overwrite used to intercept API calls without touching function prologues (favored because it survives prologue-integrity checks); also catches LD_PRELOAD-style symbol interposition resident in the game process on Linux.
- **FP risk:** medium — API hooking by overlays, API-set forwarding (api-ms-win-*), AppCompat shims (sdb), and legitimate function interposers (e.g. allocator replacements like jemalloc/tcmalloc via LD_PRELOAD, or graphics shims) produce foreign-module redirects. Gate by reporting the backing module's signing identity and name; allowlist Microsoft API-set hosts and signed vendor overlays server-side; treat redirect-into-unbacked-memory as high severity vs redirect-into-signed-module as informational.
- **Horkos slot:** `ac/src/anti_analysis/ImportRedirectScan.cpp`
- **Refs:** IMAGE_IMPORT_DESCRIPTOR, GetMappedFileNameW, VirtualQueryEx, ELF .rela.plt/.got, dladdr, GetModuleInformation

### 194. Frida Gadget / Agent Residency Fingerprint (thread + mapping + RPC artifacts)

- **Platform / layer:** cross / userspace
- **Signal:** A clean process has no thread executing in an anonymous RX region whose name matches the instrumentation runtime, no module exporting frida-specific symbols, and no TCP listener on the default 27042 control port owned by the process tree; presence of any of these is dynamic-instrumentation residency beyond the basic named-mapping check.
- **Mechanism:** Enumerate threads and their start addresses (NtQueryInformationThread ThreadQuerySetWin32StartAddress / Linux /proc/self/task/*/stat + maps) and flag start addresses inside anonymous RX mappings not backed by any module; scan loaded-module export tables for the runtime's well-known exported symbol names; on Linux read /proc/self/maps for the gadget's tell-tale named regions and check /proc/net/tcp for a listening socket on the framework's default port owned by the process; on macOS use mach thread_get_state + proc_pidinfo to find unbacked thread start addresses.
- **Catches:** Frida/QBDI/Dynamo-style dynamic binary instrumentation injected as a gadget or via ptrace-spawn, including agents that renamed their pthreads/mappings to evade the basic named-thread fingerprint by falling back on start-address-in-unbacked-memory and the RPC-port observable.
- **FP risk:** low — JITs (Java/V8/.NET/LuaJIT, anti-cheat's own JIT-less by design) also create anonymous RX regions with threads executing inside them — that is the principal false positive. Gate by combining signals: unbacked-RX-thread ALONE is informational, but unbacked-RX-thread PLUS framework export symbols PLUS control-port listener is high confidence. Allowlist known JIT module signatures server-side.
- **Horkos slot:** `ac/src/anti_analysis/InstrumentationResidency.cpp`
- **Refs:** NtQueryInformationThread (ThreadQuerySetWin32StartAddress), /proc/self/task, /proc/net/tcp, proc_pidinfo PROC_PIDLISTTHREADS, thread_get_state, dlsym/export enumeration

### 195. Self-ptrace / PTRACE_TRACEME Anti-Anti-Debug Tripwire State

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A clean game process is its own ptrace tracer only if Horkos established it; observing that the game's task has a real_parent/ptracer that is a foreign userspace process, OR that a ptrace(PTRACE_ATTACH/SEIZE) syscall targeted the game's tgid from outside, indicates an external tracer — and the absence of an expected self-TRACEME tripwire (if the client armed one) indicates it was stripped.

- **Catches:** GDB/LLDB/strace attach and ptrace-based code injection on Linux/Steam Deck, including tracers that attach after exec to dodge the exec-time check; surfaces the tracer's own pid/comm so the server distinguishes a debugger from the Horkos supervisor.
- **FP risk:** low — Crash reporters (Steam's crash handler, Sentry/Breakpad's ptrace-based unwinder), and developer-attached debuggers on a dev build legitimately ptrace the game. Gate by reporting tracer comm + exe path; allowlist Steam's crashhandler and the Horkos daemon; on Steam Deck Game Mode no legitimate tracer should be present, so weight higher there.
- **Horkos slot:** `kernel/linux/bpf/src/ptrace_lsm.bpf.c (extends existing tracepoints.bpf.c)`

### 196. macOS Exception Port Hijack (task-level Mach exception handler set to foreign task)

- **Platform / layer:** macos-es / kernel
- **Signal:** A clean game task has its EXC_BAD_ACCESS/EXC_BREAKPOINT Mach exception ports owned by the kernel default or by the OS crash reporter (ReportCrash); an exception port whose receiving task is a foreign, non-Apple-signed process means a debugger/instrumentation tool has registered itself to catch the game's traps — the Mach-level analogue of a debugger attach that does not set P_TRACED.

- **FP risk:** low — Apple's own ReportCrash and legitimate in-app crash SDKs (PLCrashReporter, KSCrash, Sentry) install task exception ports. Gate on the receiver's signing identity: Apple-platform-signed or the app's own embedded crash SDK is benign; a foreign or ad-hoc-signed receiver is the anomaly. ### 197. Memory-Editor / Debugger Host Presence via Named-Object & Driver Handle Fingerprint

- **Platform / layer:** windows-user / userspace
- **Signal:** A clean host has no kernel device object, named pipe, mutex, or window-class matching the well-documented signatures of memory editors and debuggers (e.g. Cheat Engine's \Device\PhysicalMemory access driver DBK64/dbk, x64dbg/OllyDbg window classes, ReClass device names); the existence of one of these named kernel objects or top-level windows indicates a memory-editing or debugging application is running on the machine even if not yet attached to the game.
- **Mechanism:** Enumerate user objects with EnumWindows + RealGetWindowClass for known debugger window classes; probe the object-manager namespace for known device/symlink names via NtOpenDirectoryObject + NtQueryDirectoryObject over \Device and \GLOBAL??; enumerate loaded kernel drivers via EnumDeviceDrivers/GetDeviceDriverBaseName and match base names; cross-check the driver whitelist already maintained kernel-side (Whitelist.c) to attribute unsigned/BYOVD editor drivers.
- **Catches:** Host-resident memory editors and debuggers (Cheat Engine, x64dbg, ReClass.NET, ArtMoney, HxD-as-memory-editor) and their helper drivers — including the BYOVD driver an editor loads to read \Device\PhysicalMemory — present on the system but not yet bound to the game process.
- **FP risk:** medium — x64dbg/HxD/Process Hacker are legitimately used by developers and power users for non-cheating purposes; presence on the host is weaker evidence than attachment. Gate by severity tiering: a known cheat-helper driver (DBK64) is high; a generic reverse-engineering tool window with no handle to the game is low/informational. Reuse the kernel ObRegisterCallbacks handle records to escalate only when the editor actually opens a handle to the game.
- **Horkos slot:** `ac/src/anti_analysis/HostToolFingerprint.cpp (driver attribution via kernel/win/src/Whitelist.c)`
- **Refs:** EnumWindows, RealGetWindowClassW, NtOpenDirectoryObject, NtQueryDirectoryObject, EnumDeviceDrivers, GetDeviceDriverBaseNameW

### 198. RDTSC / Single-Step Timing Variance (instruction-stream latency anomaly)

- **Platform / layer:** cross / userspace
- **Signal:** On a clean process a tight, fixed-length sequence of arithmetic instructions executes within a stable nanosecond envelope across repeated samples; a sustained, large multiplicative inflation of that envelope on the threads running integrity/licence code indicates each instruction is being trapped and serviced by a single-stepping debugger or instrumentation engine.
- **Mechanism:** Sample a monotonic high-resolution counter twice around a fixed instruction block using QueryPerformanceCounter (Windows) / clock_gettime(CLOCK_MONOTONIC_RAW) (Linux) / mach_absolute_time (macOS); compute the per-sample delta distribution over many iterations and report the median and tail inflation factor versus a calibration baseline captured at clean startup. No raw RDTSC required, avoiding rdtsc-virtualization confounders.
- **Catches:** Single-step / trace-based debugging and heavyweight dynamic instrumentation (DBI engines that recompile basic blocks) that impose per-instruction or per-block overhead the integrity path can feel even when the debugger hides every other artifact.
- **FP risk:** high — Timing is noisy: scheduler preemption, SMT contention, CPU frequency scaling, thermal throttling, VM/cloud-gaming hosts, and background load all inflate the envelope and produce false positives. Gate hard: require sustained inflation across many samples on a thread pinned where possible, capture a per-machine baseline at startup, treat as a low-weight corroborating signal that only fires in combination with a structural detection (debug registers, exception-port hijack) — never as a standalone ban input.
- **Horkos slot:** `ac/src/anti_analysis/TimingProbe.cpp`
- **Refs:** QueryPerformanceCounter, clock_gettime(CLOCK_MONOTONIC_RAW), mach_absolute_time, std::chrono::steady_clock

## Process Genealogy & Loader Trust  `process-genealogy`

### 199. Parent-launcher PPID-spoof / reparent detection

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean launch shows the game's true creator equal to its recorded parent (store client / launcher); a spoofed launch via PROC_THREAD_ATTRIBUTE_PARENT_PROCESS shows the kernel-visible creator thread's owning process diverging from the inherited InheritedFromUniqueProcessId.
- **Mechanism:** In PsSetCreateProcessNotifyRoutineEx the callback runs in the context of the CREATING thread; capture PsGetCurrentProcessId() (true creator) and compare against PsProcessCreateNotifyInfo->ParentProcessId (the inherited/assigned parent). Cross-check against ZwQueryInformationProcess(ProcessBasicInformation).InheritedFromUniqueProcessId read in userspace.
- **Catches:** PPID spoofing — an unauthorized loader uses UpdateProcThreadAttribute(PROC_THREAD_ATTRIBUTE_PARENT_PROCESS) to forge the game's ancestry to look launcher-spawned while actually being spawned by an external tool.
- **FP risk:** medium — Legitimate elevation brokers and some launchers (Battle.net, certain Steam helper relaunches, AppContainer brokers, svchost-mediated CreateProcessAsUser) deliberately set an explicit parent. Gate by whitelisting the (true_creator_image, declared_parent_image) pair against signed-launcher catalog rather than flagging any divergence.
- **Horkos slot:** `kernel/win/src/proc_genealogy.c emitting a new HK_EVENT_PROCESS_CREATE flag HK_PROC_FLAG_REPARENT_SUSPECT; mirror in sdk/include/horkos/event_schema.h`
- **Refs:** PsSetCreateProcessNotifyRoutineEx, PS_CREATE_NOTIFY_INFO, PsGetCurrentProcessId, ZwQueryInformationProcess ProcessBasicInformation, UpdateProcThreadAttribute PROC_THREAD_ATTRIBUTE_PARENT_PROCESS

### 200. Create-suspended-then-resume launch window (hollowing/early-inject precursor)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean game start runs its first user thread within milliseconds of process-create; a process created CREATE_SUSPENDED for hollowing/early injection shows a measurable gap between process-create-notify and the first thread-resume, often with section/image base manipulation in between.
- **Mechanism:** Correlate PsSetCreateProcessNotifyRoutineEx (create timestamp) with PsSetCreateThreadNotifyRoutine for the initial thread, and infer initial-suspend by the absence of thread RUN transitions; on the userspace side sample the initial thread's state via NtQuerySystemInformation(SystemProcessInformation) thread state / wait-reason at first image-load. Flag image-load events arriving before the first thread leaves suspended.
- **Catches:** Process hollowing / set-thread-context injection — external loader creates the game suspended, unmaps/remaps or patches the image, then resumes; the suspend-to-resume gap plus pre-resume image edits are the artifact.
- **FP risk:** medium — Debuggers, Steam/EAC/BattlEye's own protected-launch shims, and DRM wrappers (Denuvo, Arxan) legitimately create suspended and patch before resume. Gate by allowing the case only when the suspending creator is the expected signed launcher or the registered protector; otherwise raise.
- **Horkos slot:** `kernel/win/src/launch_timing.c (timing correlation), surfaced via new HK_PROC_FLAG_SUSPENDED_LAUNCH; server correlation in server/telemetry/src/schema.rs`
- **Refs:** PsSetCreateThreadNotifyRoutine, PsSetCreateProcessNotifyRoutineEx, NtQuerySystemInformation SystemProcessInformation, KTHREAD state, CREATE_SUSPENDED

### 201. LOLBin / signed-binary-proxy in launch ancestry

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A clean ancestry is launcher->game; a proxied launch shows a Microsoft-signed living-off-the-land binary (rundll32, regsvr32, mshta, wmic, msbuild, conhost-detached, explorer-spawn) sitting between the store client and the game, or as the immediate parent.
- **Mechanism:** Walk the recorded ancestry chain built from PsSetCreateProcessNotifyRoutineEx events (already captured create/exit -> persist parent images by PID+create_time to survive PID reuse). Match each ancestor image path against a signed-LOLBin name+path catalog; emit when a known proxy binary appears in the chain to the protected game.
- **Catches:** Signed-binary-proxy execution — an external tool uses a trusted Microsoft LOLBin to spawn or DLL-proxy-load into the game, evading naive 'unsigned parent' checks because the proxy itself is validly signed.
- **FP risk:** medium — Some games/launchers genuinely shell through rundll32 or explorer.exe (shell-launched shortcuts), and modding frameworks use them. Gate on the full chain (store_client must be the root) plus command-line argument shape captured in userspace; a bare rundll32->game with no launcher root is the strong signal.
- **Horkos slot:** `sdk/src/backends/win ancestry walker + a signed-LOLBin catalog under server/api (signed-rule plumbing); flag HK_PROC_FLAG_LOLBIN_ANCESTOR`
- **Refs:** PsSetCreateProcessNotifyRoutineEx (ancestry), PS_CREATE_NOTIFY_INFO ImageFileName/CommandLine, MITRE T1218 signed binary proxy execution list

### 202. Unsigned / catalog-absent module mapped without a backing image-load (manual map)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Every legitimately loaded DLL inside the protected process fires PsSetLoadImageNotifyRoutine and has a backing FILE_OBJECT; a manually-mapped module occupies executable private/image-disguised memory with no corresponding load-image notify and no on-disk catalog entry.
- **Mechanism:** Reconcile the kernel image-load notify stream (authoritative list of loader-loaded modules with WinVerifyTrust/catalog status) against a userspace enumeration of executable regions via VirtualQueryEx MEM_PRIVATE/MEM_IMAGE with PAGE_EXECUTE_* and the loader PEB module list (NtQueryInformationProcess ProcessBasicInformation -> PEB Ldr). Executable region present, not in image-load set and not in PEB Ldr = manual-map artifact.
- **Catches:** Manual-mapped / reflectively-loaded unsigned modules — injected cheat DLLs that bypass LoadLibrary to avoid the loader and the load-image callback entirely.
- **FP risk:** medium — JITs (.NET, JVM, V8, anti-cheat's own JIT, EAC), packers, and DRM allocate executable private memory legitimately. Gate by attributing the region to a known JIT host module range and by requiring the region to also be unbacked AND outside any registered JIT allocator before flagging.
- **Horkos slot:** `kernel/win/src/module_reconcile.c (notify set) + sdk/src/backends/win/region_walk.cpp; new HK_EVENT type or HK_IMAGE_FLAG_MANUAL_MAP_SUSPECT`
- **Refs:** PsSetLoadImageNotifyRoutine, IMAGE_INFO, VirtualQueryEx MEMORY_BASIC_INFORMATION, NtQueryInformationProcess ProcessBasicInformation/PEB Ldr, WinVerifyTrust

### 203. Integrity-level / token-mismatch between game and its launcher

- **Platform / layer:** windows-kernel / kernel
- **Signal:** A store-launched game inherits the launcher's medium integrity and user token; a tool-launched or token-manipulated game shows an integrity level, elevation state, or token session ID inconsistent with its recorded parent launcher (e.g. game at High while launcher at Medium, or game in a different logon session).
- **Mechanism:** Userspace: OpenProcessToken on game + recorded parent and compare TOKEN_MANDATORY_LABEL (GetTokenInformation TokenIntegrityLevel), TokenElevation, TokenSessionId, and TokenLinkedToken. Kernel corroboration: SeQueryInformationToken in a process-create context. Flag integrity escalation or session divergence relative to the signed launcher baseline.
- **Catches:** Token/integrity manipulation — privilege-escalated relaunch, CreateProcessWithTokenW with a stolen/duplicated token, or RunAs-bypass loaders that give the game rights its real launcher never had.
- **FP risk:** medium — UAC elevation prompts and admin-required launchers legitimately produce a High-integrity game from a Medium shell; some anti-cheats relaunch elevated. Gate by baselining the expected integrity per known launcher and only flagging divergence from that launcher's documented level, not absolute High.
- **Horkos slot:** `sdk/src/backends/win/token_check.cpp; telemetry field token_integrity_delta declared in server/api/data-categories.md`
- **Refs:** OpenProcessToken, GetTokenInformation TokenIntegrityLevel/TokenElevation/TokenSessionId/TokenLinkedToken, SeQueryInformationToken, CreateProcessWithTokenW

### 204. Job-object / silo containment anomaly around the game

- **Platform / layer:** windows-user / userspace
- **Signal:** A normally launched game is either in no job or in its launcher's known job; a tool-instrumented game is found inside an unexpected nested job object or server silo created by an external supervisor to intercept handles, throttle, or sandbox it for tooling.
- **Mechanism:** IsProcessInJob + QueryInformationJobObject(JobObjectBasicProcessIdList / JobObjectExtendedLimitInformation) on the game; NtQueryInformationProcess(ProcessJobMemoryInformation) and silo membership via NtQueryInformationProcess(ProcessConsoleHostProcess)/silo APIs. Compare the job's other member PIDs and creator against the expected launcher's job.
- **Catches:** External supervision / containment — loaders or analysis harnesses that place the game in their own job object or silo to mediate kill-on-close, handle inheritance, or resource interception.
- **FP risk:** high — Windows itself, Xbox/UWP AppContainers, Steam (steamwebhelper job), GameBar, Sandboxie, and Game Mode all use jobs/silos heavily. Gate hard: only flag jobs whose creator process is unsigned/unknown AND whose member set includes a non-whitelisted instrumentation binary.
- **Horkos slot:** `sdk/src/backends/win/job_silo_check.cpp; advisory-only telemetry field, no autonomous ban`
- **Refs:** IsProcessInJob, QueryInformationJobObject JobObjectBasicProcessIdList, NtQueryInformationProcess ProcessJobMemoryInformation, Windows Server Silos

### 205. execve preceded by PTRACE_TRACEME/ATTACH in the same genealogy (Linux launch-under-tracer)

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A clean game exec has tracer_pid == 0 at first run; a tool-launched game shows a ptrace TRACEME/SEIZE/ATTACH on the to-be-exec'd child immediately before or after sched_process_exec, i.e. it was launched under a debugger/instrumentation parent.
- **Mechanism:** Join the existing tracepoint/syscalls sys_enter_ptrace stream with sched_process_exec over the CO-RE ringbuf, keyed by (tgid, pid): correlate a ptrace request (PTRACE_TRACEME from the child pre-exec, or ATTACH/SEIZE from parent) with the subsequent exec of the protected image, and read task->ptrace / real_parent via a BPF LSM hook (lsm/ptrace_access_check) or task_struct CO-RE field.
- **Catches:** Launch-under-tracer instrumentation — game started by an external supervising process (frida-style spawn-and-attach, gdb-launched) rather than the Steam client, even when /proc/self/status TracerPid is later cleared.
- **FP risk:** medium — Steam itself, gameoverlayrenderer, crash handlers (crashpad), and Proton/Wine launch children under supervision; some launchers ptrace for overlay injection. Gate by requiring the tracer to be outside the Steam/Proton process group and not a registered overlay helper.
- **Horkos slot:** `kernel/linux/bpf/genealogy.bpf.c (join ptrace+exec); lsm/ptrace_access_check hook; flag in the CO-RE ringbuf record consumed by kernel/linux/userspace`
- **Refs:** tracepoint syscalls:sys_enter_ptrace, tracepoint sched:sched_process_exec, BPF LSM lsm/ptrace_access_check, task_struct->ptrace/real_parent, PTRACE_TRACEME/PTRACE_SEIZE

### 206. LD_PRELOAD / dynamic-linker hijack in the exec environment (Linux loader-trust)

- **Platform / layer:** linux-ebpf / kernel
- **Signal:** A clean Steam/native launch has no LD_PRELOAD, default LD_LIBRARY_PATH, and the stock ld.so as PT_INTERP; an instrumented launch carries LD_PRELOAD/LD_AUDIT, a rewritten PT_INTERP, or a /etc/ld.so.preload entry forcing an unsigned .so into the game at exec.
- **Mechanism:** On the sched_process_exec / tracepoint sys_enter_execve path, read the argv/envp pointers from the process stack via bpf_probe_read_user to detect LD_PRELOAD/LD_AUDIT/LD_LIBRARY_PATH; corroborate in userspace by reading /proc/<pid>/maps (mapped .so set vs DT_NEEDED) and the binary's PT_INTERP, plus stat of /etc/ld.so.preload, and reconcile against the BPF LSM lsm/file_open mmap-exec audit stream already collected.
- **Catches:** Preload/linker-based injection — forcing a cheat shared object into the game through the dynamic loader without modifying the ELF, the classic Linux/Proton injection vector.
- **FP risk:** medium — Steam overlay (gameoverlayrenderer.so), MangoHud, vkBasalt, OBS game-capture, and accessibility tools all legitimately LD_PRELOAD. Gate against a signed-allowlist of known overlay/HUD .so by path+hash; only unlisted preloaded objects flag.
- **Horkos slot:** `kernel/linux/bpf/loader_trust.bpf.c (envp scan on execve) + kernel/linux/userspace maps/PT_INTERP reconcile; reuses lsm/file_open audit stream`
- **Refs:** tracepoint syscalls:sys_enter_execve, bpf_probe_read_user (argv/envp), BPF LSM lsm/file_open, /proc/<pid>/maps, ELF PT_INTERP, /etc/ld.so.preload, LD_AUDIT

### 207. Responsible-process / parent-responsibility mismatch (macOS launch-trust)

- **Platform / layer:** macos-es / kernel
- **Signal:** A clean game launch has its ES 'responsible' process and parent equal to the expected Steam/store client and a matching Team ID / signing identity; a tool-launched game shows responsibility attributed to Terminal, an unsigned/ad-hoc launcher, or a process whose Team ID does not match the launcher baseline.
- **Mechanism:** In the EndpointSecurity ES_EVENT_TYPE_NOTIFY_EXEC handler, read es_process_t->responsible_audit_token (resolve via es_message responsible fields), es_process_t->parent_audit_token, and the target's es_process_t->team_id / es_process_t->signing_id / cdhash; compare the launch chain's signing identities against the expected launcher's code-signing baseline.
- **Catches:** Untrusted-launcher / responsibility laundering — game spawned by an unsigned or non-store launcher, or re-parented so TCC/Gatekeeper responsibility points at a benign signed app while the real launcher is an instrumentation tool.
- **FP risk:** medium — Developer launches from Terminal/Xcode, and some legitimate third-party launchers (Whisky, CrossOver, Heroic) are valid responsible parents. Gate by an allowlist of accepted launcher Team IDs/cdhashes; flag only when responsibility resolves outside that set with no store client in the chain.
- **Horkos slot:** `kernel/macos/es/genealogy_handler.cpp (NOTIFY_EXEC responsible/team-id extraction); daemon/macos/horkosd.cpp bring-up path mirror`
- **Refs:** ES_EVENT_TYPE_NOTIFY_EXEC, es_process_t responsible_audit_token/parent_audit_token/team_id/signing_id/cdhash, audit_token_to_pid, SecCode signing-identity APIs

## Windows Kernel — Syscall / ETW / PatchGuard Surface Integrity  `win-kernel-syscall-etw-integrity`

> Hand-authored. The automated design pass could not generate this domain (the kernel syscall-table integrity topic reads as offensive tooling to the content filter, even when framed defensively). These are standard read-only integrity checks an anti-cheat performs on the dispatch surface; every entry only *reads and bounds-checks* kernel pointers and reports anomalies to the server — none of them modify the surface.

### 208. KiServiceTable (SSDT) entry bounds validation

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Each native syscall dispatch entry must resolve to an address inside the ntoskrnl.exe `.text` range; a clean system has every decoded `KeServiceDescriptorTable.ServiceTableBase[i]` pointing into the kernel image, an SSDT hook points an entry outside it.
- **Mechanism:** Resolve `KeServiceDescriptorTable` (on x64 the entries are 4-byte sign-extended offsets from the table base), bound-check each decoded target against the ntoskrnl image base..base+SizeOfImage obtained from `PsLoadedModuleList` / `AuxKlibQueryModuleInformation`.
- **Catches:** Classic SSDT hooking / entry-offset rewriting used to intercept `NtReadVirtualMemory`/`NtOpenProcess`/`NtQuerySystemInformation` for read-access cheats and process hiding.
- **FP risk:** low — On modern PatchGuard-protected x64 no legitimate driver hooks the SSDT, so any out-of-image entry is high-confidence. The only risk is build-fragile table decoding: ship per-build offset tables and report "unverifiable" rather than false-positive on an unrecognized build.
- **Horkos slot:** `kernel/win/src/SyscallIntegrity.c (new), HkSsdtValidate in kernel/win/include/horkos_kernel.h`
- **Refs:** KeServiceDescriptorTable, KiSystemServiceStart, PsLoadedModuleList, AuxKlibQueryModuleInformation

### 209. Shadow SSDT (W32pServiceTable) bounds validation

- **Platform / layer:** windows-kernel / kernel
- **Signal:** GUI-thread syscall entries in the shadow service table must resolve inside win32k.sys/win32kbase.sys/win32kfull.sys; an entry pointing elsewhere indicates a shadow-SSDT hook on the window/input path.
- **Mechanism:** Locate `KeServiceDescriptorTableShadow` from a GUI thread context (KTHREAD with a non-null Win32Thread), decode `W32pServiceTable` entries as on the native table, bound-check each against the win32k module range from `PsLoadedModuleList`. Sample by `KeStackAttachProcess` into a known GUI process.
- **Catches:** Shadow-SSDT hooks on `NtUserGetAsyncKeyState`/`NtUserGetForegroundWindow`/`NtGdiBitBlt` used by overlay/ESP and input-state cheats to read or fabricate window/input state.
- **FP risk:** low — Same build-fragility gating as the native SSDT check; report unverifiable on unknown win32k layouts.
- **Horkos slot:** `kernel/win/src/SyscallIntegrity.c (new), HkShadowSsdtValidate`
- **Refs:** KeServiceDescriptorTableShadow, W32pServiceTable, KeStackAttachProcess, win32kbase.sys

### 210. IA32_LSTAR MSR syscall-entry validation

- **Platform / layer:** windows-kernel / kernel
- **Signal:** The `IA32_LSTAR` MSR (0xC0000082) holds the x64 SYSCALL entry point and on a clean system equals `&KiSystemCall64` (or `&KiSystemCall64Shadow` under KVA-shadow); any other value means the whole syscall path has been redirected before dispatch.
- **Mechanism:** `__readmsr(0xC0000082)` on each processor (via `KeIpiGenericCall` / a per-CPU DPC), compared against the resolved address of `KiSystemCall64`/`KiSystemCall64Shadow`; mismatch and per-CPU divergence are both reported.
- **Catches:** Whole-syscall redirection (LSTAR repoint) — the strongest blinding primitive against syscall-based telemetry.
- **FP risk:** low — KVA-shadow makes the expected value `KiSystemCall64Shadow`; account for both. No legitimate software rewrites LSTAR on modern Windows. Run the IPI off the hot path.
- **Horkos slot:** `kernel/win/src/SyscallIntegrity.c (new), HkLstarValidate via KeIpiGenericCall`
- **Refs:** __readmsr, IA32_LSTAR (0xC0000082), KiSystemCall64, KiSystemCall64Shadow, KeIpiGenericCall

### 211. Infinity-hook detection via kernel perf-trace callback validation

- **Platform / layer:** windows-kernel / kernel
- **Signal:** The "infinity hook" technique abuses the kernel ETW/WMI performance-trace path so a callback fires on every syscall; a clean system's perf-trace callback resolves into ntoskrnl/hal, a hooked one points into foreign memory.
- **Mechanism:** Resolve the kernel performance-interrupt callback chain (`HalpPerfInterrupt` and the `WMI_LOGGER_CONTEXT` trace-clock callback) and bound-check the target against ntoskrnl/hal image ranges; corroborate by sampling whether circular-kernel-trace was silently enabled.
- **Catches:** Infinity-hook-style syscall interception that evades PatchGuard by hijacking the legitimate ETW trace callback instead of patching the SSDT or LSTAR.
- **FP risk:** medium — Relies on undocumented `WMI_LOGGER_CONTEXT` internals that shift across builds (high version-fragility). Legitimate tracing tools (xperf/WPR) toggle the same logger; gate by correlating with a known-tracing-session census and flagging only a callback target outside any signed image.
- **Horkos slot:** `kernel/win/src/SyscallIntegrity.c (new), HkInfinityHookProbe`
- **Refs:** HalpPerfInterrupt, WMI_LOGGER_CONTEXT, EtwpDebuggerData

### 212. ETW Threat-Intelligence provider liveness check

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Microsoft-Windows-Threat-Intelligence (ETW-TI) feeds kernel-sourced ReadVm/WriteVm/AllocVm events Horkos consumes; a clean system has the provider registered and enabled, whereas a nulled `EtwThreatIntProvRegHandle` means a cheat disabled the TI feed to blind memory-access telemetry.
- **Mechanism:** Validate the ETW-TI provider registration handle is non-null and its enable mask is set; corroborate by confirming Horkos's own ETW-TI consumer session is still receiving keepalive events within an expected interval.
- **Catches:** ETW-TI teardown (the common anti-EDR step of zeroing the TI provider handle) that defeats the kernel ReadVm/WriteVm signals other Horkos sensors depend on.
- **FP risk:** low — `EtwThreatIntProvRegHandle` is an unexported global recovered by offset per build; fail to "unverifiable" on unknown builds. Liveness-via-keepalive is the version-independent half — weight it higher than the raw handle read.
- **Horkos slot:** `kernel/win/src/EtwIntegrity.c (new), HkEtwTiLiveness; keepalive counter in kernel/win/src/Notify.c`
- **Refs:** Microsoft-Windows-Threat-Intelligence, EtwThreatIntProvRegHandle, EtwRegister

### 213. Syscall-entry prologue tamper scan (KiSystemCall64 / KiSystemServiceStart)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** The first bytes of the syscall dispatch entry (`KiSystemCall64`) and dispatch helper (`KiSystemServiceStart`) match the on-disk ntoskrnl image on a clean system; an inline patch (jmp/push-ret trampoline at the entry) diverges from the disk bytes.
- **Mechanism:** Map the on-disk ntoskrnl.exe, relocate its `.text`, and byte-compare the in-memory prologue against the disk image, accounting for legitimate boot-time relocations (retpoline/spectre thunks, KVA-shadow, hotpatch) before the diff.
- **Catches:** Inline trampoline hooks placed directly on the syscall entry stub — an alternative to SSDT/LSTAR hooking that survives if the table pointers look clean.
- **FP risk:** medium — The kernel legitimately patches itself at boot; a naive disk-vs-memory diff false-positives on these. Gate by applying the same relocations/hotpatch deltas before comparison, or restrict the window to bytes known-stable across mitigations.
- **Horkos slot:** `kernel/win/src/SyscallIntegrity.c (new), HkSyscallPrologueScan`
- **Refs:** KiSystemCall64, KiSystemServiceStart, ntoskrnl on-disk image, IMAGE_BASE_RELOCATION, KVA shadow

### 214. IDT handler bounds validation

- **Platform / layer:** windows-kernel / kernel
- **Signal:** Every IDT gate's handler resolves into ntoskrnl/hal on a clean system; a descriptor pointing into foreign memory indicates an IDT hook (e.g. #PF or #DB hooks used to trap memory access or single-step).
- **Mechanism:** Read the per-CPU IDT base via the KPCR (or `__sidt` through `KeIpiGenericCall`), walk each `KIDTENTRY64` to reconstruct the 64-bit handler address, bound-check against ntoskrnl/hal module ranges; report per-CPU divergence and any out-of-image gate.
- **Catches:** IDT hooks on #PF/#DB/#BP used by stealth cheats to intercept page faults (hidden-memory tricks) or hijack debug/breakpoint exceptions.
- **FP risk:** low — In-guest IDT handlers still resolve into the kernel even under a hypervisor, so an out-of-image handler is high-confidence. Run the read off the hot path via IPI.
- **Horkos slot:** `kernel/win/src/SyscallIntegrity.c (new), HkIdtValidate`
- **Refs:** __sidt, KIDTENTRY64, KPCR, KeIpiGenericCall, KINTERRUPT

### 215. Kernel ETW logger-session census for telemetry suppression

- **Platform / layer:** windows-kernel / kernel
- **Signal:** The sudden disable of the NT Kernel Logger or of a security provider Horkos relies on — or a foreign session that stops a provider — is the suppression signature against a boot-time baseline of active sessions and enabled providers.
- **Mechanism:** Query the ETW logger table (`NtTraceControl` EtwQueryAllTraces from a kernel helper) and diff the enabled-provider set against the boot-time baseline, flagging disables of security-relevant providers.
- **Catches:** Telemetry-suppression cheats that stop or reconfigure ETW sessions (Kernel Logger, Threat-Intelligence, security auditing) to blind host monitoring before acting.
- **FP risk:** medium — Profiling and legitimate EDR toggle ETW sessions constantly. Gate by alerting only on disables of providers Horkos depends on, correlated with our own consumer losing keepalives.
- **Horkos slot:** `kernel/win/src/EtwIntegrity.c (new), HkEtwSessionCensus`
- **Refs:** NtTraceControl, EtwQueryAllTraces, WMI_LOGGER_CONTEXT, EVENT_TRACE_PROPERTIES

### 216. KeServiceDescriptorTable pointer integrity (table-base swap detection)

- **Platform / layer:** windows-kernel / kernel
- **Signal:** The `ServiceTableBase`/`ServiceLimit` fields of `KeServiceDescriptorTable[Shadow]` are constant after boot; a swapped `ServiceTableBase` pointing at an attacker-built copy of the SSDT (rather than patching individual entries) is detectable as a base-pointer change from baseline.
- **Mechanism:** Snapshot `ServiceTableBase` + `ServiceLimit` (and the shadow descriptor) at arm time, re-read on the self-check timer, confirm the base still points into ntoskrnl and equals the boot baseline; a relocated base or altered limit is reported.
- **Catches:** Whole-table substitution (ServiceTableBase redirected to a cloned, pre-hooked SSDT) that leaves the original table pristine to fool per-entry validators.
- **FP risk:** low — The descriptor base is stable across the OS lifetime; no legitimate post-boot change occurs. Pair with the per-entry bounds check (#208) so both a base swap and individual entry hooks are covered.
- **Horkos slot:** `kernel/win/src/SyscallIntegrity.c (new), baseline in HK_DEVICE_CONTEXT.SsdtBaseline`
- **Refs:** KeServiceDescriptorTable, KeServiceDescriptorTableShadow, KSERVICE_DESCRIPTOR_TABLE

### 217. Multivariate behavioral reconstruction anomaly (server autoencoder)

- **Platform / layer:** server / ML fusion input
- **Signal:** An honest-population reconstruction model observes a complete window spanning aim kinematics, direction-change reaction latency, large-gap target switching, HID cadence variance, injected-event fraction, and client clock drift. A window whose joint feature relationships lie outside the learned honest manifold receives a high reconstruction strength even when no single constituent feature is independently decisive.
- **Mechanism:** `server/ml/train_anomaly_model.py` fits a rank-4 linear/PCA autoencoder on standardized honest-only window vectors and exports normalization, reconstruction MSE, and empirical honest-tail calibration in one ONNX graph. `server/ban-engine/src/anomaly.rs` constructs the identical ten-feature order from existing schema-v4+ ticks, enforces independent per-modality recurrence floors, runs the graph through `ort`, and emits signal 217 before unchanged fusion.
- **Catches:** Broad bot-like coordination across input timing, reaction, switching, and clock domains; input or timing automation that breaks normal cross-feature relationships without necessarily crossing an individual hand threshold.
- **FP risk:** high before real-corpus calibration; the PoC corpus and anomalous evaluation modes are synthetic and generator-defined, making reported separation partly circular. Missing modalities produce no signal; the empirical 99th-percentile honest calibration is only a synthetic pipeline check; production requires platform/cohort-aware honest telemetry and drift monitoring. This signal is never standalone-bannable and requires independent corroboration through fusion.
- **Horkos slot:** `server/ml/{behavior_features.json,synth_behavior.py,train_anomaly_model.py,eval_anomaly_model.py}; server/ban-engine/src/anomaly.rs; server/ban-engine/src/pipeline.rs`
